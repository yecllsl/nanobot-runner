# 【学习工作赛道】Nanobot Runner - 桌面端私人AI跑步助理

## 1. Demo 简介

**是什么：** Nanobot Runner 是一款基于 nanobot-ai 底座的桌面端本地化 AI 跑步助理应用，通过 CLI 命令行 + WebUI 可视化双端交互，为技术型跑者提供企业级 BI 能力。

**面向谁：** 拥有 2年+ 跑步数据的资深跑者，特别是：

- 技术型跑者（喜欢用数据分析训练）
- 马拉松爱好者（需要科学训练规划）
- 数据隐私敏感用户（希望数据完全本地化）

**主要功能：**

### 📊 专业数据分析

- **VDOT 跑力值分析**：基于 Jack Daniels 公式追踪最大摄氧量等效值，预测比赛成绩
- **训练负荷监控**：TSS/ATL/CTL/TSB 四大核心指标，科学量化训练压力
- **身体信号分析**：HRV 心率变异、疲劳度评估、恢复状态监控
- **心率漂移检测**：分析相同配速下心率变化，评估有氧能力提升

### 🤖 AI 智能能力

- **自然语言交互**：通过对话查询数据、获取分析建议
- **智能训练计划**：基于体能状态和目标生成个性化计划
- **ML 增强预测**：VDOT 趋势预测、比赛成绩预测、伤病风险预警
- **数字孪生引擎**：5 维度状态向量、What-If 推演、计划对比

### 🌐 WebUI 可视化（10 大页面）

仪表盘、VDOT 趋势、训练负荷、活动列表、活动详情、身体信号、训练计划、进化引擎控制台、进化报告、设置中心

## 2. Demo 创作思路

### 灵感来源

作为一名跑步爱好者和程序员，我发现市面上的跑步 APP 虽然功能丰富，但存在两个核心痛点：

1. **数据隐私问题**：所有数据存储在云端，用户无法完全掌控
2. **分析深度不足**：提供的分析指标过于基础，无法满足资深跑者的专业需求

我希望打造一个**真正懂自己、能预测未来、会自我进化**的 AI 训练科学家，让数据只属于自己，同时提供企业级的分析能力。

### 想解决的问题

- **数据隐私与深度分析的矛盾**：如何在保护隐私的前提下提供专业的数据分析？
- **AI 建议的可信度**：如何让 AI 的训练建议可追溯、可解释、可验证？
- **个性化不足**：如何让系统越来越懂跑者的个人特点，而不是千篇一律的通用建议？

### 为什么做这个方向

1. **技术可行性**：nanobot-ai 底座 + Polars 高性能计算 + Parquet 列式存储，可以在本地实现企业级 BI 能力
2. **市场需求**：资深跑者对数据分析和隐私保护有强烈需求，但市场上缺乏专业工具
3. **技术挑战**：数字孪生、自适应进化、ML 预测等前沿技术的应用，具有很高的技术价值

## 4. TRAE 实践过程

### 开发流程概览

整个项目使用 **TRAE IDE** 完成，采用多智能体协作模式：

```
需求分析 → 架构设计 → 任务拆解 → 代码开发 → 测试验证 → 文档生成
   ↓           ↓          ↓          ↓          ↓          ↓
 用户        架构师      架构师     开发者     测试工程师   文档智能体
```

### TRAE 协同开发体系

本项目严格遵循 TRAE IDE 的**协同黄金法则**，构建了完整的 Rules-Skills-Subagents 三位一体开发体系：

```
协同黄金法则
├─ 1. Rules先行 — 任何操作前先检查规则约束
├─ 2. Skills指导 — 根据任务类型激活对应技能
├─ 3. Subagents执行 — 复杂任务分解给专业代理
└─ 4. Rules验证 — 执行结果必须通过规则检查
```

#### 规则体系（Rules）

项目建立了完整的规则约束体系，确保开发过程规范可控：

| 规则类型           | 文件                    | 核心约束                |
| -------------- | --------------------- | ------------------- |
| **安全规则**       | `security-rules.md`   | 输入验证、敏感数据保护、访问控制    |
| **合规规则**       | `compliance-rules.md` | 代码审计、许可证检查、数据隐私     |
| **质量规则**       | `quality-rules.md`    | 开发方法论、测试覆盖≥80%、代码规范 |
| **流程规则**       | `process-rules.md`    | 开发过程、分支策略、审批流程      |
| **Ponytail规则** | `ponytail.md`         | YAGNI原则、最小实现、避免过度设计 |

所有代码开发必须通过四层规则检查：输入层（请求验证）→ 执行层（行为管控）→ 输出层（结果检查）→ 审计层（日志追踪）。

#### 技能体系（Skills）

采用\*\*全局Skills（方法论）+ 项目Skills（执行器）\*\*双层技能架构：

**全局Skills（方法论层）**：

- `brainstorming` — 需求探索与创意设计（铁律：不经设计批准，不写任何代码）
- `writing-plans` — 实施计划编写（铁律：每步必须包含实际内容，禁止占位符）
- `test-driven-development` — TDD循环（铁律：没有失败测试，不写生产代码）
- `systematic-debugging` — 四阶段根因定位（铁律：没有根因调查，不提任何修复）
- `verification-before-completion` — 交付前验证（铁律：没有新鲜验证证据，不做完成声称）
- `subagent-driven-development` — 子代理驱动开发
- `dispatching-parallel-agents` — 并行代理调度

**项目Skills（执行器层）**：

| 阶段     | Skill  | 指令ID   | 核心产出              |
| ------ | ------ | ------ | ----------------- |
| **产品** | 产品规划   | PM-01  | 产品规划方案.md         |
| **需求** | 需求分析   | ARC-01 | REQ\_需求规格说明书.md   |
| **架构** | 架构设计   | ARC-02 | 架构设计说明书.md        |
| **架构** | 架构评审   | ARC-04 | 架构评审报告.md         |
| **规划** | 任务拆解   | ARC-03 | task\_list\_\*.md |
| **开发** | 功能开发   | DEV-01 | 代码+测试+交付报告        |
| **开发** | Bug修复  | DEV-02 | 修复代码+修复报告         |
| **评审** | 代码评审   | REV-01 | 代码评审报告.md         |
| **测试** | 测试策略   | TST-01 | strategy\_\*.md   |
| **测试** | 测试执行   | TST-02 | 测试报告+Bug清单        |
| **测试** | 回归测试   | TST-03 | 回归报告+上线结论         |
| **运维** | 文档更新   | OPS-01 | 更新后的文档集           |
| **运维** | CICD验证 | OPS-02 | 流水线执行报告           |
| **运维** | 自动化发布  | OPS-03 | 版本Tag+发布报告        |
| **运维** | 文档归档   | OPS-05 | 归档目录              |

**技能协作模式**：

1. **串联模式**：brainstorming → 需求分析 → 架构设计 → writing-plans → 功能开发
2. **嵌入模式**：TDD循环嵌入功能开发过程，verification嵌入每个完成节点
3. **守门模式**：功能开发 → \[verification] → 代码评审 → \[verification] → 测试执行
4. **并行模式**：dispatching-parallel-agents 调度多Agent并行开发独立功能

#### 智能体体系（Subagents）

构建了六类专业智能体，各司其职、协同工作：

| 角色          | 核心职责                     | 关键交付物                | 绑定模型      |
| ----------- | ------------------------ | -------------------- | --------- |
| **产品经理**    | 产品规划、需求收集、价值验收           | 产品规划方案、用户需求清单        | Kimi-K2.5 |
| **架构师**     | 需求分析、架构设计、任务拆解、方案评审      | 需求规格说明书、架构设计说明书、任务清单 | GLM-5     |
| **开发工程师**   | 功能开发、单元测试、Bug修复、代码优化     | 功能代码、单元测试、开发交付报告     | GLM-5.1   |
| **测试工程师**   | 测试策略、用例设计、测试执行、质量评估      | 测试用例、测试报告、Bug清单、上线结论 | —         |
| **代码评审工程师** | 代码质量评审、架构一致性检查、安全审计      | 代码评审报告               | —         |
| **发布运维工程师** | CICD搭建、自动化发布、FIX-CI、监控配置 | 流水线配置、发布报告、运维文档      | —         |

**智能体协作链路**：

```
产品经理（产品规划）
    ↓
架构师（需求分析→架构设计→架构评审→任务拆解）
    ↓
开发工程师（功能开发→单元测试→本地验证）
    ↓               ← [TDD] 编码过程内嵌
                    ← [verification] 每步完成验证
代码评审工程师（代码评审）
    ↓
测试工程师（测试策略→测试执行→Bug反馈）
    ↓
开发工程师（Bug修复）
    ↓               ← [systematic-debugging] Bug根因不明时前置
测试工程师（回归测试→上线结论）
    ↓
发布运维（文档更新→CICD验证→自动化发布→文档归档）
```

### TRAE 核心能力应用总结

**1. Rules 规则约束能力**

- 建立了完整的规则约束体系（安全、合规、质量、流程、Ponytail）
- 所有代码开发必须通过四层规则检查
- 确保开发过程规范可控，避免过度设计

**2. Skills 技能指导能力**

- 全局Skills提供方法论指导（brainstorming、TDD、verification等）
- 项目Skills提供标准化执行流程（PM-01、ARC-02、DEV-01等）
- 四种协作模式（串联、嵌入、守门、并行）确保技能高效协同

**3. Subagents 智能体执行能力**

- 六类专业智能体各司其职（产品经理、架构师、开发工程师、测试工程师、代码评审工程师、发布运维工程师）
- 智能体间通过文档交付机制协同工作
- 复杂任务自动分解给专业代理执行

**4. 全局Skill铁律保障**

- brainstorming：不经设计批准，不写任何代码
- TDD：没有失败测试，不写生产代码
- systematic-debugging：没有根因调查，不提任何修复
- verification-before-completion：没有新鲜验证证据，不做完成声称
- writing-plans：每步必须包含实际内容，禁止占位符

### 开发心得

**踩坑经历：**

1. **Polars LazyFrame 性能优化**：初期直接使用 collect() 导致性能下降，后改为全程使用 LazyFrame 链式调用，性能提升 3 倍
2. **nanobot Config Schema 严格校验**：配置字段必须严格匹配 schema，unrecognized fields 会导致启动失败
3. **WebSocket 连接稳定性**：初期 Gateway 服务因未捕获 ConnectionClosedError 而崩溃，后在事件循环层添加异常处理

**经验总结：**

1. **Rules先行**：任何操作前先检查规则约束，避免违规操作
2. **Skills指导**：根据任务类型激活对应技能，方法论先行，执行标准化
3. **Subagents执行**：复杂任务分解给专业代理，各司其职，协同增效
4. **verification守门**：每个完成节点必须有验证证据，避免虚假完成
5. **渐进式迭代**：采用小步快跑的迭代方式，每个版本聚焦核心功能
6. **充分利用 TRAE 协同体系**：Rules-Skills-Subagents 三位一体，大幅提升开发效率和质量
