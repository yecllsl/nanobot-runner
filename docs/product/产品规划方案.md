## 1. 产品愿景与战略定位

### 1.1 核心价值主张

> **"让拥有2年+跑步数据的资深跑者，拥有一个真正懂自己、能预测未来、数据完全可控的 AI 训练科学家——从数据沉淀到预测洞察，从经验训练到智能优化，全程本地、全程透明、全程可控。"**

| 战略维度 | 用户价值 | 市场盲点 |
|----------|----------|----------|
| **隐私可控** | 跑者的数据只属于自己，不上传、不泄露 | Strava/Garmin数据泄露事件频发 |
| **专业可信** | 基于运动科学的分析，AI建议可追溯、可验证 | 云端AI建议黑箱，不可解释 |
| **预测未来** ⭐ | 基于2年+数据，预测伤病风险、VDOT趋势、比赛成绩 | 现有产品仅事后分析，无预测能力 |
| **智能优化** ⭐ | RL驱动的个性化训练负荷推荐，真正"因人而异" | 通用模型，非个性化 |
| **即时可得** | 终端内秒级响应，无需打开浏览器/APP | 云端产品响应慢、需要联网 |

### 1.2 目标用户画像

**主要用户：数据敏感的严肃跑者**

| 属性 | 描述 |
|------|------|
| **年龄段** | 25-45 岁 |
| **职业背景** | 程序员、工程师、数据分析师等技术从业者 |
| **跑步经验** | 规律跑步 2 年以上，有马拉松参赛经历 |
| **数据意识** | 习惯佩戴心率带、功率计等专业设备 |
| **核心痛点** | 担心云端数据隐私，需要更专业的数据分析 |
| **技术能力** | 熟悉命令行操作，具备一定编程基础 |

## 3. v0.19.0 详细规划：让身体信号"会说话"

### 3.1 用户问题定义

**核心痛点**：
> "我知道心率、功率这些数据很重要，但我看不懂它们之间的关系。为什么今天同样配速心率却更高？我的身体到底恢复好了没有？我想根据自己的情况自定义分析指标。"

**用户场景**：
1. **晨脉监测**：每天醒来想快速了解身体恢复状态
2. **训练后反思**：想知道为什么某次训练感觉特别累
3. **长期趋势**：想追踪身体对训练负荷的适应情况

### 3.2 版本核心价值

| 价值维度 | 交付内容 | 用户收益 |
|----------|----------|----------|
| **洞察深度** | 心率变异、疲劳度、恢复状态分析 | 读懂身体信号，避免过度训练 |
| **可操作性** | 基于身体信号的 actionable 建议 | 知道今天该练什么、练多狠 |

### 3.3 功能规划

#### P0 - 必须交付（核心功能）

**1. 心率变异（HRV）分析**

| 功能项 | 用户价值 | 验收标准 |
|--------|----------|----------|
| 静息心率趋势 | 了解基础体能变化 | 支持7/30/90天趋势查看 |
| 心率恢复分析 | 评估心肺恢复能力 | 运动后1分钟/3分钟恢复率计算 |
| 心率漂移检测 | 识别过度训练信号 | 长距离跑步中心率漂移>10%预警 |
| HRV基础统计 | 了解自主神经状态 | 基于现有心率数据的HRV估算 |

**2. 疲劳度与恢复评估**

| 功能项 | 用户价值 | 验收标准 |
|--------|----------|----------|
| 综合疲劳度评分 | 量化当前疲劳状态 | 0-100分，基于ATL/睡眠/主观感受 |
| 恢复状态指示器 | 判断是否可以高强度训练 | 红/黄/绿三色状态 |
| 连续训练日监控 | 避免连续高强度训练 | 7天内高强度训练次数统计 |
| 休息日后效果评估 | 验证休息是否充分 | 对比休息前后关键指标变化 |

**3. 身体信号智能解读**

| 功能项 | 用户价值 | 验收标准 |
|--------|----------|----------|
| 异常信号预警 | 及时发现身体异常 | 静息心率突增>10%预警 |
| 训练建议生成 | 基于身体状态给出建议 | "今天适合轻松跑"等 actionable 建议 |
| 身体信号摘要 | 快速了解身体状态 | 每日/每周身体信号一句话总结 |

#### P1 - 应该交付（增强功能）

**5. 深度对比分析**

| 功能项 | 用户价值 | 验收标准 |
|--------|----------|----------|
| 周期对比 | 对比不同训练周期 | 支持同期对比（如今年vs去年） |
| 相似训练对比 | 找出最佳训练模式 | 按距离/配速筛选相似训练对比 |
| 训练负荷-表现关联 | 找到最佳负荷区间 | 散点图显示负荷与表现关系 |

#### P2 - 可选交付（Nice to have）

**6. 睡眠与恢复数据集成**

| 功能项 | 用户价值 | 备注 |
|--------|----------|------|
| 睡眠数据导入 | 更全面的恢复评估 | 支持CSV导入睡眠数据 |
| 睡眠质量与表现关联 | 了解睡眠对训练的影响 | 关联分析 |

### 3.4 用户交互设计

**新增命令规划**：

```bash
# 身体状态快速查看
uv run nanobotrun status today
uv run nanobotrun status weekly

# 心率分析
uv run nanobotrun analysis hrv --days 30
uv run nanobotrun analysis hr-recovery

# 疲劳度评估
uv run nanobotrun analysis fatigue
uv run nanobotrun analysis recovery
```

**AI 交互场景**：

| 场景 | 用户输入 | AI 响应 |
|------|----------|---------|
| 晨间检查 | "今天状态如何？" | 综合心率、疲劳度、恢复情况给出建议 |
| 异常询问 | "为什么最近静息心率变高了？" | 分析近期训练负荷，指出可能原因 |
| 训练决策 | "今天能跑间歇吗？" | 基于恢复状态给出是/否建议及理由 |

### 3.5 版本成功标准

| 维度 | 标准 | 测量方式 |
|------|------|----------|
| 功能完成 | P0功能100%实现，P1功能实现≥60% | 功能清单核对 |
| 用户价值 | 用户能读懂身体信号并做出训练决策 | 用户测试反馈 |
| 预警准确 | 异常信号预警准确率≥80% | 与主观感受对比验证 |
| 建议质量 | 训练建议被采纳率≥70% | 用户反馈统计 |
| 性能要求 | 身体状态查询<2秒 | 性能测试 |

### 3.6 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| HRV计算准确性 | 高 | 明确说明基于估算，非医疗级精度 |
| 疲劳度模型普适性 | 中 | 提供校准机制，让用户调整权重 |
| 用户过度依赖指标 | 中 | 强调"倾听身体"的重要性，指标仅作参考 |

| **疲劳度** | 综合当前训练负荷、恢复情况的评分 |
