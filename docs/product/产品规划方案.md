## 1. 战略对齐：我们为什么做这件事

### 1.1 产品愿景

**让严肃跑者拥有一个真正懂自己、数据完全可控的 AI 训练伙伴**

这不是一个"数据分析工具"，而是一个**能建立信任关系的训练伙伴**。技术只是手段，核心价值是：

| 战略维度 | 用户价值 | 不做会怎样 |
|----------|----------|------------|
| **隐私可控** | 跑者的数据只属于自己，不上传、不泄露 | 用户因隐私顾虑拒绝使用（Garmin/Strava的痛点） |
| **专业可信** | 基于运动科学的分析，不是黑箱建议 | 建议不可信，用户回归 TrainingPeaks |
| **个性适配** | AI 理解用户的训练偏好和身体响应模式 | 千篇一律的建议，用户失去使用动力 |
| **即时可得** | 终端内秒级响应，无需打开浏览器/APP | 体验断层，用户懒得用 |

> **1.x 战略聚焦**：单用户、本地运行、单智能体。不碰多租户、不碰 Web UI、不碰云端。把"一个人的 AI 训练伙伴"做到极致。

### 1.2 目标用户与核心场景

我们服务的是**"数据敏感的严肃跑者"**——他们佩戴心率带、记录每次训练、关心进步曲线，但厌倦了云端产品的隐私风险和千篇一律。

**三个核心场景（按优先级排序）**：

```
场景A「训练后复盘」——高频、高价值
  用户跑完步，想知道：这次训练效果如何？身体反馈说明什么？下周怎么调整？
  → 需要：即时分析、专业解读、可执行建议

场景B「赛前备赛」——中频、极高价值
  用户有比赛目标（如全马破4），需要：当前能力评估、训练计划、进度跟踪、比赛策略
  → 需要：计划生成、目标达成预测、动态调整

场景C「长期趋势洞察」——低频、信任建设
  用户想看看半年/一年的训练趋势，发现规律、验证假设
  → 需要：数据可视化、趋势分析、异常识别
```

**不做的事（明确边界）**：
- 不服务"偶尔跑步的健身用户"（他们不需要 VDOT、TSS）
- 不做社交/排行榜（与隐私优先战略冲突）
- 不做实时运动指导（不在跑步时干扰用户，跑后复盘才是核心价值）

## 2. 主题化路线图：我们要带用户去哪里

路线图不是"功能清单的时间表"，而是**用户价值递进的旅程**。每个版本解决一个核心用户问题，让用户感受到"又向前迈了一步"。

```
2026.05  v0.17.0  【主题】让 AI 真正"懂我"——角色化个性适配
         核心问题：用户觉得 AI 建议千篇一律，不够贴合自己的训练风格和目标

2026.06  v0.18.0  【主题】让数据"看得见、带得走"——可视化与可移植
         核心问题：用户看不懂原始数据，也无法把分析结果用于自己的工具链

2026.07  v0.19.0  【主题】让身体信号"会说话"——深度分析与自定义
         核心问题：用户知道数据里有规律，但不知道怎么挖掘、怎么验证自己的假设

2026.07  v0.20.0  【主题】让体验"稳如磐石"——性能与稳定性基线
         核心问题：工具响应慢、偶尔报错，破坏用户信任和使用习惯

2026.08  v1.0.0   【里程碑】"你的私人 AI 教练"——稳定、可信、可依赖
         核心承诺：用户可以放心地把训练分析交给 Nanobot，它不会出错、不会泄露、不会胡说

2026.Q4  v1.x     【方向】从"分析过去"到"预判未来"——主动预警与智能闭环
         核心探索：AI 能否在用户还没意识到问题时，提前给出干预建议？

2027.Q2  v2.0.0   【方向】从"一个伙伴"到"一个团队"——多智能体协作
         核心探索：多个专精 Agent 协作，是否比单个全能 Agent 更专业可信？
```

### 2.1 版本节奏与资源约束

**现实约束**：单人开发，每个版本周期 3-4 周，必须保证质量不退化。

| 阶段 | 版本 | 用户价值目标 | 开发周期 |
|------|------|-------------|----------|
| 功能完善 | v0.17 → v0.20 | 完成 v1.0 所需的全部用户价值 | 3个月 |
| 稳定化 | v0.20 → v1.0 | API 冻结、Bug 清零、文档完善 | 4周 |
| 成熟期 | v1.1 → v1.4 | 基于真实用户反馈迭代（方向待定） | 每4-6周 |

## 3. 版本规划：每个版本解决什么问题、带来什么改变

### 3.1 v0.17.0 — 让 AI 真正"懂我"（2026-05-20）

**用户问题**：
> "AI 给的建议太泛了。我是准备全马的严肃跑者，想要教练式的严格指导；但我朋友只是保持健康，他想要轻松鼓励式的交流。同一个 AI，怎么同时服务我们两个人？"

**我们要让用户感受到的变化**：
- 初始化时可以选择"我想要什么样的训练伙伴"
- 选择后，AI 的语气、建议风格、分析深度都会跟着变
- 切换角色后，AI 记得住之前聊过什么，不会"失忆"

**必须做（P0）——不做用户会流失**：
| 用户价值 | 为什么必须做 | 验证方式 |
|----------|-------------|----------|
| 4 种预置角色可选（严肃教练/数据分析师/康复顾问/陪伴型朋友） | 角色化是差异化核心，没有角色 AI 就是千篇一律的聊天机器人 | 用户能完成角色选择并完成一次对话 |
| `init --role` 初始化流程 | 用户第一次接触产品时的"啊哈时刻"，决定留存 | 新用户从安装到第一次获得角色化建议 < 3 分钟 |

**应该做（P1）——做了体验更好**：
| 用户价值 | 为什么重要 | 验证方式 |
|----------|-----------|----------|
| 角色切换不丢失历史 | 用户可能想尝试不同角色，丢失记忆会挫败 | 切换角色后，AI 能引用之前的训练记录 |
| CLI 错误提示更友好 | 用户输错命令时，知道怎么改，而不是看到堆栈跟踪 | 错误场景下用户能自助解决 |

**不做（本次版本）**：
- 角色自定义（让用户自己写角色模板）——使用频率低，先验证预置角色是否被接受
- 6 种角色——4 种足够覆盖 80% 场景，多了分散精力

**风险与缓解**：
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 角色隔离存储改造破坏现有记忆系统 | 中高 | 开发前写接口契约测试，渐进式重构 |
| 角色差异不够明显，用户感知不到 | 中 | 内部走查：同一问题，不同角色的回答是否一眼可区分 |

### 3.2 v0.18.0 — 让数据"看得见、带得走"（2026-06-10）

**用户问题**：
> "我知道你们分析了我的心率、配速、VDOT，但终端里一堆数字我看不懂趋势。我想看看我过去三个月的进步曲线，还想把数据导出来用 Python 自己分析。"

**我们要让用户感受到的变化**：
- 输入一个命令，终端里直接画出 VDOT 趋势图、心率分布图
- 周报/月报自动生成，关键信息一目了然
- 数据可以导出 CSV/JSON，用户可以用自己的工具继续分析

**必须做（P0）——不做产品不完整**：
| 用户价值 | 为什么必须做 | 验证方式 |
|----------|-------------|----------|
| 终端内直接显示图表（负荷趋势、心率分布、配速直方图） | 技术型跑者的核心诉求"数据可视化"，没有图表产品像半成品 | 图表渲染 < 2 秒，数据与原始记录一致 |
| CSV/JSON 数据导出 | "数据可移植"是隐私价值的延伸——用户不仅拥有数据，还能带走数据 | 导出文件可用 Excel/Python 正常读取 |
| 周报/月报自动生成 | 用户懒得每次手动查数据，定期推送降低使用门槛 | 报告数据与手动查询结果 100% 一致 |

**应该做（P1）——做了更完整**：
| 用户价值 | 为什么重要 | 验证方式 |
|----------|-----------|----------|
| Excel 导出 | 部分用户习惯 Excel，openpyxl 引入需谨慎验证兼容性 | 导出文件可用 Excel 打开，格式正确 |
| 本地 HTML 交互式报告 | 终端图表能力有限，HTML 可提供更丰富的交互体验 | 浏览器打开即可查看，无需服务器 |

**不做（本次版本）**：
- PDF 导出——使用场景少，HTML 可以打印为 PDF
- 图表自定义样式——先保证基础图表可用，再谈个性化

**风险与缓解**：
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 终端图表库（plotext）在某些终端显示异常 | 中 | 提供 Rich 表格 fallback，确保任何终端都能正常显示 |
| openpyxl 与现有依赖冲突 | 低 | 版本初期在隔离环境验证兼容性 |

### 3.3 v0.19.0 — 让身体信号"会说话"（2026-06-25）

**用户问题**：
> "我知道我的训练负荷数据里有规律，但你们只给了 ATL/CTL/TSB。我想知道：我最近是不是训练过度了？我的跑步经济性有没有进步？我能不能写个脚本自己分析一些你们没提供的指标？"

**我们要让用户感受到的变化**：
- 系统自动判断"最近是否训练过度"，给出明确结论
- 有功率计数据的用户，能看到自己的跑步经济性变化
- 技术型用户可以写 Python 脚本扩展分析能力

**必须做（P0）——不做深度分析名不副实**：
| 用户价值 | 为什么必须做 | 验证方式 |
|----------|-------------|----------|
| 疲劳度综合评估（基于 ATL/CTL/TSB 比值） | 用户最关心"我现在能不能跑高强度"，系统必须给出明确判断 | 与 TrainingPeaks 同类指标差异 < 5% |
| 自定义分析脚本框架 | 技术型用户的核心诉求——产品不可能覆盖所有指标，但可以给用户工具自己挖 | 用户能写一个脚本并成功运行，获得结果 |
| 性能基准测试套件 | 为 v0.20.0 性能优化建立可对比的基线，没有基准的优化是盲目的 | 覆盖导入/查询/报告/导出核心链路 |

**应该做（P1）——锦上添花**：
| 用户价值 | 为什么重要 | 验证方式 |
|----------|-----------|----------|
| 跑步经济性分析（RE = Speed/Power） | 功率计用户的高级需求，但渗透率不足 30%，不宜阻塞版本 | 有功率计数据的用户能正常计算 |

**不做（本次版本）**：
- 自动识别异常数据（GPS 漂移、心率带脱落）——属于 v1.x 的"主动预警"主题
- 相似跑者对比——需要积累更多用户反馈后才能设计合理的对比维度

**风险与缓解**：
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 自定义脚本框架存在安全风险 | 中 | 沙箱执行、限制文件系统访问范围 |
| 疲劳度算法改动影响现有训练负荷测试 | 低 | 基于现有模块扩展，保持向后兼容 |

### 3.4 v0.20.0 — 让体验"稳如磐石"（2026-07-15）

**用户问题**：
> "工具偶尔报错，有时候查询慢。我知道你们功能很多，但我更想要一个'不会出错、不会卡'的工具。"

**我们要让用户感受到的变化**：
- 任何命令都在 2 秒内响应
- 出错了能看到人话提示，知道怎么解决
- 文档完整，遇到问题能自助

**必须做（P0）——不做 v1.0 不敢叫稳定版**：
| 用户价值 | 为什么必须做 | 验证方式 |
|----------|-------------|----------|
| CLI 全局错误处理重构 | 用户遇到报错时的体验，直接决定信任度 | 所有错误场景都有友好提示，无裸堆栈 |
| 核心计算向量化改造（消灭 Python 循环） | 5 年数据量下查询不能卡，否则用户放弃使用 | 性能基准不劣于 v0.19.0，核心操作 < 2 秒 |
| API 冻结 + 文档 100% 覆盖 | v1.0 的承诺：API 稳定、文档齐全，用户可以放心依赖 | 文档与代码一致性检查通过 |

**技术债务清理（本次必须还）**：
| 债务 | 位置 | 不清理的后果 |
|------|------|-------------|
| TSS 计算仍用 Python 循环 | `analytics.py:442-448` | 数据量大时查询慢，用户体验差 |
| PlanManager 裸 Exception | `plan_manager.py:87-89` | 错误信息不可控，用户看不懂 |
| 报告模板硬编码简化值 | `generator.py:644-648` | 报告数据不准确，破坏信任 |

**风险与缓解**：
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 全局错误处理重构引入大规模回归 | 高 | 分模块逐步替换，每步执行全量 3127 测试 |
| 性能优化改变计算结果 | 中 | 优化前后数据对比，确保结果一致 |

### 3.5 v1.0.0 — "你的私人 AI 教练"（2026-08-15）

**不是"功能更多了"，而是"可以放心依赖了"**。

v1.0.0 的发布标准不是"所有功能做完了"，而是"用户是否愿意把训练分析全权交给 Nanobot"。

**发布承诺**：
| 承诺 | 验证标准 | 不达成的后果 |
|------|----------|-------------|
| 功能稳定 | 全量 3127 测试 100% 通过，无致命/严重 Bug | 用户遇到 Bug，信任崩塌 |
| API 可信 | 公共 API 冻结，向后兼容 | 用户不敢基于 Nanobot 做二次开发 |
| 数据安全 | 本地存储，零外联，导出格式标准 | 隐私承诺落空，核心差异化消失 |
| 响应迅速 | 核心操作 < 2 秒，查询 < 100ms/年 | 慢 = 不用 |
| 文档完整 | 100% 覆盖，用户自助解决问题 | 降低支持成本，提升满意度 |

**发布亮点（用户视角的故事）**：
> "我跑完步，输入一个命令，Nanobot 告诉我今天的训练负荷、心率漂移情况，还画出了我过去三个月的 VDOT 趋势。如果我想换种交流风格——严肃教练或轻松朋友——随时切换。所有数据都在我电脑上，我可以随时导出用 Python 分析。它不会出错，不会泄露，不会胡说。"

**稳定化期（v0.20 → v1.0）的 4 周做什么**：
- 2 周：Bug 清零 + 性能压测 + 跨平台验证
- 1 周：RC（Release Candidate）用户试用（如果有早期用户）
- 1 周：文档最终校对 + 发布流程

## 4. v1.x 及以后：方向而非清单

> **原则**：v1.0 之后的规划只定"主题方向"和"要回答的问题"，不做详细功能清单。具体做什么，由 v1.0 发布后的真实用户反馈决定。

### 4.1 v1.x 主题方向（2026.Q4）

**主题**：从"分析过去"到"预判未来"

**要回答的核心问题**：
- AI 能否在用户还没意识到问题时，提前给出干预建议？（如：TSB 持续走低时主动建议休息）
- 用户是否愿意接受"主动推送"，还是只喜欢"被动查询"？
- 角色化是否成功？用户是否真的切换角色，还是始终用一个？

**可能探索的方向（不做承诺）**：
- 主动预警机制（基于训练数据的异常检测）
- 建议-执行-反馈闭环（训练后收集 RPE，AI 评估建议效果）
- 自然语言数据查询（"我去年秋天 10km 以上的长距离平均配速是多少？"）

### 4.2 v2.0.0 主题方向（2027.Q2）

**主题**：从"一个伙伴"到"一个团队"

**要回答的核心问题**：
- 多个专精 Agent（训练/分析/报告/提醒）协作，是否比单个全能 Agent 更专业可信？
- 多 Agent 的协调成本是否值得？
- 用户是否感知到"团队"比"个人"更好？

**前提条件**：
- v1.x 积累足够的用户反馈，证明"单 Agent 天花板已现"
- OpenHarness/HKUDS 框架成熟可用

## 5. 优先级决策逻辑：为什么先做 A 而不是 B

### 5.1 优先级框架

每个功能的优先级由三个维度决定：

```
优先级 = 用户价值 × 战略契合度 ÷ 实现成本

- 用户价值：不做这个功能，用户会不会流失/失望？
- 战略契合度：这个功能是否强化"隐私优先、专业可信、个性适配、即时可得"？
- 实现成本：开发周期、技术风险、对现有代码的影响范围
```

### 5.2 关键决策说明

| 决策 | 选择 | 理由 |
|------|------|------|
| **为什么 v0.17 做角色化，而不是先做可视化？** | 角色化是差异化核心 | 没有角色化，Nanobot 只是另一个数据分析工具；没有可视化，用户还能看数字。角色化决定"用户为什么选你" |
| **为什么 CLI 错误处理提前到 v0.17，而不是留在 v0.20？** | 错误处理是高频触点 | 用户每次输错命令都遇到糟糕体验，累积的负面印象会抵消所有功能价值 |
| **为什么疲劳度评估是 P0，跑步经济性是 P1？** | 用户覆盖面 | 所有用户都关心"是否训练过度"；只有 30% 用户有功率计，跑步经济性受众窄 |
| **为什么 v1.x 不做详细规划？** | 避免假设驱动 | v1.0 发布后的真实用户反馈，可能比现在的假设更准确。规划太远 = 规划作废 |

## 6. 风险与依赖：只关心当前和下一步

### 6.1 当前版本（v0.17.0）的关键风险

| 风险 | 等级 | 触发条件 | 缓解措施 |
|------|------|----------|----------|
| 角色隔离存储改造破坏记忆系统 | 中 | MemoryManager 重构时 | 接口契约测试 + 渐进式重构 |
| 角色差异不够明显 | 中 | 内部走查不通过 | 推迟发布，强化角色 prompt 差异 |
| nanobot-ai Agent 角色机制变动 | 低 | 底座框架更新 | 锁定底座版本，v0.19 后再考虑升级 |

### 6.2 下一版本（v0.18.0）的前置依赖

| 依赖 | 来源 | 不达成的后果 |
|------|------|-------------|
| v0.17.0 角色存储结构稳定 | v0.17.0 | v0.18.0 的报告模板无法按角色差异化展示 |
| plotext 终端兼容性验证 | 外部库 | 图表在某些终端显示异常，需 fallback 方案 |

### 6.3 版本延期规则

1. **P0 功能未达标 → 版本延期**：P0 是"不做就失败"，不能妥协
2. **P1 功能未达标 → 可裁剪**：P1 是"做了更好"，可以移到下一版本
3. **质量门禁未通过 → 必须延期**：测试覆盖率、全量测试通过率是硬门槛
4. **延期超过 2 周 → 后续版本顺延**：保持节奏，不赶工

## 7. 验证与闭环：怎么知道我们做对了

### 7.1 每个版本的成功标准

不是"功能做完了"，而是"用户感受到了价值"。

| 版本 | 成功标准（用户视角） | 测量方式 |
|------|---------------------|----------|
| v0.17.0 | 用户能选择一个角色，并感受到 AI 风格的变化 | 角色切换后对话风格差异度评分 ≥ 4/5 |
| v0.18.0 | 用户能一眼看懂自己的训练趋势，并能导出数据 | 图表理解度测试 + 导出文件可用性验证 |
| v0.19.0 | 用户能得到"是否训练过度"的明确结论 | 疲劳度判断与专家判断一致性 ≥ 80% |
| v0.20.0 | 用户觉得"这个工具很快、很稳" | 核心操作响应时间 < 2 秒，错误率 < 0.1% |
| v1.0.0 | 用户愿意推荐给别人 | NPS ≥ 30（如果有条件测量） |

### 7.2 持续校准机制

| 机制 | 频率 | 责任人 |
|------|------|--------|
| 版本复盘 | 每个版本发布后 1 周内 | 产品经理 |
| 用户反馈收集 | 持续（GitHub Issues + CLI 内置反馈） | 产品经理 |
| 竞品动态跟踪 | 每月 | 产品经理 |
| 技术债务评估 | 每个版本规划前 | 技术负责人 |

**版本复盘三问**：
1. 用户是否获得了我们承诺的价值？（功能完成率 + 用户反馈）
2. 质量是否退步了？（Bug 引入率 + 测试通过率）
3. 下一版本的假设还成立吗？（优先级是否需要调整）

### v4.1 核心改进

1. **从"功能清单"到"用户场景"**：每个版本以"用户遇到什么问题"开头，而非"我们要做什么功能"
2. **从"版本号列表"到"主题化旅程"**：版本命名用"让用户能做什么"而非"我们做什么"
3. **从"P0/P1 标签"到"可解释优先级"**：每个 P0 说明"不做用户会流失/产品会失败"的理由
4. **从"详细功能表"到"方向性探索"**：v1.x 及以后只定主题和问题，不做功能承诺
5. **从"风险罗列"到"当前+下一步聚焦"**：只关心当前版本和下一版本的风险，不过度预判
6. **增加"验证与闭环"**：每个版本定义"怎么知道做对了"，建立持续校准机制
