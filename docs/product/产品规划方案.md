## 1. 产品愿景

**成为技术型跑者首选的本地化 AI 跑步数据分析工具**

| 维度 | 价值主张 |
|------|----------|
| 隐私优先 | 本地存储，零外联设计，数据完全用户可控 |
| 专业分析 | VDOT、TSS、心率漂移等运动科学指标 |
| AI 赋能 | 自然语言交互，智能训练建议，可定制角色 |
| 高性能 | Polars + Parquet，查询性能优异 |
| 数据可移植 | 支持 CSV/JSON/Excel 导出，用户拥有完整数据主权 |

### 1.1 目标用户分层

| 层级 | 画像 | 占比预估 | 核心诉求 | 1.x 支持策略 |
|------|------|----------|----------|-------------|
| 核心层 | 技术型严肃跑者（程序员/工程师，规律跑步2年+，有马拉松经历） | 30% | 深度分析、隐私、可编程、CLI 操控 | 完整功能，专业模式 |
| 扩展层 | 非技术背景但数据敏感的严肃跑者 | 50% | 专业报告、简单交互、隐私保护 | 向导式 CLI、报告导出 |
| 潜在层 | 进阶业余跑者（1-2年经验） | 20% | 成长追踪、计划指导、易用性 | 基础分析、计划生成 |

> **1.x 系列核心定位**：面向个人用户群体，专注于单智能体功能方向的开发与优化。

## 5. 技术基线

### 5.1 业务规则

| 规则 | 内容 |
|------|------|
| VDOT计算 | Powers公式，距离≥1500m |
| TSS计算 | 时长×IF²×100 |
| 心率漂移 | 相关性<-0.7 |
| 存储 | Parquet按年分片，SHA256去重 |
| 时长格式 | HH:MM:SS |
| 配速格式 | M'SS"/km |

### 5.2 质量门禁

| 类别 | 规则 |
|------|------|
| 代码规范 | ❌ 禁止硬编码敏感信息 |
| 调试规范 | ❌ 禁止print调试，统一使用日志框架 |
| 架构规范 | ❌ 禁止直接实例化核心组件（必须通过依赖注入） |
| 性能规范 | ❌ 禁止在LazyFrame中过早调用`.collect()` |
| 类型规范 | ❌ 禁止使用裸`Exception`，必须使用自定义异常 |
| 覆盖率 | core≥80%、agents≥70%、cli≥60% |
| 类型检查 | mypy零错误 |
| 代码质量 | ruff零警告 |

### 5.3 技术栈版本管理

| 技术 | 当前版本 | v0.19.0冻结 | v1.0.0承诺 |

| R03 | 角色隔离存储破坏现有记忆测试 | 中 | 开发前编写接口契约测试 | 开发负责人 |
| R04 | openpyxl依赖冲突 | 低 | v0.18.0初期验证兼容性 | 开发负责人 |
| R05 | v1.0.0稳定版定义与市场预期不符 | 中 | RC阶段收集早期用户反馈 | 产品负责人 |
| R06 | v2.0.0多智能体技术跨度大 | 中 | 2027.Q2前持续技术预研 | 架构师 |

### 6.3 版本延期策略

1. **允许延期**：任何版本未达到质量门禁，自动延期至达标
2. **功能裁剪**：P1功能可裁剪至下一版本，P0功能不可裁剪
3. **日期联动**：某版本延期超过2周，后续版本日期顺延
4. **v1.0.0特殊规则**：必须满足所有质量门禁，无例外
