## 1. 战略对齐：底座驱动的产品演进

### 1.1 产品愿景（重新定义）

> **"让拥有2年+跑步数据的资深跑者，拥有一个真正懂自己、能预测未来、数据完全可控的 AI 训练科学家——从数据沉淀到预测洞察，从经验训练到智能优化，全程本地、全程透明、全程可控。"**

这不是一个"数据分析工具"，而是一个**能建立信任关系的AI训练科学家**。技术只是手段，核心价值是：

| 战略维度 | 用户价值 | 市场盲点 | 不做会怎样 |
|----------|----------|----------|------------|
| **隐私可控** | 跑者的数据只属于自己，不上传、不泄露 | Strava/Garmin数据泄露事件频发，用户数据被用于AI训练 | 用户因隐私顾虑拒绝使用 |
| **专业可信** | 基于运动科学的分析，AI建议可追溯、可验证 | 云端AI建议黑箱，不可解释 | 建议不可信，用户回归 TrainingPeaks |
| **预测未来** ⭐ | **基于2年+数据，预测伤病风险、VDOT趋势、比赛成绩** | **现有产品仅事后分析，无预测能力** | **无法差异化，陷入功能同质化竞争** |
| **智能优化** ⭐ | **RL驱动的个性化训练负荷推荐，真正"因人而异"** | **通用模型，非个性化，无法适应个人训练响应** | **建议千篇一律，用户感受不到价值** |
| **底座原生** | 紧跟nanobot-ai底座演进，不重复造轮子 | 其他产品自建AI框架，维护成本高 | 技术债务累积，无法持续迭代 |
| **即时可得** | 终端内秒级响应，无需打开浏览器/APP | 云端产品响应慢、需要联网 | 体验断层，用户懒得用 |

### 1.2 战略调整说明

**放弃原规划**：
- ❌ 1.x单智能体 → 2.x多智能体的线性演进路线
- ❌ 预设"严肃教练/数据分析师"等固定角色化方案
- ❌ 脱离底座能力独立发展AI框架

**新战略核心**：
- ✅ **底座跟随策略**：nanobot-ai底座支持什么，我们就用什么，不重复造轮子
- ✅ **渐进式智能化**：从工具辅助 → 单Agent助手 → 多Agent协作，随底座能力自然演进
- ✅ **场景驱动**：不预设角色，而是根据用户实际使用场景动态适配

### 1.3 底座能力映射

| nanobot-ai底座能力 | 版本 | 我们的产品应用 |
|-------------------|------|---------------|
| MCP协议支持 | v0.1.3+ | 工具生态接入（天气、地图、健康数据）✅已具备 |
| 流式输出 | v0.1.4+ | CLI实时显示AI思考过程✅已具备 |
| Subagent调用 | v0.1.4+ | 多Agent协作基础能力验证✅已具备 |
| A2A协议 | RFC阶段 | 未来多Agent协作架构（待底座成熟） |
| 向量记忆 | 规划中 | 长期训练知识沉淀（跟随底座） |
| 计划任务 | 规划中 | 主动训练提醒（跟随底座） |

> **当前底座版本**: nanobot-ai >=0.1.5.post3（2026-04-29发布，包含ask_user工具、Memory调优、Session持久性增强等能力）

## 2. 用户痛点与市场盲点分析

### 2.1 严肃跑者的核心痛点

基于市场调研（Strava隐私事件、Garmin数据政策、TrainingPeaks用户反馈）：

```
痛点1：数据隐私焦虑（高频、高敏感）
  "Strava又把我的训练数据用于AI训练了"
  "Garmin的隐私设置太复杂，我不知道哪些数据被分享了"
  → 需要：本地存储、零外联、数据完全可控

痛点2：AI建议不可信（中频、高影响）
  "TrainingPeaks的建议我看不懂，不知道是怎么算出来的"
  "Strava的AI建议太泛，不适合我的情况"
  → 需要：可解释的AI、基于运动科学的分析

痛点3：数据孤岛（中频、高阻碍）
  "我在Garmin、Strava、TrainingPeaks都有数据，分散在各处"
  "想导出数据自己分析，格式不兼容"
  → 需要：标准格式存储、易于导出

痛点4：专业分析门槛（低频、高价值）
  "VDOT、TSS、心率漂移这些指标我想用，但学习曲线太陡"
  → 需要：AI自然语言交互，降低专业分析门槛
```

### 2.2 市场产品盲点

| 产品 | 优势 | 盲点 | 我们的机会 |
|------|------|------|-----------|
| **Strava** | 社交功能强、用户基数大 | 隐私问题频发、数据用于AI训练、过度社交压力 | 专注个人分析，无社交功能，数据本地存储 |
| **Garmin Connect** | 硬件生态完整 | 云端依赖、隐私设置复杂、分析功能有限 | 更深度的数据分析、更透明的AI建议 |
| **TrainingPeaks** | 专业分析功能强 | 学习曲线陡峭、价格昂贵、界面复杂 | AI自然语言交互，降低使用门槛 |
| **Intervals.icu** | 免费、功能丰富 | 需要上传数据到云端、界面不够友好 | 本地部署、CLI交互、数据可控 |

### 2.3 目标用户画像

**主要用户：数据敏感的严肃跑者**

| 属性 | 描述 |
|------|------|
| **年龄段** | 25-45 岁 |
| **职业背景** | 程序员、工程师、数据分析师等技术从业者 |
| **跑步经验** | 规律跑步 2 年以上，有马拉松参赛经历 |
| **数据意识** | 习惯佩戴心率带、功率计等专业设备 |
| **核心痛点** | 担心云端数据隐私，需要更专业的数据分析 |
| **技术能力** | 熟悉命令行操作，具备一定编程基础 |
| **价值观** | 重视数据隐私，愿意为隐私牺牲一定便利性 |

## 4. 版本规划：每个版本解决什么问题

### 4.1 v0.17.0 — 底座能力全面激活（2026-05-20）

**用户问题**：
> "AI响应有点慢，而且看不到它在想什么。另外，我想知道未来能不能有更智能的协作能力？还有，能不能定时提醒我训练？"

**我们要让用户感受到的变化**：
- AI响应变快了，能看到实时输出（激活底座流式输出能力）
- 为未来多Agent能力做好准备（验证底座Subagent能力）
- 定时收到训练提醒，不再错过训练（激活底座Cron能力）
- CLI错误提示更友好，遇到问题能自助解决

**底座能力全景与激活状态**：

| 能力模块 | 底座组件 | 当前状态 | v0.17.0目标 |
|----------|----------|----------|-------------|
| Agent核心 | `AgentLoop` | ✅ 已激活 | 保持 |
| 消息总线 | `MessageBus` | ✅ 已激活 | 保持 |
| MCP工具 | `connect_mcp_servers` | ✅ 已激活 | 保持 |
| Cron定时任务 | `CronService` | ⚠️ 部分激活 | **深度激活** |
| Heartbeat心跳 | `HeartbeatService` | ⚠️ 部分激活 | **扩展场景** |
| 流式输出 | `on_stream` Hook | ⚠️ 已注册未实现 | **完整实现** |
| Subagent调用 | `call_subagent` | ❌ 未激活 | **基础验证** |
| Agent Hook | `AgentHook` | ✅ 已激活 | **增强实现** |

**必须做（P0）——底座能力全面激活**：

| 用户价值 | 底座能力 | 实现方案 | 验证方式 |
|----------|----------|----------|----------|
| **CLI流式输出** | `on_stream` Hook | 实现Hook方法，通过Rich Console实时输出 | 响应时间<3秒，流式效果流畅 |
| **Subagent基础验证** | `call_subagent` | 创建2个Subagent（数据分析+报告生成），验证调用链 | 主Agent能调用Subagent并获取结果 |
| **Cron训练提醒** | `CronService` | 添加每日训练提醒、周期性报告推送 | 定时任务准时触发，消息送达 |
| **CLI错误处理重构** | `AgentHook` + 自定义 | 统一错误处理，结合Hook记录错误上下文 | 所有错误场景都有友好提示 |

**应该做（P1）——体验增强**：

| 用户价值 | 底座能力 | 实现方案 | 验证方式 |
|----------|----------|----------|----------|
| **工具执行进度显示** | `before_execute_tools` Hook | 在工具调用前显示进度指示器 | 用户知道AI正在调用什么工具 |
| **Heartbeat任务扩展** | `HeartbeatService` | 扩展HEARTBEAT.md任务自动执行 | 周期性任务自动触发 |
| **多通道配置优化** | `ChannelManager` | 统一飞书/微信/CLI配置 | 新用户配置时间<5分钟 |
| **ask_user训练确认** | `ask_user` 工具 | 训练计划生成后让用户确认/调整 | 用户可通过按钮选择确认或修改 |
| **LLM超时控制** | `NANOBOT_LLM_TIMEOUT_S` | 配置超时防止请求挂起 | 超时后自动重试或提示用户 |

**不做（本次版本）**：
- 角色化系统（等待底座更成熟的Persona支持）
- 复杂的多Agent协作生产化（仅验证基础能力）
- 工具并发执行优化（P2优先级，后续版本考虑）

### 4.2 v0.18.0 — 让数据"看得见、带得走"（2026-06）

**用户问题**：
> "我知道你们分析了我的心率、配速、VDOT，但终端里一堆数字我看不懂趋势。我想看看我过去三个月的进步曲线，还想把数据导出来用 Python 自己分析。"

**核心价值**：终端内可视化 + 数据可导出

**版本目标**：让跑者能直观看到训练趋势，并能自由导出数据进行二次分析

#### 4.2.1 用户价值与功能映射

| 用户价值 | 功能支撑 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| 一眼看懂训练趋势 | 终端图表显示（VDOT/CTL/ATL/TSB趋势） | P0 | 图表清晰可读，支持时间范围筛选 |
| 数据自由导出 | CSV/JSON/Parquet多格式导出 | P0 | 导出文件可直接用Pandas读取 |
| 周期性训练总结 | 周报/月报自动生成 | P0 | 报告包含关键指标变化趋势 |
| 更丰富的报告格式 | Excel导出、本地HTML报告 | P1 | Excel含图表，HTML可离线查看 |
| 自定义可视化 | 支持用户指定指标组合图表 | P1 | 用户可配置2-4个指标对比显示 |

#### 4.2.2 P0功能详细规划

**1. 终端图表显示系统**

| 功能项 | 实现方案 | 技术选型 | 验收标准 |
|--------|----------|----------|----------|
| VDOT趋势图 | 基于Rich的文本图表 + Matplotlib ASCII渲染 | `rich` + `plotext`/`matplotlib` | 支持7/30/90/365天时间范围 |
| 训练负荷曲线 | CTL/ATL/TSB三线同屏显示 | 终端内多线图 | 曲线清晰，峰值标注 |
| 心率区间分布 | 堆叠柱状图显示各区间时长占比 | 文本图表 | 百分比精确到1% |
| 配速分布热力图 | 按周/时段统计配速分布 | 热力图矩阵 | 支持自定义配速区间 |
| 周跑量趋势 | 柱状图显示每周跑量变化 | 文本柱状图 | 标注最高/最低周 |

**技术选型说明**：
- 主方案：`rich`库（已集成）+ `plotext`（纯Python，无依赖）
- 备选方案：`matplotlib`的ASCII渲染（更精确但依赖重）
- 决策理由：保持CLI轻量，不引入重量级图形库

**2. 数据导出系统**

| 功能项 | 导出格式 | 数据范围 | 验收标准 |
|--------|----------|----------|----------|
| 训练记录导出 | CSV/JSON/Parquet | 按日期范围筛选 | 导出字段完整，格式标准 |
| 指标摘要导出 | CSV/JSON | 聚合指标（周/月/年） | 可直接用于Excel分析 |
| VDOT历史导出 | CSV | 全部VDOT计算记录 | 含计算日期、距离、配速 |

## 6. 优先级决策逻辑

### 6.1 优先级框架

每个功能的优先级由三个维度决定：

```
优先级 = 用户价值 × 战略契合度 ÷ 实现成本

- 用户价值：不做这个功能，用户会不会流失/失望？
- 战略契合度：这个功能是否强化"隐私优先、底座原生、渐进演进"？
- 实现成本：开发周期、技术风险、对现有代码的影响范围
```

### 6.2 关键决策说明

| 决策 | 选择 | 理由 |
|------|------|------|
| **为什么v0.17先做底座升级，而不是先做可视化？** | 底座升级是技术基础 | 没有底座新能力，后续功能无法展开；底座升级有版本窗口，错过可能需要大幅重构 |
| **为什么放弃角色化系统？** | 等待底座更成熟的Persona支持 | 自建角色化系统与底座未来能力可能冲突，且增加维护负担 |
| **为什么不做详细的多Agent规划？** | 底座A2A协议还在RFC阶段 | 规划太远=规划作废，等待底座成熟后再决策 |
| **为什么疲劳度评估是P0，跑步经济性是P1？** | 用户覆盖面 | 所有用户都关心"是否训练过度"；只有30%用户有功率计 |

| **痛点** | "知道怎么练但不知道是否适合自己" | AI能记住偏好，越用越懂我 |
| **技术接受度** | 高，愿意尝试新方法 | 自适应AI，无需重复配置 |

### B.3 市场竞品盲点

| 产品 | 盲点 | 我们的类RL机会 |
|------|------|----------------|
| **TrainingPeaks** | 静态模板，无法学习用户偏好 | Dream记忆：记住偏好，自动应用 |
| **Strava** | 黑盒算法，用户无法反馈优化 | 反馈闭环：用户反馈直接改进建议 |
| **Intervals.icu** | 需手动配置，无自适应能力 | 自适应行为：自动调整输出风格 |
| **Garmin** | 计划僵化，无法复用自定义方案 | 技能复用：保存和复用个人训练方案 |

### B.4 版本激活路线

| 版本 | 主题 | 激活能力 | 核心目标 |
|------|------|----------|----------|
| v0.21.0 | 让AI"记住"用户 | Dream + 偏好学习 | 自动提取训练偏好，持久化应用 |
| v0.22.0 | 让AI"听得懂反馈" | 反馈闭环 + 自适应行为 | 收集反馈并调整建议，自适应报告风格 |
| v1.1.0 | 让AI"积累经验" | 技能复用 | 训练计划技能化，效果评估与进化 |
| v1.2.0+ | 全能力协同 | 全能力组合 | 个人训练知识库，预测性建议 |

### B.5 方案对比：底座类RL vs 自建ML/RL

| 维度 | 底座类RL方案（采用） | 自建ML/RL方案（废弃） |
|------|---------------------|---------------------|
| **实现复杂度** | 低（复用底座能力） | 高（需自建LSTM/XGBoost/PPO模型） |
| **维护成本** | 低（跟随底座升级） | 高（需持续维护） |
| **开发周期** | 短（3个月） | 长（6个月+） |
| **与底座一致性** | 高（原生集成） | 低（外部系统） |
| **项目基础** | 已集成dream_integration/feedback_loop等模块 | 需从零搭建 |

**核心转变**：从"自建ML/RL模型做预测"转向"利用底座类RL能力实现持续进化的个性化AI"。
