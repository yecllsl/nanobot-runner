## 2. 变更内容详解

### 2.1 Fixed（4 项关键修复）

#### 修复 1：进程爆炸（pytest-xdist 嵌套并行）

**根因**: `ThreadPoolExecutor(max_workers=4)` + `pytest -n 4` 双层并行叠加，导致 pytest-xdist 启动 4 个 worker，每个 worker 各自加载完整依赖树（polars/pyarrow/sklearn），峰值 4 × 4.5GB ≈ 18GB。

**修复**:
- 移除 `pytest` 命令中的 `-n 4` 参数（4 处：默认配置/全量命令/增量命令/相关路径）
- 引入两阶段调度，重量级检查（mypy/bandit/pytest）改为串行执行

**验证**: 单元测试 `test_peak_concurrency_under_threshold` 守护峰值并发 ≤ max_workers + 1 = 5

#### 修复 2：双进程并发（缺乏单实例锁）

**根因**: `.git/hooks/pre-commit` 与手动 `python scripts/pre-commit-check.py` 可同时触发，两个进程并行执行导致内存叠加。

**修复**: 新增 `SingleInstanceLock` 类
- 锁文件位置: `~/.nanobot-runner/.pre-commit.lock`
- 文件格式: `{PID}\n{ISO时间戳}`
- 行为: PID 仍存活时拒绝新实例，超时（默认 300s）后视为过期锁自动覆写

**验证**: 5 个单元测试（TestSingleInstanceLock）覆盖锁获取/拒绝/过期/PID 记录/释放清理

#### 修复 3：Windows 路径解析（shlex 转义误吃）

**根因**: `shlex.split(posix=True)` 把 `scripts\diagnose_process_tree.py` 中的 `\d` 当作转义序列并吃掉反斜杠，导致路径解析为 `scriptsdiagnose_process_tree.py`，ruff 找不到文件。

**修复**:
1. 构造命令时使用 `f.relative_to(self.project_root).as_posix()` 强制正斜杠
2. `shlex.split` 前用 `command.replace("\\", "/")` 规范化路径

**验证**: 单元测试 `test_run_command_normalizes_windows_backslash_to_posix` 守护

#### 修复 4：shell=True 冗余进程

**根因**: Windows 上 `subprocess.run(cmd, shell=True)` 每个检查会额外启动一个 `cmd.exe` 包装进程，6 个检查 → 6 个冗余进程。

**修复**: `subprocess.run(shlex.split(command), shell=False)` 直接调用，避免 cmd.exe 中间层。

**验证**: 2 个单元测试（TestShellFalse）覆盖 `run_command` 和 `auto_fix_issues` 两条路径

## 8. 经验总结

### 8.1 关键技术决策

1. **方法论选择**: 选用 systematic-debugging 而非直接打补丁，成功识别 4 大独立根因
2. **并发策略**: 两阶段调度（轻量并行 + 重量级串行）取代全局并行，避免嵌套并行
3. **跨平台处理**: 用 `as_posix()` 在源头强制正斜杠，避免下游 shlex 误解析
4. **单实例保护**: 文件锁 + PID 记录 + 超时过期，跨平台兼容且自愈

### 8.2 监控与诊断工具

- `verify_memory_peak.py` —— 用 psutil 量化内存峰值
- `diagnose_process_tree.py` —— 进程树结构可视化
- `pytest` 单元测试 —— 用线程并发计数器直接验证根因消除

### 8.3 可复用经验

- 进程级并发问题优先用 `psutil` 量化，而非依赖直觉
- 单实例锁使用文件锁而非网络端口，简化跨平台实现
- Windows 路径处理统一在源头（`as_posix()`），避免在多个 shlex 调用点重复修复
