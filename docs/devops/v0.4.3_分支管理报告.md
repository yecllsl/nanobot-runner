## 🔄 分支管理策略

### 主分支（main）

- **用途**: 生产环境代码
- **保护级别**: 高
- **合并来源**: 仅接受来自develop或hotfix分支的合并
- **当前版本**: v0.4.1

### 开发分支（develop）

- **用途**: 日常开发集成分支
- **保护级别**: 中
- **合并来源**: 接受来自feature、fix分支的合并
- **当前版本**: v0.4.3

### 功能分支（feature/*）

- **用途**: 新功能开发
- **命名规范**: feature/功能名称
- **生命周期**: 短期，开发完成后合并到develop并删除

### 修复分支（fix/*）

- **用途**: Bug修复
- **命名规范**: fix/问题描述
- **生命周期**: 短期，修复完成后合并到develop并删除

### 热修复分支（hotfix/*）

- **用途**: 生产环境紧急修复
- **命名规范**: hotfix/版本号
- **生命周期**: 短期，修复完成后合并到main和develop并删除

## 📈 分支管理最佳实践

### 1. 分支命名规范

- **feature分支**: feature/功能名称（如：feature/user-profile）
- **fix分支**: fix/问题描述（如：fix/login-error）
- **hotfix分支**: hotfix/版本号（如：hotfix/0.4.3）
- **release分支**: release/版本号（如：release/0.4.3）

### 2. 分支生命周期

```
feature/fix分支 → develop分支 → release分支 → main分支
                    ↑              ↓
                    └── hotfix分支 ──┘
```

### 3. 分支保护规则

- **main分支**: 
  - 禁止直接提交
  - 必须通过PR合并
  - 需要至少1个审核通过
  - 必须通过CI检查

- **develop分支**:
  - 禁止直接提交
  - 必须通过PR合并
  - 需要至少1个审核通过
  - 必须通过CI检查

### 4. 分支清理策略

- **功能分支**: 合并后立即删除
- **修复分支**: 合并后立即删除
- **热修复分支**: 合并后立即删除
- **发布分支**: 发布后保留一段时间，确认无问题后删除
