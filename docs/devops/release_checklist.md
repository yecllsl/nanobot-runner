# 发布检查清单

## 1. 发布前检查（Pre-Release Checklist）

### 1.1 代码质量检查

#### 代码格式化
- [ ] 执行 `uv run black src tests --check`
- [ ] 执行 `uv run isort src tests --check-only`
- [ ] 确认无格式化警告

#### 类型检查
- [ ] 执行 `uv run mypy src --ignore-missing-imports`
- [ ] 确认类型检查通过
- [ ] 核心模块类型覆盖率≥80%

#### 安全扫描
- [ ] 执行 `uv run bandit -r src -s B101,B601`
- [ ] 确认无高危安全漏洞
- [ ] 中低危漏洞已评估风险

### 1.2 测试验证

#### 单元测试
- [ ] 执行 `uv run pytest tests/unit/ --cov=src --cov-fail-under=80`
- [ ] 确认测试通过率100%
- [ ] 代码覆盖率达标

#### 集成测试
- [ ] 执行 `uv run pytest tests/integration/`
- [ ] 确认集成流程正常
- [ ] 数据流验证通过

#### 端到端测试
- [ ] 执行 `uv run pytest tests/e2e/`
- [ ] 确认用户场景正常
- [ ] 性能指标达标

### 1.3 文档更新

#### 发布文档
- [ ] 更新 `docs/devops/release_notes/vX.Y.Z.md`
- [ ] 包含功能更新说明
- [ ] 包含已知问题说明

#### 用户文档
- [ ] 更新 `docs/guides/cli_usage.md`
- [ ] 更新 API 文档
- [ ] 更新配置指南

### 1.4 配置验证

#### 版本号管理
- [ ] 更新 `pyproject.toml` 中的版本号
- [ ] 确认版本号符合语义化版本规范
- [ ] 版本号与发布分支一致

#### 依赖检查
- [ ] 执行 `uv sync --all-extras`
- [ ] 确认依赖版本兼容
- [ ] 更新依赖变更说明

## 2. 发布执行（Release Execution）

### 2.1 分支管理

#### 发布分支创建
- [ ] 从 develop 分支创建 release 分支
- [ ] 分支命名：`release/vX.Y.Z`
- [ ] 推送 release 分支到远程

#### 最终验证
- [ ] 在 release 分支执行完整测试套件
- [ ] 验证生产环境配置
- [ ] 确认无回归问题

### 2.2 代码合并

#### PR 创建与审查
- [ ] 创建 Release PR（release → main）
- [ ] 至少1名核心开发者审查
- [ ] 确认所有检查通过

#### 合并操作
- [ ] 使用 `Squash and merge` 或 `Rebase and merge`
- [ ] **禁用** `--delete-branch` 选项
- [ ] 保留合并提交记录

### 2.3 标签管理

#### 标签创建
- [ ] 切换到 main 分支：`git checkout main`
- [ ] 拉取最新代码：`git pull origin main`
- [ ] 创建标签：`git tag -a vX.Y.Z -m "Release vX.Y.Z"`

#### 标签推送
- [ ] 推送标签：`git push origin vX.Y.Z`
- [ ] 确认标签推送成功
- [ ] 验证远程标签存在

## 3. 发布后验证（Post-Release Validation）

### 3.1 CI/CD 流程监控

#### GitHub Actions 状态
- [ ] 监控 release.yml workflow 执行状态
- [ ] 确认构建成功（绿色状态）
- [ ] 检查构建日志无错误

#### 发布产物验证
- [ ] 访问 GitHub Releases 页面
- [ ] 确认包文件正常上传
- [ ] 验证包文件完整性

### 3.2 环境同步

#### 分支同步
- [ ] 切换到 develop 分支：`git checkout develop`
- [ ] 同步 main 分支内容：`git merge main`
- [ ] 解决可能的合并冲突
- [ ] 推送更新：`git push origin develop`

#### 环境验证
- [ ] 在 develop 分支验证功能正常
- [ ] 确认无回归问题
- [ ] 更新开发环境配置

### 3.3 通知与文档

#### 团队通知
- [ ] 发送发布完成通知
- [ ] 包含发布说明链接
- [ ] 提供回滚指南（如需要）

#### 文档归档
- [ ] 归档发布相关文档
- [ ] 更新项目状态报告
- [ ] 记录发布经验教训

## 4. 紧急情况处理（Emergency Procedures）

### 4.1 发布回滚

#### 回滚条件
- 生产环境严重问题
- 安全漏洞暴露
- 数据丢失风险

#### 回滚步骤
1. 创建 hotfix 分支：`git checkout -b hotfix/rollback-vX.Y.Z main`
2. 回滚代码变更
3. 创建紧急发布标签
4. 执行紧急发布流程

### 4.2 问题响应

#### 响应时间要求
- **严重问题**：30分钟内响应
- **一般问题**：2小时内响应
- **功能请求**：24小时内评估

#### 沟通渠道
- 团队聊天群组
- 问题跟踪系统
- 邮件通知列表

## 5. 检查清单使用说明

### 5.1 使用时机

- **常规发布**：每次发布前必须完成
- **紧急发布**：简化流程，重点关注核心检查项
- **预发布验证**：用于发布候选版本验证

### 5.2 责任分配

| 检查项类别 | 主要责任人 | 辅助责任人 |
|-----------|----------|----------|
| 代码质量 | 开发工程师 | 技术负责人 |
| 测试验证 | 测试工程师 | 开发工程师 |
| 文档更新 | 技术文档工程师 | 产品经理 |
| 发布执行 | DevOps工程师 | 技术负责人 |
| 发布验证 | 全体团队成员 | QA工程师 |

### 5.3 记录与追踪

#### 检查记录
- 每次发布必须保存检查记录
- 记录检查时间、执行人、结果
- 异常情况必须记录详细说明

#### 改进追踪
- 记录流程中发现的问题
- 跟踪改进措施实施情况
- 定期评审检查清单有效性
