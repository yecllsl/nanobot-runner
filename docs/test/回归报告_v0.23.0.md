#### 3) evolution CLI测试

```bash
uv run pytest tests/unit/cli/test_evolution_cli.py -v --tb=short
```

- **结果**: 26 passed, 5 failed, 1 warning
- **耗时**: 7.51s
- **结论**: 存在5个失败用例

**失败用例清单**:

| 用例名称 | 失败原因 | 严重等级 |
|---------|---------|---------|
| `TestGetAccuracy::test_returns_dict` | Mock返回值类型断言失败，`isinstance(mock, dict)`返回False | 一般 |
| `TestGetAccuracy::test_default_days_30` | `_compute_accuracy_stats` 未被调用（调用次数为0） | 一般 |
| `TestGetFidelity::test_returns_dict_with_data` | Mock返回值类型断言失败 | 一般 |
| `TestGetFidelity::test_returns_empty_when_no_data` | Mock `__getitem__` 返回值不等于0 | 一般 |
| `TestGetFidelity::test_ignores_none_fidelity` | Mock `__getitem__` 返回值不等于0 | 一般 |

**根因分析**:
测试代码中的Mock设置与实际代码实现不匹配。`evolution_handler.py` 中 `get_accuracy()` 和 `get_fidelity()` 方法直接调用 `engine.outcome_collector.get_accuracy_stats()` / `get_fidelity_stats()`，但测试代码Mock的是 `_compute_accuracy_stats` 和 `_store.query_outcomes`，与实际调用链不一致。

**归属判定**: 这是测试代码与生产代码不同步导致的测试缺陷，**非BUG修复引入的回归问题**。

#### 4) evolution集成测试

```bash
uv run pytest tests/integration/test_evolution_integration.py -v --tb=short
```

- **结果**: 1 passed, 1 failed, 14 errors, 1 warning
- **耗时**: 7.51s
- **结论**: 存在严重问题

**错误清单**:

| 错误类型 | 数量 | 错误信息 | 严重等级 |
|---------|------|---------|---------|
| `TypeError` | 14 | `EvolutionEngine.__init__() got an unexpected keyword argument 'data_dir'` | **严重** |
| `AssertionError` | 1 | `assert None is not None` (execution_fidelity为None) | 一般 |

**根因分析**:
- **14个TypeError**: `test_evolution_integration.py` 中的fixture `evolution_engine` 使用 `EvolutionEngine(data_dir=data_dir)` 初始化，但 `EvolutionEngine` 已在架构重构中改为**依赖注入模式**，构造函数参数为 `(decision_logger, outcome_collector)`，不再接受 `data_dir` 参数。
- **1个AssertionError**: `check_plan_execution` 返回的 `OutcomeRecord` 中 `execution_fidelity` 为None，与测试预期不符。

**归属判定**: 这是集成测试代码未随架构重构同步更新导致的测试缺陷，**非BUG修复引入的回归问题**。

#### 5) E2E测试

```bash
uv run pytest tests/e2e/ -v --tb=short
```

- **结果**: 71 passed, 5 failed, 6 warnings
- **耗时**: 29.28s
- **结论**: 存在5个失败用例

**失败用例清单**:

| 用例名称 | 失败原因 | 严重等级 |
|---------|---------|---------|
| `test_complete_training_plan_flow` | async def functions are not natively supported | **严重** |
| `test_plan_adjustment_flow` | async def functions are not natively supported | **严重** |
| `test_plan_cancellation_flow` | async def functions are not natively supported | **严重** |
| `test_plan_deletion_after_cancellation` | async def functions are not natively supported | **严重** |
| `test_progress_hook_execution` | async def functions are not natively supported | **严重** |

**根因分析**:
E2E测试使用了 `@pytest.mark.asyncio` 装饰器，但运行时提示 "async def functions are not natively supported"。经检查，**`pytest-asyncio` 插件未安装**（`uv pip list` 中无 `pytest-asyncio`）。

虽然 `pyproject.toml` 中已配置 `asyncio_mode = "auto"`（BUG-001修复），且 `[project.optional-dependencies]` 的 `test` 组中声明了 `pytest-asyncio>=0.23.0`，但**实际虚拟环境中该依赖未安装**。

**归属判定**: **BUG-001修复不完整**。配置已添加但依赖未实际安装，导致E2E异步测试无法执行。

## 六、Bug清单与状态

### 6.1 回归测试中新发现的Bug

| Bug编号 | 所属模块 | 严重等级 | 标题 | 状态 |
|---------|---------|---------|------|------|
| REG-BUG-001 | 测试基础设施 | **严重** | pytest-asyncio依赖未安装导致E2E异步测试全部失败 | 待修复 |
| REG-BUG-002 | 测试代码 | 一般 | CLI测试Mock与实际代码调用链不匹配 | 待修复 |
| REG-BUG-003 | 测试代码 | **严重** | 集成测试未适配EvolutionEngine依赖注入模式 | 待修复 |

### 6.2 详细说明

#### REG-BUG-001: pytest-asyncio依赖未安装

- **复现步骤**:
  1. 运行 `uv run pytest tests/e2e/test_plan_e2e.py -v`
  2. 所有 `@pytest.mark.asyncio` 装饰的测试用例失败
- **实际结果**: 报错 "async def functions are not natively supported"
- **预期结果**: 异步测试正常执行
- **根因分析**: `pyproject.toml` 中虽声明了 `pytest-asyncio>=0.23.0` 在 `[project.optional-dependencies]` 的 `test` 组，但当前虚拟环境未安装该依赖
- **修复建议**: 执行 `uv pip install pytest-asyncio>=0.23.0` 或 `uv sync --extra test` 安装测试依赖

#### REG-BUG-002: CLI测试Mock不匹配

- **复现步骤**:
  1. 运行 `uv run pytest tests/unit/cli/test_evolution_cli.py::TestGetAccuracy -v`
  2. 5个用例失败
- **实际结果**: Mock断言失败
- **预期结果**: 测试通过
- **根因分析**: 测试代码Mock的是 `_compute_accuracy_stats` 和 `_store.query_outcomes`，但 `EvolutionHandler` 实际调用的是 `get_accuracy_stats()` 和 `get_fidelity_stats()`
- **修复建议**: 更新测试代码中的Mock目标，与实际调用链保持一致

#### REG-BUG-003: 集成测试未适配依赖注入

- **复现步骤**:
  1. 运行 `uv run pytest tests/integration/test_evolution_integration.py -v`
  2. 14个用例报TypeError
- **实际结果**: `EvolutionEngine.__init__() got an unexpected keyword argument 'data_dir'`
- **预期结果**: 测试通过
- **根因分析**: `EvolutionEngine` 已重构为依赖注入模式，但集成测试仍使用旧版构造函数签名 `EvolutionEngine(data_dir=data_dir)`
- **修复建议**: 更新集成测试fixture，先创建 `EvolutionStore` -> `DecisionLogger` / `OutcomeCollector` -> 再注入 `EvolutionEngine`

## 七、上线结论

### 7.1 上线评估

| 评估项 | 标准 | 实际 | 是否达标 |
|--------|------|------|---------|
| P0-P1级用例通过率 | 100% | BUG-001部分通过 | **否** |
| 无致命/严重级bug | 是 | 存在REG-BUG-001/003（严重） | **否** |
| 核心业务流程闭环 | 是 | evolution单元测试+性能测试通过 | 是 |
| 一般级bug修复率 | >=90% | N/A | - |
| 符合需求验收标准 | 是 | - | 待确认 |

### 7.2 结论

**不建议上线**

理由如下：

1. **BUG-001修复不完整**: 虽然 `pyproject.toml` 中已添加 `asyncio_mode = "auto"` 配置，但 `pytest-asyncio` 依赖未实际安装，导致5个E2E异步测试全部失败。这是一个**严重的测试基础设施缺陷**。

2. **集成测试严重失效**: 14个集成测试因 `EvolutionEngine` 构造函数签名变更而报错，集成测试覆盖率几乎为0，无法验证跨模块交互的正确性。

3. **CLI测试存在Mock不匹配问题**: 5个CLI单元测试失败，虽然属于测试代码问题，但表明测试代码与生产代码存在同步滞后。

### 7.3 整改建议

| 优先级 | 整改项 | 负责角色 | 预计工作量 |

| P0 | 修复集成测试 `EvolutionEngine` 初始化方式，适配依赖注入模式 | 开发工程师 | 2小时 |
