## 1. 文档目的

本指南用于指导用户（技术型跑者/跑步数据分析师）或AI Agent对 Nanobot Runner v0.10.0-v0.12.0 版本进行验收测试，验证智能跑步计划功能是否满足业务需求和使用预期。

**测试范围**:
| 版本 | 功能模块 | 核心能力 |
|------|---------|---------|
| v0.10.0 | 数据感知层 | 计划执行记录、统计报告、训练响应分析 |
| v0.11.0 | 智能调整层 | 计划调整、规则引擎校验、Prompt模板、对话管理 |

## 2. 测试准备

### 2.1 环境要求

| 项目 | 要求 | 验证方法 |
|------|------|---------|
| 操作系统 | Windows 10+/macOS 12+ | `uname -a` 或系统设置 |
| Python | 3.11+ | `python --version` |
| uv包管理器 | 已安装 | `uv --version` |
| 磁盘空间 | ≥500MB | `df -h` |
| 网络 | 可选（仅Agent功能需要） | `ping github.com` |

### 2.2 安装步骤

```bash
# 1. 克隆项目
git clone https://github.com/yecllsl/nanobot-runner.git
cd nanobot-runner

# 2. 创建虚拟环境
uv venv

# 3. 激活虚拟环境（Windows PowerShell）
.venv\Scripts\activate

# 4. 同步依赖
uv sync --all-extras

# 5. 验证安装
uv run nanobotrun --help
```

**安装成功标志**: 显示帮助信息，包含 `data`、`analysis`、`agent`、`report`、`system`、`init`、`plan` 等子命令。

**自动化判定规则**:
- ✅ **通过**: 命令退出码为0，输出包含"Usage:"或"Commands:"关键字
- ❌ **失败**: 命令退出码非0，或输出包含"Error"、"ModuleNotFoundError"

### 2.3 测试数据准备

**方式一：使用项目自带样本数据**

```bash
# 样本数据位于 tests/data/fixtures/ 目录
ls tests/data/fixtures/
```

| 文件 | 类型 | 用途 |
|------|------|------|
| `easy_run_*.fit` | 轻松跑 | 基础功能验证 |
| `tempo_run_*.fit` | 节奏跑 | 强度分析验证 |
| `long_run_*.fit` | 长距离跑 | 耐力分析验证 |
| `interval_*.fit` | 间歇跑 | 间歇训练验证 |

**方式二：使用个人真实数据**

```bash
# 准备个人FIT文件目录
mkdir ~/test-fit-files
# 将Garmin/Coros导出的FIT文件复制到该目录
```

### 2.4 初始化配置

```bash
# 交互式初始化（推荐新用户）
uv run nanobotrun system init

# 自动模式（适合有经验用户）
uv run nanobotrun system init --auto
```

**初始化成功标志**:
- 创建 `~/.nanobot-runner/` 目录
- 生成 `config.json` 配置文件
- 显示初始化完成提示

**自动化判定规则**:
- ✅ **通过**: 命令退出码为0，输出包含"初始化完成"或"initialized successfully"
- ❌ **失败**: 命令退出码非0，或输出包含"Error"、"Failed"

## 3. v0.10.0 数据感知层验收测试

### 3.1 计划执行记录功能

**测试目标**: 验证能够记录训练计划的执行反馈

**前置条件**: 
- 已完成系统初始化
- 已创建至少一个训练计划

**测试步骤**:

#### 步骤1: 记录计划执行

```bash
# 记录某天的计划执行情况
uv run nanobotrun plan log plan_20240101 2024-01-01 \
  --completion 0.8 \
  --effort 6 \
  --feedback "感觉良好，心率稳定"
```

**预期结果**:
- 显示记录成功提示
- 包含计划ID、日期、完成率、努力程度等信息

**标准输出示例**:
```
✅ 执行记录已保存
  计划ID: plan_20240101
  日期: 2024-01-01
  完成率: 80%
  努力程度: 6/10
  反馈: 感觉良好，心率稳定
```

**自动化判定规则**:
- ✅ **通过**: 退出码为0，输出包含"已保存"或"saved"，包含计划ID和日期
- ❌ **失败**: 退出码非0，或输出包含"Error"、"Failed"、"计划不存在"

#### 步骤2: 交互式记录（可选）

```bash

```

**预期结果**:

- 提示输入努力程度（1-10之间）
- 提示输入反馈备注

**自动化判定规则**:
- ✅ **通过**: 退出码为0，输出包含交互式提示（"请输入"、"completion"等）
- ❌ **失败**: 退出码非0，或无交互式提示

**预期结果**:
- 边界值均能正常保存

**自动化判定规则**:
- ✅ **通过**: 所有命令退出码为0
- ❌ **失败**: 任一命令退出码非0

### 12.3 常见问题

**Q1: 计划ID如何获取？**  
A: 计划ID由系统生成，格式为 `plan_YYYYMMDD`，可通过 `plan list` 命令查看已有计划。

**Q2: 目标时间为什么用秒？**  
A: 为了统一计算标准，所有时间目标均以秒为单位传入。例如5K=20分钟=1200秒。

**Q3: 规则引擎可以自定义吗？**  
A: 当前版本内置5条核心规则，暂不支持自定义。后续版本将开放规则配置。

**Q4: Agent功能需要联网吗？**  
A: 是的，Agent功能需要连接LLM服务，请确保网络畅通且已配置API密钥。

**Q5: 飞书Gateway如何配置？**  
A: 需要在config.json中配置飞书应用凭证（App ID、App Secret），然后运行 `uv run nanobotrun gateway start`。

**Q6: Agent Chat和Gateway功能是否一致？**  
A: 是的，两者使用相同的Agent工具集，只是交互方式不同。Agent Chat通过本地终端交互，Gateway通过飞书消息交互。

### 12.4 术语表

| 术语 | 说明 |
|------|------|
| VDOT | 跑步能力值，基于Powers公式计算 |
| TSS | 训练压力分数，衡量单次训练负荷 |
| 完成率 | 实际完成训练量与计划训练量的比值 |
| 努力程度 | 主观感受的训练强度，1-10分 |
| 训练响应 | 身体对训练负荷的适应情况 |
| 规则引擎 | 校验计划调整合理性的规则集合 |
| 达成概率 | 基于当前水平预测目标达成的可能性 |
| Agent Chat | 本地终端自然语言交互模式 |
| Gateway | 飞书机器人远程交互模式 |
