## BUG-2201: ConfigManager.get() 未读取环境变量

### 根因

`ConfigManager.get()` 方法只调用 `self.load_config()`，该方法仅从 `config.json` 读取配置。
虽然 `load_config_with_env_override()` 方法已存在且正确映射了 `ENV_KEY_MAPPING`（包含 `feishu_app_id` → `NANOBOT_FEISHU_APP_ID` 等），
但 `get()` 未使用它。

**调用链**：
```
FeishuAuth.__init__()
  → self.config.get("feishu_app_id")
    → ConfigManager.get()
      → self.load_config()           # 只读config.json
      ✗ self.load_config_with_env_override()  # 未调用！
```

### 修复方案

**文件**: `src/core/config/manager.py`

**修改**: `get()` 方法改用 `load_config_with_env_override()`

```python
# 当前代码（第265行附近）:
def get(self, key: str, default: Any = None) -> Any:
    config = self.load_config()
    return config.get(key, default)

# 修改为:
def get(self, key: str, default: Any = None) -> Any:
    config = self.load_config_with_env_override()
    return config.get(key, default)
```

### 影响范围

- 所有通过 `ConfigManager.get()` 读取的配置项都将支持环境变量覆盖
- `ENV_KEY_MAPPING` 中已定义的映射：data_dir, workspace_dir, auto_push_feishu, feishu_app_id, feishu_app_secret, feishu_receive_id, feishu_receive_id_type, timezone, default_year, llm_provider, llm_model, llm_base_url
- **风险**：低。`load_config_with_env_override()` 是已有方法，逻辑已验证

### 验证方法

```powershell
# 1. 设置环境变量
$env:NANOBOT_FEISHU_APP_ID="test_app_id"
$env:NANOBOT_FEISHU_APP_SECRET="test_secret"

# 2. 执行cron status
uv run nanobotrun cron status

# 预期：不再显示"未配置飞书应用凭证"
```

### 单元测试

```python
# tests/unit/core/config/test_manager.py
def test_get_reads_env_override(monkeypatch):
    """验证get()方法读取环境变量覆盖"""
    monkeypatch.setenv("NANOBOT_FEISHU_APP_ID", "env_app_id")
    config = ConfigManager(allow_default=True)
    assert config.get("feishu_app_id") == "env_app_id"
```

## BUG-2202: fatigue/recovery 传入空DataFrame导致"暂无数据"

### 根因

`FatigueAssessor.assess_fatigue()` 和 `RecoveryMonitor.get_recovery_status()` 都调用：
```python
load_data = self.training_load_analyzer.calculate_training_load_from_dataframe(
    pl.DataFrame()  # ← 空DataFrame！
)
```

`calculate_training_load_from_dataframe()` 收到空DataFrame后直接返回 `runs_count=0`，
导致fatigue/recovery判断为"暂无训练数据"。

而 `analysis load` 命令通过 `AnalyticsEngine.get_training_load()` 正确从 `self.storage.read_parquet()` 读取数据。

### 修复方案

**文件1**: `src/core/body_signal/fatigue_assessor.py`

**修改**: `assess_fatigue()` 方法从 `session_repo` 读取实际数据

```python
# 当前代码（第56行附近）:
load_data = self.training_load_analyzer.calculate_training_load_from_dataframe(
    pl.DataFrame()
)

# 修改为:
try:
    lf = self.session_repo.storage.read_parquet()
    session_df = lf.collect()
except Exception:
    session_df = pl.DataFrame()

load_data = self.training_load_analyzer.calculate_training_load_from_dataframe(
    session_df
)
```

**文件2**: `src/core/body_signal/recovery_monitor.py`

**修改**: `get_recovery_status()` 方法从 `session_repo` 读取实际数据

```python
# 当前代码（第58行附近）:
load_data = self.training_load_analyzer.calculate_training_load_from_dataframe(
    pl.DataFrame()
)

# 修改为:
try:
    lf = self.session_repo.storage.read_parquet()
    session_df = lf.collect()
except Exception:
    session_df = pl.DataFrame()

load_data = self.training_load_analyzer.calculate_training_load_from_dataframe(
    session_df
)
```

### 影响范围

- `FatigueAssessor` 和 `RecoveryMonitor` 都有 `session_repo` 依赖，可直接使用
- **风险**：中。需确保 `session_repo.storage.read_parquet()` 在无数据时不抛异常

### 验证方法

```powershell
# 1. 确认已导入数据
uv run nanobotrun data stats

# 2. 执行fatigue
uv run nanobotrun analysis fatigue
# 预期：显示疲劳度评分（非"暂无训练数据"）

# 3. 执行recovery
uv run nanobotrun analysis recovery
# 预期：显示恢复状态（非"暂无恢复状态数据"）
```

### 单元测试


## BUG-2203: StateVectorBuilder.build_load() 传入空列表导致CTL/ATL=0

### 根因

`StateVectorBuilder.build_load()` 调用：
```python
result = self._training_load_analyzer.calculate_atl_ctl([])  # ← 空列表！
```

`calculate_atl_ctl([])` 收到空列表直接返回 `{"atl": 0.0, "ctl": 0.0}`。

而 `analysis load` 通过 `AnalyticsEngine.get_training_load()` 使用 `calculate_training_load_from_dataframe()` 从Parquet读取数据。

### 修复方案

**文件**: `src/core/twin/state_vector_builder.py`

**修改**: `build_load()` 方法从 `session_repo` 读取数据并计算TSS

```python
# 当前代码（第128行附近）:
def build_load(self) -> LoadDimension:
    if self._training_load_analyzer is None:
        return LoadDimension(ctl=0.0, atl=0.0, tsb=0.0, acwr=0.0)
    try:
        result = self._training_load_analyzer.calculate_atl_ctl([])
        ...

# 修改为:
def build_load(self) -> LoadDimension:
    if self._training_load_analyzer is None:
        return LoadDimension(ctl=0.0, atl=0.0, tsb=0.0, acwr=0.0)
    try:
        if self._session_repo is not None:
            try:
                lf = self._session_repo.storage.read_parquet()
                session_df = lf.collect()
            except Exception:
                session_df = pl.DataFrame()
            result = self._training_load_analyzer.calculate_training_load_from_dataframe(
                session_df
            )
        else:
            result = self._training_load_analyzer.calculate_atl_ctl([])
        ...
```

### 影响范围

- `StateVectorBuilder` 已有 `_session_repo` 依赖，可直接使用
- 需确保 `calculate_training_load_from_dataframe()` 返回格式与 `calculate_atl_ctl()` 一致
- **风险**：中。`calculate_training_load_from_dataframe` 返回更多字段（含fitness_status等），但 `atl/ctl` 字段名一致

### 验证方法

```powershell
# 1. 确认analysis load有数据
uv run nanobotrun analysis load
# 记录CTL/ATL值

# 2. 执行twin snapshot
uv run nanobotrun twin snapshot
# 预期：CTL/ATL值与analysis load一致（非0）
```

### 单元测试

```python
# tests/unit/core/twin/test_state_vector_builder.py
def test_build_load_with_session_data():
    """验证build_load从session_repo读取数据"""

```
