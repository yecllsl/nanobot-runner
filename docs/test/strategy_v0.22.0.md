## 1. 测试范围

### 1.1 版本定位

v0.22.0是**质量收口版本（Hardening Release）**，无新功能开发，核心目标：
1. 系统化UAT验证v0.5-v0.21全版本功能稳定性
2. 缺陷收敛：致命/严重清零，一般缺陷修复率≥80%
3. 质量兜底：边界测试+性能基线+数据一致性
4. 需求洞察：挖掘用户痛点，形成新需求

### 1.2 测试覆盖模块

| 版本 | 模块 | 测试重点 | 优先级 |
|------|------|----------|--------|
| v0.5+ | 数据导入/查询/分析 | FIT解析、SHA256去重、VDOT/TSS计算 | P0 |
| v0.5+ | 训练计划 | 智能建议、目标评估、长期规划 | P0 |
| v0.5+ | 报告生成 | 周报、月报、报告导出 | P0 |
| v0.5+ | 系统管理 | 配置验证、数据迁移、备份 | P0 |
| v0.17 | MCP工具/Gateway/Cron | 工具管理、飞书通道、训练提醒 | P1 |
| v0.17 | AI透明化/偏好/技能 | 决策追踪、偏好管理、技能管理 | P1 |
| v0.18 | 数据可视化/导出 | 终端图表、多格式导出 | P1 |
| v0.19 | 身体信号分析 | HRV分析、疲劳度评估、恢复状态 | P0 |
| v0.20 | ML增强预测 | VDOT趋势、比赛成绩、伤病风险 | P0 |
| v0.21 | 数字孪生引擎 | 状态向量、What-If推演、计划对比 | P0 |

### 1.3 不纳入测试范围

- 第三方依赖内部实现（fitparse、polars、pyarrow等）
- nanobot-ai框架核心功能
- LLM模型输出内容的不确定性验证
- 操作系统级别的文件系统行为

## 2. 测试类型

### 2.1 用户验收测试（UAT）

**职责**: 测试工程师主责，AI Agent辅助执行

**覆盖范围**: v0.5-v0.21全版本功能模块，100个UAT用例

**测试模式**: AI Agent + 用户交互逐用例验证

### 2.2 回归测试

**触发条件**: 缺陷修复后

**覆盖范围**: 
- 修复缺陷对应的测试用例
- 受影响模块的核心功能用例
- P0级用例全量回归

### 2.3 边界测试

**覆盖范围**:
- 异常输入处理（空文件、损坏文件、非法参数）
- 边界值测试（数据量极限、时间范围极限）
- 数据缺失降级策略验证

### 2.4 性能测试

**测试重点**:
- 批量导入性能（100个FIT文件）
- 查询响应时间（data stats、twin snapshot）
- ML预测响应时间（<5秒）
- 推演计算时间（<10秒）
- 状态向量聚合时间（<3秒）

### 2.5 数据一致性测试

**测试重点**:
- VDOT/TSS/CTL/ATL/TSB核心指标计算准确性
- twin引擎与analysis模块数据一致性
- 历史数据兼容性（v0.19及之前）
- 跨模块数据传递准确性

## 3. 门禁规则

### 3.1 测试准入规则

代码进入测试环节前，必须满足：

| 条件 | 验证方式 | 责任人 | 状态 |
|------|---------|--------|------|
| 需求规格已评审通过 | 文档存在且版本匹配 | 架构师 | ✅ |
| 架构设计已评审通过 | 文档存在且版本匹配 | 架构师 | ✅ |
| 开发完成并通过自测 | 开发者自测报告 | 开发工程师 | ❌ 待验证 |
| 单元测试覆盖率达标 | `pytest --cov` 报告 | 开发工程师 | ❌ 待验证 |
| 代码质量检查通过 | `ruff check` 零警告 | 开发工程师 | ❌ 待验证 |
| 类型检查通过 | `mypy` 无新增错误 | 开发工程师 | ❌ 待验证 |
| 无未解决的P0/P1 Bug | Bug清单状态 | 开发工程师 | ❌ 当前7个bug待修复 |

**准入验证命令**:
```bash
uv run ruff check src/ tests/
uv run ruff format --check src/ tests/
uv run mypy src/ --ignore-missing-imports
uv run pytest tests/unit/ --cov=src --cov-report=term-missing
```

### 3.2 测试准出规则

测试完成并允许发布，必须满足：

| 条件 | 标准 | 验证方式 | 当前状态 |
|------|------|---------|----------|
| P0级用例通过率 | 100% | UAT测试报告 | ✅ 100% |
| P1级用例通过率 | ≥90% | UAT测试报告 | ❌ 85%（3个P1 bug） |
| 致命Bug | 0个 | Bug清单 | ✅ 0个 |
| 严重Bug | 0个 | Bug清单 | ❌ 3个（P1-001/002/003） |
| 一般Bug修复率 | ≥80% | Bug清单 | ❌ 0%（4个未修复） |
| 核心业务流程 | 全量闭环 | E2E测试报告 | ✅ |
| 性能退化 | 无显著退化 | 性能测试报告 | ⚠️ 部分偏慢 |
| 安全合规 | 无敏感信息泄露 | 安全扫描报告 | ✅ |

### 3.3 上线门禁规则

**绝对禁止发布的情况**:
- 存在任何致命或严重级Bug（当前3个P1 bug）
- P0级用例通过率 < 100%
- 核心业务流程未闭环
- 测试报告未输出或未评审通过
- 安全扫描发现敏感信息泄露

**允许发布的条件**（全部满足）:
- ✅ P0-P1级用例100%通过
- ✅ 无致命/严重级Bug
- ✅ 一般级Bug修复率≥80%
- ✅ 核心业务流程全量闭环
- ✅ 符合需求验收标准
- ✅ 测试报告已输出并评审通过

## 5. 缺陷管理策略

### 5.1 缺陷分级

| 级别 | 定义 | 修复时限 | 验收标准 |
|------|------|----------|----------|
| **致命** | 系统崩溃、数据丢失、核心功能不可用 | 24小时内 | 100%修复 |
| **严重** | 主要功能异常、性能严重下降 | 3天内 | 100%修复 |
| **一般** | 次要功能异常、用户体验问题 | 1周内 | 修复率≥80% |
| **轻微** | 界面瑕疵、文案问题 | 可选修复 | 记录待后续优化 |

## 8. 风险评估与应对
