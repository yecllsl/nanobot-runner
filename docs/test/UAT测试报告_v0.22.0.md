## 4. 严重Bug详情

### P1-001: Cron飞书凭证读取失败
- **现象**: `cron status`显示"未配置飞书应用凭证"
- **根因**: ConfigManager从config.json读取feishu_app_id/feishu_app_secret，但初始化向导将凭证写入.env.local（NANOBOT_FEISHU_APP_ID格式），两处配置源不一致
- **修复建议**: ConfigManager增加从.env.local环境变量读取飞书凭证的fallback逻辑

### P1-002: fatigue/recovery命令未读取已导入数据
- **现象**: `analysis fatigue`和`analysis recovery`显示"暂无训练数据"，但实际已导入637条记录
- **根因**: fatigue/recovery模块的数据查询条件可能与数据存储格式不匹配，或使用了不同的数据源路径
- **修复建议**: 排查fatigue/recovery的数据读取逻辑，确保与storage模块使用相同的查询路径

### P1-003: twin snapshot与analysis load数据不一致
- **现象**: `twin snapshot`显示CTL=0/ATL=0，但`analysis load`显示CTL=64.6/ATL=64.6
- **根因**: twin引擎的状态向量构建器可能使用了不同的数据查询逻辑或时间窗口
- **修复建议**: 统一twin引擎与analysis模块的负荷计算逻辑，确保数据源一致

### 整体评估

| 评估项 | 结果 | 说明 |
|--------|------|------|
| P0级用例通过率 | 100% | 核心业务流程（导入/查询/分析/导出）全部通过 |

### 上线风险评估

| 风险项 | 等级 | 说明 |
|--------|------|------|
| 飞书通知不可用 | 中 | P1-001导致cron提醒无法推送飞书消息 |
| 身体信号分析缺失 | 中 | P1-002导致fatigue/recovery无数据 |
| 数字孪生数据不准 | 高 | P1-003导致twin快照负荷数据为0，影响推演准确性 |
| 性能偏慢 | 低 | data stats/twin snapshot响应时间偏长 |

### 测试结论：❌ 不通过

**原因**: 存在3个严重(P1)级bug，不满足上线门禁标准（无致命/严重bug）。

**必须修复的问题**:
1. P1-001: Cron飞书凭证读取失败
2. P1-002: fatigue/recovery未读取已导入数据
3. P1-003: twin snapshot CTL/ATL与analysis load不一致

**整改建议**:
1. 修复P1-001后执行回归测试验证cron status正常
2. 修复P1-002后验证fatigue/recovery能正确显示数据
3. 修复P1-003后验证twin snapshot与analysis load数据一致
4. 同步修复P2-003（CLI命令名文档一致性）
5. 所有P1 bug修复后重新执行UAT回归测试

## 7. 后续优化建议

1. **配置统一**: 统一ConfigManager对config.json和.env.local的读取逻辑，避免配置源不一致
2. **数据源统一**: 统一twin/analysis/body_signal模块的数据查询路径，确保一致性
3. **性能优化**: 优化data stats和twin snapshot的查询性能，目标<5s
4. **Polars最佳实践**: 将`lf.columns`替换为`lf.collect_schema().names()`，消除PerformanceWarning
5. **文档同步**: 更新UAT指南中CLI命令名，确保与实现一致
6. **偏好推断优化**: 基于实际训练时间分布推断训练时段，而非默认"早晨"
