## 2. 测试类型与策略

### 2.1 单元测试（Unit Testing）

**职责**: 开发工程师主责，测试工程师负责规范指导和结果校验

**覆盖范围**:
- 核心业务逻辑函数
- 数据计算算法（VDOT、TSS、心率漂移等）
- 技能管理器（加载、启用/禁用、冲突检测）
- 工具管理器（接口兼容、错误处理、超时处理）
- 记忆管理器（存储、检索、版本控制）
- 配置解析与验证
- 工具类方法

**覆盖率要求**:
| 模块 | 最低覆盖率 | 当前状态 |
|------|-----------|---------|
| `src/core/` | ≥80% | ✅ 84% |
| `src/agents/` | ≥70% | ✅ 87%~100% |
| `src/cli/` | ≥60% | 📋 进行中 |
| `src/core/skills/` | ≥85% | 📋 新增要求 |
| `src/core/tools/` | ≥85% | 📋 新增要求 |

**Mock策略**:
- ✅ 必须Mock外部API调用（天气、地图、飞书通知、LLM）
- ✅ 必须Mock文件系统IO（使用临时目录）
- ❌ 禁止Mock内部业务逻辑

### 2.2 集成测试（Integration Testing）

**职责划分**:
| 测试类型 | 目录 | 负责人 |
|---------|------|--------|
| 模块内集成测试 | `tests/integration/module/` | 开发工程师 |
| 场景级集成测试 | `tests/integration/scene/` | 测试工程师 |

**场景级集成测试覆盖**:
- 数据导入→存储→查询完整链路
- 配置初始化→验证→应用完整链路
- 技能发现→启用→工具调用完整链路
- 多技能协同→智能推荐→结果返回完整链路
- 记忆存储→跨会话恢复→人格进化完整链路
- 决策透明化→状态看板→洞察报告完整链路

### 2.3 端到端测试（E2E Testing）

**职责**: 测试工程师主责

**覆盖范围**:
- CLI命令完整执行流程
- Agent对话交互流程
- 报告生成与推送流程
- Gateway服务消息流
- 智能技能生态全流程（发现→启用→使用→统计）
- AI教练进化全流程（记忆→反馈→进化→建议）
- 透明洞察全流程（决策展示→状态看板→洞察报告）

**测试环境**:
- 使用临时测试目录，不影响真实用户数据
- Mock所有外部依赖（天气API、地图API、飞书API、LLM API）
- 使用`tests/data/fixtures/`下的样本FIT文件

## 3. 门禁规则

### 3.1 测试准入规则

代码进入测试环节前，必须满足以下条件：

| 条件 | 验证方式 | 责任人 |
|------|---------|--------|
| 需求规格说明书已评审通过 | 文档存在且版本匹配 | 架构师 |
| 架构设计说明书已评审通过 | 文档存在且版本匹配 | 架构师 |
| 开发完成并通过自测 | 开发者自测报告 | 开发工程师 |
| 单元测试覆盖率达标 | `pytest --cov` 报告 | 开发工程师 |
| 代码质量检查通过 | `ruff check` 零警告 | 开发工程师 |
| 类型检查通过 | `mypy` 无新增错误 | 开发工程师 |
| 无未解决的P0/P1 Bug | Bug清单状态 | 开发工程师 |

**准入验证命令**:
```bash
# 代码质量检查
uv run ruff check src/ tests/
uv run ruff format --check src/ tests/
uv run mypy src/ --ignore-missing-imports

# 单元测试覆盖率
uv run pytest tests/unit/ --cov=src --cov-report=term-missing

# 覆盖率验证
# core ≥ 80% | agents ≥ 70% | cli ≥ 60% | skills ≥ 85% | tools ≥ 85%
```

- ✅ 核心业务流程全量闭环

## 5. Bug管理规范

### 5.1 严重等级定义

| 等级 | 定义 | 响应时间 | 修复优先级 |
|------|------|---------|-----------|
| 致命 | 阻断核心流程，无法继续测试 | 立即 | P0 |
| 严重 | 核心功能异常，影响主流程 | 4小时内 | P0 |
| 一般 | 非核心功能异常，不影响主流程 | 24小时内 | P1 |
| 优化 | 体验/规范类问题 | 迭代内 | P2 |

### 5.2 Bug提交规范

每个Bug必须包含以下信息：

```markdown
## Bug ID: BUG-XXX-001

**所属模块**: 技能管理  
**严重等级**: 严重  
**Bug标题**: 启用技能后AI无法调用对应工具  
**复现步骤**:
1. 启用"天气助手"技能
2. 询问"明天天气如何"
3. 观察AI响应

**实际结果**: AI提示技能未启用  
**预期结果**: AI应调用天气工具返回天气信息  
**根因分析**: 技能启用状态未正确同步到AgentLoop  
**修复建议**: 在技能启用后刷新Agent上下文  
**出现版本**: v0.13.0  
**优先级**: P0  
**创建时间**: 2026-04-27  
**状态**: 待修复
```

### 5.3 Bug跟踪流程

```
发现Bug → 提交Bug清单 → 开发工程师修复 → 回归测试 → 状态更新
    ↓                                              ↓
  记录日志                                      已闭环/驳回
```

**Bug状态**:
- 待修复：已提交，等待修复
- 修复中：开发工程师正在修复
- 待回归：修复完成，等待回归测试
- 已闭环：回归测试通过
- 已驳回：修复无效，退回重新修复

### 6.2 测试数据管理

**测试数据来源**:
- 使用 `tests/data/fixtures/` 下的样本FIT文件
- 禁止使用真实用户数据
- 测试数据必须脱敏
- 技能测试需要构造技能配置文件
- 工具测试需要Mock外部API响应

**临时目录管理**:
```python
import tempfile
import shutil
from pathlib import Path

@pytest.fixture
def temp_data_dir():
    """创建临时测试目录"""
    temp_dir = Path(tempfile.mkdtemp())
    yield temp_dir
    shutil.rmtree(temp_dir)  # 清理

@pytest.fixture
def temp_workspace(temp_data_dir):
    """创建临时工作区"""
    workspace = temp_data_dir / "workspace"
    workspace.mkdir()
    return workspace
```

## 11. 测试风险控制

### 11.1 技术风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| nanobot-ai底座API变更 | 高 | 锁定版本，建立API兼容性测试 |
| 第三方工具API不稳定 | 中 | Mock外部依赖，增加重试机制测试 |
| 测试数据不足 | 中 | 提前构造测试数据集 |

### 11.2 进度风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 开发延期 | 高 | 测试左移，提前准备测试用例 |
| Bug修复周期长 | 中 | 优先修复P0/P1 Bug |
| 测试环境不稳定 | 中 | 使用临时目录，隔离测试环境 |

### 11.3 质量风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 覆盖率不达标 | 高 | 开发阶段同步检查覆盖率 |
| 核心场景遗漏 | 高 | 基于用户旅程地图设计用例 |
| 性能退化 | 中 | 建立性能基线，每次发布前对比 |
