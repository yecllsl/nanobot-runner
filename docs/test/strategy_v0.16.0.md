## 1. 版本概述与测试目标

### 1.1 v0.16.0版本定位

**纯架构重构版本**，不新增任何业务功能，核心目标是将`src/core`目录下散落的30+个文件按职责划分到6个独立子模块中，提升代码可维护性和可扩展性。

| 重构模块 | 源文件数 | 受影响文件数 | 风险等级 |
|---------|---------|-------------|---------|
| `base/` | 7 | 93 | 🔴 高风险 |
| `calculators/` | 7 | 27 | 🟢 低风险 |
| `config/` | 6 | 42 | 🟠 中高风险 |
| `storage/` | 5 | 33 | 🟡 中风险 |
| `report/` | 3 | 14 | 🟢 低风险 |
| `models/` | 1(拆分) | 5 | 🔴 高风险 |

### 1.2 测试目标

| 目标 | 说明 | 优先级 |
|------|------|--------|
| 验证重构后功能无退化 | 所有原有功能正常工作，无功能损失 | P0 |
| 验证模块导入路径正确 | 所有import语句正确指向新位置 | P0 |
| 验证向后兼容性 | 公共API保持不变，外部调用无影响 | P0 |
| 验证数据一致性 | 数据处理结果与重构前完全一致 | P0 |
| 验证性能无退化 | 查询性能、导入速度无显著下降 | P1 |
| 验证类型检查通过 | mypy类型注解完整无错误 | P1 |

### 2.2 测试策略

#### 策略1: 全量回归测试

**核心策略**: 执行全部已有测试用例，确保重构引入零回归。

```
测试执行流程:
1. 执行单元测试 (2749个用例)
   ↓ 通过率必须100%
2. 执行集成测试 (272个用例)
   ↓ 通过率必须100%
3. 执行CLI冒烟测试 (核心命令)
   ↓ 全部通过
4. 执行性能基准测试
   ↓ 无显著退化(<5%)
```

#### 策略2: 模块导入专项验证

针对6个重构模块，逐一验证导入路径：

| 模块 | 验证点 | 验证方式 |
|------|--------|---------|
| `base/` | exceptions, logger, decorators, result, schema, context, profile | import测试 |
| `calculators/` | vdot, race_prediction, heart_rate, training_load, training_history, injury_risk, statistics | import测试 |
| `config/` | manager, schema, llm_config, env_manager, backup_manager, sync | import测试 |
| `storage/` | parquet_manager, session_repository, indexer, parser, importer | import测试 |
| `report/` | generator, service, anomaly_filter | import测试 |
| `models/` | user_profile, training_plan, analytics | import测试 |

#### 策略3: 向后兼容性验证

验证`__init__.py`中的公共API重导出：

```python
# 验证示例：旧导入方式仍可用
from src.core import exceptions  # 应重定向到 src.core.base.exceptions
from src.core import storage     # 应重定向到 src.core.storage.parquet_manager
```

#### 策略4: 数据一致性验证

对比重构前后的关键计算结果：

| 计算项 | 验证方式 | 容差 |
|--------|---------|------|
| VDOT计算 | 相同输入，对比输出 | 0误差 |
| TSS计算 | 相同输入，对比输出 | 0误差 |
| 心率漂移 | 相同输入，对比输出 | 0误差 |
| 训练负荷 | 相同输入，对比输出 | 0误差 |

## 5. 测试执行计划

### 5.1 测试阶段划分

| 阶段 | 测试内容 | 预计耗时 | 执行方式 |
|------|---------|---------|---------|
| 阶段1: 自动化回归 | 单元测试+集成测试 | 15分钟 | 自动化 |
| 阶段2: 导入验证 | 模块导入+API验证 | 5分钟 | 自动化 |
| 阶段3: CLI冒烟 | 核心CLI命令 | 10分钟 | 手工+E2E |
| 阶段4: 数据一致性 | 计算结果对比 | 10分钟 | 自动化 |
| 阶段5: 性能基准 | 性能对比测试 | 10分钟 | 自动化 |

### 5.2 测试执行命令

```bash
# 阶段1: 自动化回归测试
uv run pytest tests/unit/ -v --tb=short
uv run pytest tests/integration/ -v --tb=short

# 阶段2: 覆盖率验证
uv run pytest tests/ --cov=src --cov-report=term-missing

# 阶段3: 代码质量检查
uv run ruff check src/ tests/
uv run ruff format --check src/ tests/
uv run mypy src/ --ignore-missing-imports
uv run bandit -r src/

# 阶段4: CLI冒烟测试（手工）
nanobotrun data import tests/data/fixtures/
nanobotrun data list
nanobotrun analyze vdot
nanobotrun analyze tss
nanobotrun config show
nanobotrun report generate

# 阶段5: 性能基准测试
uv run pytest tests/performance/ -v
```

## 6. 风险评估与应对

### 6.1 风险识别

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|---------|
| 导入路径遗漏 | 中 | 高 | 全量import扫描+自动化验证 |
| 公共API破坏 | 低 | 高 | `__init__.py`重导出验证 |
| 性能退化 | 低 | 中 | 性能基准对比测试 |
| 数据计算偏差 | 极低 | 高 | 数据一致性专项验证 |
| 类型注解错误 | 低 | 中 | mypy严格检查 |

### 6.2 回滚策略

若测试发现严重问题：
1. 立即停止测试执行
2. 记录失败用例和Bug
3. 评估修复复杂度
4. 若修复时间>2小时，回滚到重构前版本
5. 修复后重新执行全量测试
