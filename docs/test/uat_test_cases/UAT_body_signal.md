# UAT - 身体信号分析模块 (v0.19)

> **模块版本**: v0.19.0
> **用例范围**: UAT-071 ~ UAT-080
> **优先级分布**: P0: 5个, P1: 5个

---

## UAT-071: HRV分析 - 7天窗口

- **优先级**: P0
- **前置条件**: 已导入≥7天的跑步数据（含心率信息）
- **操作步骤**:
  1. 执行 `uv run nanobotrun analysis hrv --days 7`
- **预期结果**: 显示7天HRV分析结果，包含RMSSD、SDNN、HRV趋势
- **验证要点**: 数值在合理范围（RMSSD 20-100ms），趋势图正常显示

## UAT-074: 心率恢复分析

- **优先级**: P0
- **前置条件**: 已导入含心率数据的跑步记录
- **操作步骤**:
  1. 执行 `uv run nanobotrun analysis hr-recovery`
- **预期结果**: 显示心率恢复分析结果，含恢复速率和恢复质量评估
- **验证要点**: 恢复速率数值合理（1分钟恢复心率≥20bpm为良好）

## UAT-075: 疲劳度评估

- **优先级**: P0

  1. 执行 `uv run nanobotrun analysis fatigue`

## UAT-079: 身体信号 - 数据不足场景

- **优先级**: P1
- **前置条件**: 无跑步数据或数据极少
- **操作步骤**:
  1. 清空测试数据目录
  2. 执行 `uv run nanobotrun status today`
- **预期结果**: 显示数据不足提示，不崩溃
- **验证要点**: 优雅降级，提示用户导入数据，显示data_quality为empty

## UAT-080: 身体信号 - 无心率数据场景

- **优先级**: P1
- **前置条件**: 已导入跑步数据但无心率信息
- **操作步骤**:
  1. 导入不含心率数据的FIT文件（如有）
  2. 执行 `uv run nanobotrun analysis hrv --days 7`
- **预期结果**: 显示数据不足提示或降级提示
- **验证要点**: 不崩溃，给出明确提示说明需要心率数据
