> **文档版本**: v2.0

> **更新日期**: 2026-04-20  
> **适用版本**: v0.9.5+  
> **维护者**: 测试工程师智能体  
> **更新说明**: 基于v0.9.4 UAT反馈问题，新增第三方库兼容性测试、真实数据验证、环境配置加载测试

## 1. 测试目标与范围

### 1.1 测试目标

| 目标 | 说明 | 优先级 |
|------|------|--------|
| 保障核心业务功能正确性 | 数据导入、存储、分析、Agent交互 | P0 |
| 验证跨模块集成链路 | 依赖注入、配置管理、数据迁移 | P0 |
| 确保端到端用户体验 | CLI命令、Agent对话、报告生成 | P0 |
| 保障性能与稳定性 | LazyFrame查询、大数据量处理 | P1 |
| 验证安全与隐私合规 | 本地存储、零外联、敏感信息保护 | P0 |

### 1.2 测试范围

#### ✅ 纳入测试范围

| 模块 | 测试类型 | 负责人 |
|------|---------|--------|
| `src/core/` 核心模块 | 单元测试 + 集成测试 | 开发工程师 |
| `src/cli/` CLI模块 | 单元测试 + E2E测试 | 测试工程师 |
| `src/agents/` Agent工具 | 单元测试 + 集成测试 | 开发工程师 |
| `src/notify/` 通知模块 | 单元测试（Mock外部API） | 开发工程师 |
| 跨模块业务流程 | 场景级集成测试 | 测试工程师 |
| 端到端用户旅程 | E2E全链路测试 | 测试工程师 |

#### ❌ 不纳入测试范围

- 第三方依赖内部实现（fitparse、polars、pyarrow等）
- nanobot-ai框架核心功能
- 操作系统级别的文件系统行为

## 2. 测试类型与策略

### 2.1 单元测试（Unit Testing）

**职责**: 开发工程师主责，测试工程师负责规范指导和结果校验

**覆盖范围**:
- 核心业务逻辑函数
- 数据计算算法（VDOT、TSS、心率漂移等）
- 配置解析与验证
- 工具类方法

**覆盖率要求**:
| 模块 | 最低覆盖率 |
|------|-----------|
| `src/core/` | ≥80% |
| `src/agents/` | ≥70% |
| `src/cli/` | ≥60% |

**Mock策略**:
- ✅ 必须Mock外部API调用（飞书通知、LLM）
- ✅ 必须Mock文件系统IO（使用临时目录）
- ❌ 禁止Mock内部业务逻辑

### 2.2 集成测试（Integration Testing）

**职责划分**:
| 测试类型 | 目录 | 负责人 |
|---------|------|--------|
| 模块内集成测试 | `tests/integration/module/` | 开发工程师 |
| 场景级集成测试 | `tests/integration/scene/` | 测试工程师 |

**场景级集成测试覆盖**:
- 数据导入→存储→查询完整链路
- 配置初始化→验证→应用完整链路
- 数据迁移→校验→备份完整链路
- Agent工具调用→数据分析→结果返回完整链路

### 2.3 端到端测试（E2E Testing）

**职责**: 测试工程师主责

**覆盖范围**:
- CLI命令完整执行流程
- Agent对话交互流程
- 报告生成与推送流程
- Gateway服务消息流

**测试环境**:
- 使用临时测试目录，不影响真实用户数据
- Mock所有外部依赖（飞书API、LLM API）
- 使用`tests/data/fixtures/`下的样本FIT文件

## 3. 门禁规则

### 3.1 测试准入规则

代码进入测试环节前，必须满足以下条件：

| 条件 | 验证方式 | 责任人 |
|------|---------|--------|
| 需求规格说明书已评审通过 | 文档存在且版本匹配 | 架构师 |
| 架构设计说明书已评审通过 | 文档存在且版本匹配 | 架构师 |
| 开发完成并通过自测 | 开发者自测报告 | 开发工程师 |
| 单元测试覆盖率达标 | `pytest --cov` 报告 | 开发工程师 |
| 代码质量检查通过 | `ruff check` 零警告 | 开发工程师 |
| 类型检查通过 | `mypy` 无新增错误 | 开发工程师 |
| 无未解决的P0/P1 Bug | Bug清单状态 | 开发工程师 |

**准入验证命令**:
```bash
# 代码质量检查
uv run ruff check src/ tests/
uv run ruff format --check src/ tests/
uv run mypy src/ --ignore-missing-imports

# 单元测试覆盖率
uv run pytest tests/unit/ --cov=src --cov-report=term-missing

# 覆盖率验证
# core ≥ 80% | agents ≥ 70% | cli ≥ 60%
```

### 3.2 测试准出规则

测试完成并允许发布，必须满足以下条件：

| 条件 | 标准 | 验证方式 |
|------|------|---------|
| P0级用例通过率 | 100% | 测试报告 |
| P1级用例通过率 | ≥95% | 测试报告 |
| 致命Bug | 0个 | Bug清单 |
| 严重Bug | 0个 | Bug清单 |
| 一般Bug修复率 | ≥90% | Bug清单 |
| 核心业务流程 | 全量闭环 | E2E测试报告 |
| 性能退化 | 无显著退化 | 性能测试报告 |
| 安全合规 | 无敏感信息泄露 | 安全扫描报告 |

**准出验证命令**:
```bash
# 全量测试执行
uv run pytest tests/ -v --tb=short

# 集成测试执行
uv run pytest tests/integration/ -v

# E2E测试执行
uv run pytest tests/e2e/ -v

# 性能测试执行
uv run pytest tests/performance/ -v
```

### 3.3 上线门禁规则

**绝对禁止发布的情况**:
- ❌ 存在任何致命或严重级Bug
- ❌ P0级用例通过率 < 100%
- ❌ 核心业务流程未闭环
- ❌ 测试报告未输出或未评审通过
- ❌ 安全扫描发现敏感信息泄露

**允许发布的条件**（全部满足）:
- ✅ P0-P1级用例100%通过
- ✅ 无致命/严重级Bug
- ✅ 一般级Bug修复率≥90%
- ✅ 核心业务流程全量闭环
- ✅ 符合需求验收标准
- ✅ 测试报告已输出并评审通过

## 4. 测试用例设计规范

### 4.1 用例结构

每个测试用例必须包含以下信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例ID | 唯一标识 | `TC-DATA-IMPORT-001` |
| 所属模块 | 功能模块 | 数据导入 |
| 用例名称 | 简洁描述 | 正常FIT文件导入 |
| 前置条件 | 执行前状态 | 测试目录已创建 |
| 操作步骤 | 详细步骤 | 1. 执行导入命令 |
| 预期结果 | 明确结果 | 导入成功，数据正确 |
| 优先级 | P0/P1/P2 | P0 |
| 用例类型 | 功能/边界/异常 | 功能测试 |
| 适用阶段 | 单元/集成/E2E | 集成测试 |

### 4.2 优先级定义

| 优先级 | 定义 | 覆盖率要求 |
|--------|------|-----------|
| P0 | 核心业务流程，阻断性 | 100% |
| P1 | 重要功能，影响主流程 | ≥95% |
| P2 | 边缘场景，可选覆盖 | ≥80% |

### 4.3 用例分类

| 测试类型 | 覆盖内容 | 示例 |
|---------|---------|------|
| 功能测试 | 正常业务流程 | 正常导入FIT文件 |
| 边界测试 | 极限值、空值 | 导入空FIT文件 |
| 异常测试 | 错误处理、容错 | 导入损坏文件 |
| 权限测试 | 访问控制 | 无权限访问配置 |
| 兼容测试 | 多版本、多平台 | Windows/Mac兼容 |
| 性能测试 | 响应时间、吞吐量 | 批量导入100文件 |
| 安全测试 | 敏感信息、数据保护 | 配置加密存储 |

## 5. Bug管理规范

### 5.1 严重等级定义

| 等级 | 定义 | 响应时间 | 修复优先级 |
|------|------|---------|-----------|
| 致命 | 阻断核心流程，无法继续测试 | 立即 | P0 |
| 严重 | 核心功能异常，影响主流程 | 4小时内 | P0 |
| 一般 | 非核心功能异常，不影响主流程 | 24小时内 | P1 |
| 优化 | 体验/规范类问题 | 迭代内 | P2 |

### 5.2 Bug提交规范

每个Bug必须包含以下信息：

```markdown
## Bug ID: BUG-XXX-001

**所属模块**: 数据导入  
**严重等级**: 严重  
**Bug标题**: 批量导入时重复文件未正确跳过  
**复现步骤**:
1. 准备包含重复文件的目录
2. 执行 `nanobotrun data import <dir>`
3. 观察导入结果

**实际结果**: 重复文件被重复导入  
**预期结果**: 重复文件应提示"已存在，跳过"  
**根因分析**: SHA256去重逻辑未正确处理批量场景  
**修复建议**: 在批量导入循环中添加去重检查  
**出现版本**: v0.9.4  
**优先级**: P0  
**创建时间**: 2026-04-19  
**状态**: 待修复
```

### 5.3 Bug跟踪流程

```
发现Bug → 提交Bug清单 → 开发工程师修复 → 回归测试 → 状态更新
    ↓                                              ↓
  记录日志                                      已闭环/驳回
```

**Bug状态**:
- 待修复：已提交，等待修复
- 修复中：开发工程师正在修复
- 待回归：修复完成，等待回归测试
- 已闭环：回归测试通过
- 已驳回：修复无效，退回重新修复

### 6.2 测试数据管理

**测试数据来源**:
- 使用 `tests/data/fixtures/` 下的样本FIT文件
- 禁止使用真实用户数据
- 测试数据必须脱敏

**临时目录管理**:
```python
import tempfile
import shutil
from pathlib import Path

@pytest.fixture
def temp_data_dir():
    """创建临时测试目录"""
    temp_dir = Path(tempfile.mkdtemp())
    yield temp_dir
    shutil.rmtree(temp_dir)  # 清理
```

## 8. 自动化测试规范

### 8.1 单元测试规范

```python
import pytest
from pathlib import Path
from unittest.mock import MagicMock, patch

class TestFitParser:
    """FIT文件解析器单元测试"""
    
    def test_parse_valid_file(self):
        """测试正常FIT文件解析"""
        # Arrange
        parser = FitParser()

        # Act
        result = parser.parse_file(fit_path)
        
        # Assert
```

### 8.2 集成测试规范

```python
import pytest
from pathlib import Path
from src.core.context import get_context

class TestDataImportFlow:
    """数据导入流程集成测试"""
    
    @pytest.fixture
    def temp_workspace(self, tmp_path):
        """创建临时工作区"""
        # 设置临时目录
        return tmp_path
    
    def test_import_to_query_flow(self, temp_workspace):
        """测试导入到查询完整链路"""
        # Arrange
        context = get_context(workspace_dir=temp_workspace)
        
        # Act - 导入
        import_result = context.storage.import_file(
            Path("tests/data/fixtures/easy_run_20240101.fit")
        )
        
        # Act - 查询
        stats = context.analytics.get_stats()
        
        # Assert
        assert import_result.success
        assert stats["total_runs"] >= 1
```

## 9. 持续集成要求

### 9.1 CI流水线检查项

| 阶段 | 检查项 | 失败处理 |
|------|--------|---------|
| 代码质量 | `ruff check` 零警告 | 阻止合并 |
| 代码格式 | `ruff format --check` 通过 | 阻止合并 |
| 类型检查 | `mypy` 无新增错误 | 阻止合并 |
| 单元测试 | 通过率100% | 阻止合并 |
| 集成测试 | 通过率≥95% | 阻止合并 |
| 覆盖率 | 达到最低要求 | 警告，不阻止 |
| 安全扫描 | 无敏感信息 | 阻止合并 |

### 9.2 本地预检查

提交前必须执行的本地检查：

```bash
# Windows PowerShell
uv run ruff format --check src/ tests/; if($?) { 
    uv run ruff check src/ tests/; if($?) { 
        uv run mypy src/ --ignore-missing-imports; if($?) { 
            uv run pytest tests/unit/ --cov=src 
        } 
    } 
}
```

## 10. 测试左移策略

### 10.1 架构设计阶段介入

- 参与架构评审，识别可测试性问题
- 制定测试策略，明确测试范围和方法
- 输出核心用例框架，与开发对齐测试标准

### 10.2 开发阶段介入

- 审核开发工程师的单元测试覆盖率
- 提供测试数据构造指导
- 提前设计集成测试用例，等待开发完成后立即执行

### 10.3 测试阶段介入

- 执行场景级集成测试和E2E测试
- 提交Bug并跟踪修复进度
- 输出测试报告，把控上线门禁

## 11. 风险应对方案

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 测试数据不足 | 用例覆盖不全 | 提前构造测试数据，使用Mock补充 |
| 环境不一致 | 测试结果不可靠 | 使用临时目录，隔离测试环境 |
| 第三方API不稳定 | 集成测试失败 | 全面Mock外部依赖 |
| 性能退化 | 用户体验下降 | 定期执行性能基准测试 |
| 需求变更 | 用例失效 | 及时更新用例，保持同步 |
