## 一、文档概述

### 1.1 目的

本文档为 v0.4.0 版本提供完整的人工验收测试执行手册，用于指导测试人员按照标准化流程完成所有验收测试用例的执行与结果记录。本文档重点覆盖三个核心应用场景：

1. **nanobotrun CLI命令模式**：通过终端直接执行独立命令，如 `import`、`stats`、`profile show`、`plan generate` 等
2. **nanobotrun chat模式**：启动交互式聊天，通过自然语言与 LLM 对话，如 `uv run nanobotrun chat`
3. **飞书App**：飞书机器人交互模式，用户通过飞书App与"Nanobot-ai助手"机器人对话

### 1.2 适用范围

本文档适用于 v0.4.0 版本的最终验收测试，涵盖以下功能模块：

| 模块                  | 功能点                    | 优先级 | 测试场景                         |
| ------------------- | ---------------------- | --- | ---------------------------- |
| Profile 用户画像        | 画像构建、双存储、新鲜度检查、异常过滤    | P0  | CLI命令 / chat模式              |
| TrainingPlan 训练计划   | 计划生成、分阶段训练、动态调整        | P0  | CLI命令                        |
| RacePrediction 比赛预测 | VDOT 预测、配速计算、置信度评估     | P0  | CLI命令 / chat模式              |
| ReportService 报告服务  | 日报/周报/月报生成、飞书推送、定时调度   | P0  | CLI命令 / chat模式 / 飞书App      |
| FeishuCalendar 日历同步 | 训练计划同步、冲突检测、Webhook 处理 | P1  | 飞书App                        |
| FeishuBot 机器人交互     | 命令解析、卡片消息、快捷指令         | P1  | 飞书App                        |
| 数据导入                | FIT文件解析、去重、存储          | P0  | CLI命令                        |
| 统计查询                | 训练统计、数据查询              | P0  | CLI命令 / chat模式 / 飞书App      |
| 自然语言交互              | Agent对话、工具调用           | P0  | chat模式 / 飞书App              |

### 1.3 测试通过标准

| 指标       | 标准   | 说明             |
| -------- | ---- | -------------- |
| P0 功能通过率 | 100% | 所有 P0 功能必须全部通过 |
| P1 功能通过率 | ≥95% | 允许少量非关键功能缺陷    |
| 严重缺陷数    | 0    | 不允许存在严重及以上级别缺陷 |
| 一般缺陷数    | ≤3   | 一般缺陷需有规避方案     |
| 测试用例执行率  | 100% | 所有测试用例必须执行     |

## 二、测试环境配置

### 2.1 硬件环境

| 项目  | 最低要求      | 推荐配置       |
| --- | --------- | ---------- |

```bash
# 1. 克隆仓库
git clone https://github.com/yecllsl/nanobot-runner.git
cd nanobot-runner

# 2. 创建虚拟环境并安装依赖
uv venv
uv sync --all-extras

# 3. 验证安装
uv run nanobotrun --version
uv run nanobotrun --help
```

## 四、缺陷等级划分标准

### 4.1 缺陷严重等级定义

| 等级     | 定义                 | 示例                      | 处理优先级    |
| ------ | ------------------ | ----------------------- | -------- |
| **致命** | 阻断核心业务流程，系统崩溃或数据丢失 | 程序崩溃、数据损坏、核心功能完全不可用     | P0，立即修复  |
| **严重** | 核心功能异常，影响主流程使用     | 画像生成失败、训练计划无法生成、数据导入失败  | P0，当日修复  |
| **一般** | 非核心功能异常，不影响主流程     | 界面显示错误、非关键数据不准确、提示信息不友好 | P1，迭代内修复 |
| **优化** | 体验/规范类问题，不影响功能使用   | UI优化、性能优化、代码规范问题        | P2，择机修复  |

### 4.2 缺陷状态流转

```
新建 → 待修复 → 修复中 → 待验证 → 已关闭
                    ↓
                  驳回 → 待修复
```

### 4.3 缺陷记录模板

每个缺陷需记录以下信息：

| 字段     | 说明            |
| ------ | ------------- |
| Bug ID | 唯一标识，格式：模块-序号 |
| 所属模块   | 缺陷所在模块        |
| 严重等级   | 致命/严重/一般/优化   |
| 优先级    | P0/P1/P2      |
| 标题     | 简明扼要描述问题      |
| 复现步骤   | 详细的重现步骤       |
| 预期结果   | 期望的正确行为       |
| 实际结果   | 实际观察到的错误行为    |
| 根因分析   | 问题产生的原因       |
| 修复建议   | 建议的修复方案       |
| 发现版本   | 发现缺陷的版本号      |
| 状态     | 当前状态          |

## 十三、附录

### 12.1 常见问题

**Q1: FIT文件导入失败怎么办？**
A: 检查文件格式是否正确，确保是有效的FIT文件。对于COROS设备导出的文件，系统已支持非标准字段大小。

**Q2: 用户画像VDOT值异常怎么办？**
A: VDOT计算基于Jack Daniels公式，需要距离>=1500m的跑步记录。确保导入的数据包含足够的跑步记录。

**Q3: 偏好训练时间不准确怎么办？**
A: 系统使用北京时间(UTC+8)分析训练时间，确保系统时间正确。

**Q4: 飞书推送失败怎么办？**
A: 检查飞书Webhook配置是否正确，确保网络连接正常。
