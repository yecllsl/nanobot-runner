## 四、测试用例设计与执行策略

### 4.1 测试层级与优先级

```
L1 单元测试 (优先级最高)
  └── body_signal模块: HRVAnalyzer, FatigueAssessor, RecoveryMonitor, BodySignalEngine
  └── 目标覆盖率: >= 85%

L2 集成测试 (次优先级)
  └── CLI命令集成: status today/weekly, analysis hrv/fatigue/recovery
  └── Agent工具集成: 6个新增工具
  └── 端到端流程: Parquet → Analyzer → Engine → Summary

L3 场景级集成测试
  └── 核心业务流: 导入FIT → 分析HRV → 评估疲劳 → 生成建议
  └── 降级场景: 空数据/单点数据/数据不足时的正确降级

L4 性能测试
  └── 500条年度数据下的查询响应时间

L5 回归测试
  └── v0.18.0已有功能不受v0.19.0影响
```

## 五、测试执行计划

### 5.1 执行顺序

```
Phase 1: 准入检查 (Ruff/Mypy/代码审查)
  ↓
Phase 2: 单元测试 (body_signal模块, 目标覆盖率>=85%)
  ↓
Phase 3: 集成测试 (CLI + Agent + E2E)
  ↓
Phase 4: 场景级集成测试 (完整业务流 + 降级场景)
  ↓
Phase 5: 性能测试 (响应时间验证)
  ↓
Phase 6: 回归测试 (已有功能不受影响)
  ↓
Phase 7: 质量评估与报告输出
```

### 5.2 测试环境与数据

| 项 | 说明 |
|----|------|
| 运行环境 | Windows PowerShell, Python 3.11+, uv |
| 测试数据 | 使用 `tests/data/fixtures/` 下的样本FIT文件 |
| Mock策略 | Mock SessionRepository/TrainingLoadAnalyzer，不Mock内部业务逻辑 |
| 隐私要求 | 严禁使用真实用户数据，全部使用虚构或fixture数据 |

### 5.3 风险应对

| 风险 | 应对方案 |
|------|---------|
| HRV估算准确性不足 | 明确标注非医疗级精度，测试重点验证降级提示 |
| 疲劳度模型普适性 | 测试权重配置功能，验证自定义权重有效 |
| FIT数据缺少RR间期 | 测试HR_ESTIMATE降级路径，确保不崩溃 |
| 覆盖率不达标 | 针对未覆盖分支补充用例，重点覆盖边界条件 |
