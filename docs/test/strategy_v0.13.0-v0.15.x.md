## 1. 测试策略概述

### 1.1 测试目标

基于探索版本产品规划v4.0，制定全量测试策略，确保三个版本的核心功能质量：

| 版本 | 核心功能 | 测试重点 |
|------|----------|----------|
| **v0.13.0 智能技能生态版** | 智能技能库（4个内置技能）、外部工具接入、技能与工具协同 | 技能启用成功率、工具调用成功率、隐私保护、多技能协同、技能冲突管理 |
| **v0.14.0 AI教练进化版** | AI教练长期记忆、人格进化、个性化建议优化、Dream自动整理 | 记忆连贯性、人格进化效果、偏好学习准确率、记忆版本回溯 |
| **v0.15.0 透明洞察版** | AI决策透明化、AI教练状态看板、用户训练洞察报告、全链路可观测性 | 决策过程展示完整性、状态看板数据准确性、洞察报告价值、性能开销 |

### 1.2 测试原则

| 原则 | 说明 |
|------|------|
| **隐私保护优先** | 所有测试必须验证数据不出本地，外部工具调用需用户授权，记忆数据本地存储 |
| **场景驱动** | 测试用例100%基于用户旅程地图和核心业务场景设计 |
| **测试左移** | 开发阶段同步进行单元测试，集成测试紧随功能开发 |
| **自动化优先** | 核心流程100%自动化，回归测试全自动执行 |
| **数据驱动** | 使用脱敏真实数据+构造边界数据，确保测试真实性 |
| **底座能力验证** | 验证nanobot-ai 0.1.5.post2底座能力（SKILL、Dream、Memory、Observability、MyTool）正确集成 |

### 1.3 测试范围

#### 1.3.1 纳入测试范围

## 3. 门禁规则

### 3.1 测试准入规则

代码必须满足以下条件才能进入测试阶段：

| 检查项 | 要求 | 验证方式 |
|--------|------|----------|
| **代码规范** | `ruff format --check` 零警告 | CI自动检查 |
| **代码质量** | `ruff check` 零警告 | CI自动检查 |
| **类型检查** | `mypy` 零错误 | CI自动检查 |
| **单元测试** | 覆盖率达标（core≥80%, agents≥70%, cli≥60%） | pytest-cov报告 |
| **单元测试通过率** | 100% | pytest执行结果 |
| **文档更新** | 用户文档同步更新 | 人工review |
| **自测通过** | 开发工程师提供自测报告 | 自测报告review |

**准入流程**:

```
开发完成 → 代码规范检查 → 单元测试 → 自测报告 → 测试工程师审核 → 准入测试
```

### 3.2 测试准出规则

测试必须满足以下条件才能输出测试通过报告：

| 检查项 | v0.13.0要求 | v0.14.0要求 | v0.15.0要求 |
|--------|-------------|-------------|-------------|
| **P0用例通过率** | 100% | 100% | 100% |
| **P1用例通过率** | ≥95% | ≥95% | ≥95% |
| **P2用例通过率** | ≥80% | ≥80% | ≥80% |
| **致命/严重Bug** | 0个 | 0个 | 0个 |
| **一般Bug修复率** | ≥90% | ≥90% | ≥90% |
| **性能指标** | 全部达标 | 全部达标 | 全部达标 |
| **安全测试** | 全部通过 | 全部通过 | 全部通过 |
| **回归测试** | 上一版本核心功能100%通过 | v0.13.0核心功能100%通过 | v0.13.0+v0.14.0核心功能100%通过 |

**准出流程**:

```
测试执行 → Bug修复跟踪 → 回归测试 → 测试报告 → 质量评估 → 准出发布
```

### 3.3 上线门禁

只有同时满足以下条件，才能放行进入发布环节：

| 条件 | 说明 |
|------|------|
| ① P0-P1级用例100%通过 | 核心业务流程无遗漏 |
| ② 无致命/严重级Bug | 质量底线 |
| ③ 一般级Bug修复率≥90% | 可接受的技术债务 |
| ④ 核心业务流程全量闭环 | E2E测试全部通过 |
| ⑤ 符合需求验收标准 | 产品规划中的成功指标达成 |
| ⑥ 隐私保护测试全部通过 | 数据安全无风险 |
| ⑦ 性能无退化 | 用户感知响应速度≥上一版本90% |
| ⑧ 技能生态验证通过 | 内置技能可用率100%，自定义技能导入成功率>95% |
| ⑨ 记忆连贯性验证通过 | 跨会话记忆恢复成功率100% |

**不满足上线标准的处理**:

- 明确告知用户与发布运维智能体**禁止发布**
- 输出《不通过原因与整改清单》
- 跟踪修复直到满足上线标准

## 7. Bug管理规范

### 7.1 Bug严重等级定义

| 等级 | 定义 | 示例 | 修复时限 |
|------|------|------|----------|
| **致命** | 阻断核心业务流程，无法继续测试 | 技能启用导致系统崩溃、记忆数据丢失 | 24小时内 |
| **严重** | 核心功能异常，影响主流程使用 | 天气工具返回错误数据、跨会话记忆不连贯 | 48小时内 |
| **一般** | 非核心功能异常，不影响主流程 | 技能配置界面显示异常、状态看板数据延迟 | 版本发布前 |
| **优化** | 体验/规范类问题，不影响功能使用 | 提示信息不够友好、展示格式可优化 | 后续版本 |

### 7.2 Bug提交流程

```
发现Bug → 记录Bug信息 → 提交给开发工程师 → 跟踪修复进度 → 回归验证 → 闭环
```

### 7.3 Bug信息模板

| 字段 | 说明 |
|------|------|
| **Bug ID** | 格式：`BUG-{版本}-{序号}` |
| **所属模块** | 发现Bug的模块 |
| **严重等级** | 致命/严重/一般/优化 |
| **Bug标题** | 简洁描述问题 |
| **复现步骤** | 详细的操作步骤 |
| **实际结果** | 实际发生的现象 |
| **预期结果** | 期望的正确结果 |
| **根因分析** | 问题根本原因（开发工程师填写） |
| **修复建议** | 修复方案建议 |
| **出现版本** | 首次发现的版本 |
| **状态** | 待修复/修复中/待回归/已闭环/已驳回 |

## 10. 风险与应对

### 10.1 测试风险

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 外部Mock服务不稳定 | 中 | 中 | 准备多套Mock方案，本地文件兜底 |
| 个性化效果验证周期长 | 高 | 中 | 使用模拟数据加速验证，真实数据补充 |
| 测试时间被压缩 | 高 | 高 | 测试左移，开发阶段同步介入 |
| 测试数据不足 | 中 | 中 | 提前准备测试数据集，脚本批量生成 |
| 技能冲突场景复杂 | 中 | 中 | 优先测试核心冲突场景，边界场景后续补充 |

### 10.2 质量风险

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 隐私保护测试遗漏 | 低 | 高 | 建立隐私测试检查清单，逐项验证 |
| 记忆数据泄露 | 低 | 高 | 专项安全测试，验证记忆数据完全本地 |
| 性能退化未及时发现 | 中 | 中 | 每次发布前执行性能基准对比 |
| 回归测试不充分 | 中 | 高 | 自动化回归测试，核心功能100%覆盖 |
| 人格进化效果不达预期 | 中 | 高 | 设置效果阈值，未达标时要求调整Dream参数 |
