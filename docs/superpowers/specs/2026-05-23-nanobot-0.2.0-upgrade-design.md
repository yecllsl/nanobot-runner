# nanobot-ai 0.2.0 底座升级与新特性适配设计

> **日期**: 2026-05-23
> **状态**: 已确认
> **涉及版本**: v0.26.0 - v0.29.0
> **当前基线**: v0.25.0

---

## 1. 背景与动机

nanobot-ai 已发布 0.2.0 版本，引入了 GoalState、FallbackProvider、WebUI 内置打包、Model Presets、推理可见化等重要新能力。RunFlowAgent 当前依赖 `nanobot-ai>=0.1.5.post2`，需要升级底座并规划新特性适配。

**核心需求**：
1. 底座升级到 nanobot-ai 0.2.0，保证已有功能正常
2. 根据 nanobot-ai 0.2.0 新功能，更新现有功能和提供新功能
3. 依托 nanobot-ai 的 WebUI 能力，规划 RunFlowAgent 的 WebUI 功能

**版本策略**：渐进式，从 v0.26.0 开始，每个版本聚焦一个主题。

---

## 2. 方案选型

### 2.1 升级策略

**选定**：保守兼容优先

- 先升级依赖，运行全量测试，逐个修复失败用例
- 确保零功能回归后再进入下个版本
- 不在升级版本中引入新功能

### 2.2 新特性选择

**选定**：GoalState 持久化目标 + 推理可见化 + Model Presets 预设管理

- FallbackProvider 和 MCP Resources/Prompts 映射推迟到 v0.31.0 评估是否加入，不是彻底放弃

### 2.3 WebUI 建设方式

**选定**：复用 nanobot-ai 原生 WebUI

**可行性论证**：

| 维度 | 论证 |
|------|------|
| 工具调用 | RunFlowAgent 的所有工具已注册为 nanobot-ai 工具，WebUI 通过 WebSocket 协议与 Agent 交互，工具调用天然支持 |
| 流式对话 | nanobot-ai 0.2.0 的 WebSocket 多路复用协议支持 delta/stream_end/reasoning_delta |
| 会话管理 | 原生 WebUI 支持多会话、会话标题自动生成、历史记录 |
| 设置面板 | 原生 WebUI 的设置 API 支持通用配置，可扩展 RunFlowAgent 专属配置项 |
| 部署方式 | WebUI 随 wheel 分发，`nanobot gateway` 启动即可用 |

**后遗症及缓解**：

| 后遗症 | 缓解措施 |
|--------|----------|
| 上游 UI 变更影响 | 锁定 nanobot-ai 版本；自定义组件独立维护 |
| 跑步数据无图表 | v0.28.0 扩展；v0.27.0 通过 Agent 文本/表格输出 |
| 品牌不一致 | 通过 `bot_name`/`bot_icon` 配置自定义 |
| 设置面板不完整 | v0.27.0 通过 Agent 对话修改配置；v0.28.0+ 扩展设置 API |

### 2.4 版本节奏

**选定**：方案 A 渐进式（4个版本）

渐进式开发可在过程中发现问题、及时调整方案。API 兼容性验证后风险很低，底座升级与新特性适配合并在同一版本完成。

## 3. v0.26.0 — 底座升级 + 新特性适配

**主题**：nanobot-ai 0.2.0 底座升级 + GoalState/推理可见化/Model Presets 适配

### 3.1 核心目标

- 将 `nanobot-ai>=0.1.5.post2` 升级为 `nanobot-ai>=0.2.0`
- 全量测试通过，所有现有功能正常工作
- 适配 GoalState、推理可见化、Model Presets 三项新特性

### 3.2 关键工作项

| 工作项 | 描述 | 风险等级 |
|--------|------|----------|
| 依赖升级 | `pyproject.toml` 中 `nanobot-ai>=0.1.5.post2` → `nanobot-ai>=0.2.0` | 低 |
| API 兼容性检查 | 检查 RunFlowAgent 使用的 nanobot-ai API 是否有 breaking change | 低（0.2.0 对 AgentHook 接口纯增量扩展，无破坏性变更） |
| 测试修复 | 运行全量测试，修复因 API 变更导致的失败用例 | 中 |
| 配置迁移 | 检查 nanobot-ai 0.2.0 配置 Schema 变更，适配 RunFlowAgent 的 config.json | 低 |
| 依赖冲突预检 | 检查 nanobot-ai 0.2.0 间接依赖是否与 RunFlowAgent 现有依赖冲突（如 pydantic≥2.12.0） | 中 |
| config.json Schema 兼容 | 用 nanobot-ai 0.2.0 的 Config loader 测试加载现有 config.json，确认新增字段不破坏旧配置 | 低 |
| ToolRegistry 注册验证 | 检查 `tools.py` 和 `tools_evolution.py` 中的工具注册 API 是否有变更 | 低 |
| GoalState 适配 | SOUL.md 注入使用指导 + DecisionLogHook 读取 metadata | 低 |
| 推理可见化适配 | DecisionLogHook 重写 `emit_reasoning()` | 低 |
| Model Presets 适配 | config.json 配置预设 + CLI 查看命令 | 低 |

### 3.3 验收标准

**底座升级**：
1. `uv run pytest tests/` 全量通过
2. `uv run ruff check src/` 无新增错误
3. `uv run mypy src/ --ignore-missing-imports` 无新增错误
4. CLI 核心命令（data import、analysis vdot、plan create、evolution status）功能正常
5. 核心 CLI 命令响应时间不退化 >20%（以 v0.25.0 为基线）
6. `uv pip install --dry-run` 无依赖冲突

**GoalState**：
7. 创建训练计划后，Agent 在新对话中能回忆当前计划目标
8. GoalState 状态在 WebUI（v0.27.0+）中可见

**推理可见化**：
9. 飞书对话中可见 Agent 推理过程
10. 推理内容与最终建议逻辑一致

**Model Presets**：
11. 可通过配置定义多个 Model Presets
12. CLI 命令可查看当前预设列表
13. 飞书/WebUI 中可通过 `/model <preset>` 切换预设（nanobot-ai 内置 chat command，RunFlowAgent 仅负责提供预设配置）

### 3.4 不做的事

- 不适配 FallbackProvider、MCP Resources/Prompts（作为独立评估任务，不占版本号）
- 不做 WebUI 相关工作
- 不重构现有代码
- 不修改用户配置文件格式

## 4. v0.27.0 — WebUI 基础

**主题**：复用 nanobot-ai 原生 WebUI，实现 AI 对话交互 + 基础设置

### 4.1 功能范围

| 功能 | 描述 | 来源 |
|------|------|------|
| AI 对话交互 | 通过 WebUI 与 AI 教练对话，获取训练建议、数据分析 | 原生 WebUI |
| 流式输出 | Agent 回复逐字流式展示 | 原生 WebSocket |
| 推理展示 | 显示 Agent 推理过程（v0.27.0 已适配） | 原生 WebUI |
| 多会话管理 | 创建/切换/删除对话，自动标题 | 原生 WebUI |
| 基础设置 | 模型切换、Provider 配置、时区等 | 原生 WebUI |
| GoalState 展示 | 显示当前持久化目标状态 | 原生 WebUI |
| 品牌自定义 | bot_name/bot_icon 配置为 RunFlowAgent | 原生配置 |

### 4.2 实现方式

- 启用 nanobot-ai 的 WebSocket Channel 配置
- 配置 `nanobot gateway` 启动参数
- 在 RunFlowAgent 的 `config.json` 中添加 WebSocket 相关配置
- 自定义 SOUL.md / AGENTS.md 模板，确保 Agent 在 WebUI 中有正确的身份和行为
- 可选：添加 CLI 命令 `nanobotrun webui` 一键启动

### 4.3 验收标准

1. `nanobot gateway` 启动后，浏览器访问 WebUI 可正常对话
2. 所有 RunFlowAgent 工具在 WebUI 中可正常调用
3. 流式输出、推理展示正常
4. 多会话创建/切换正常
5. 基础设置（模型切换）正常

## 5. v0.28.0 — WebUI 数据可视化

**主题**：扩展 WebUI，增加跑步数据可视化能力

### 5.1 功能范围

| 功能 | 描述 | 实现方式 |
|------|------|----------|
| VDOT 趋势图 | 展示 VDOT 随时间变化的折线图 | 扩展 WebUI 前端组件 |
| 训练负荷图 | 展示 ATL/CTL/TSB 趋势 | 扩展 WebUI 前端组件 |
| 身体信号面板 | 展示 HRV、疲劳度、恢复状态 | 扩展 WebUI 前端组件 |
| 数据概览仪表盘 | 近7/30/90天跑步数据汇总 | 扩展 WebUI 前端组件 |

### 5.2 技术方案

通过 Agent 工具返回结构化数据 + WebUI 前端渲染：

- 后端：新增 WebUI 专用 API 端点（在 nanobot-ai 的 HTTP API 基础上扩展），返回 JSON 格式的图表数据
- 前端：在原生 WebUI 基础上新增"数据"标签页，使用 Recharts/ECharts 渲染图表
- 数据流：WebUI 前端 → HTTP API → RunFlowAgent 后端 → Polars 查询 → JSON 响应 → 图表渲染

### 5.3 验收标准

1. WebUI 中可查看 VDOT 趋势图、训练负荷图
2. 图表数据与 CLI 命令输出一致
3. 图表支持时间范围筛选（7/30/90/365天）

## 7. 独立评估任务（不占版本号）

**主题**：FallbackProvider 故障转移 + MCP Resources/Prompts 评估

> 说明：此两项工作为评估性质，根据评估结论决定是否在后续版本中实施，不单独占用版本号。

### 7.1 FallbackProvider 故障转移

**用户价值**：LLM 调用失败时自动切换到备用模型，提升系统稳定性

**评估要点**：
- RunFlowAgent 当前是否有多 Provider 配置需求
- 故障转移对训练建议/预测结果一致性的影响
- 断路器模式与进化引擎 PromptTuner 的交互

**可能的应用场景**：
- 主模型（如 Claude）限流时，自动切换到备用模型（如 GPT-4）
- 训练提醒等低风险场景使用快速备用模型

### 7.2 MCP Resources/Prompts 映射

**用户价值**：通过 MCP 协议暴露跑步数据资源和提示模板，支持更丰富的工具生态

**评估要点**：
- 当前 MCP Tool 映射是否已满足需求
- Resources（如跑步数据查询）和 Prompts（如训练建议模板）的增量价值
- 与 WebUI 数据可视化（v0.29.0）的协同可能性

### 7.3 验收标准

1. 完成 FallbackProvider 评估报告，明确是否采纳及实施方案
2. 完成 MCP Resources/Prompts 评估报告，明确是否采纳及实施方案
3. 若采纳，相关功能开发完成并通过测试

## 9. 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| nanobot-ai 0.2.0 API breaking change | 低 | 0.2.0 对 AgentHook 接口纯增量扩展，无破坏性变更；v0.26.0 全量测试覆盖 |
| 间接依赖冲突 | 中 | `uv pip install --dry-run` 预检；nanobot-ai 0.2.0 要求 pydantic≥2.12.0 |
| config.json Schema 不兼容 | 低 | 0.2.0 新增字段均有默认值，Pydantic 向后兼容；用 0.2.0 Config loader 测试加载 |
| 原生 WebUI 不支持跑步数据可视化 | 中 | v0.27.0 先验证基础链路，v0.28.0 再扩展 |
| 上游 WebUI 版本变更影响 | 低 | 锁定 nanobot-ai 版本；自定义组件独立维护 |
| FallbackProvider 推迟导致 LLM 调用稳定性风险 | 低 | 当前单 Provider 场景下风险可控；独立评估后决定是否纳入后续版本 |
| MCP Resources/Prompts 推迟导致工具扩展性受限 | 低 | 当前 MCP Tool 映射已满足需求；独立评估后决定是否纳入后续版本 |
