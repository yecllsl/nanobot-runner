## 为什么要聊开发流程？

很多人可能会问：你这个项目不是一个人开发的吗？还需要什么"协作链路"？

确实，Nanobot Runner从v0.1到v0.28，全程只有我一个人在开发。但正是这一路的探索，让我深刻体会到：**流程和方法论不是为团队准备的，而是为"靠谱"准备的**。

一个人开发最大的风险不是技术，而是"自己骗自己"：
- 功能做完了，但测试呢？
- Bug修好了，但根因分析过吗？
- 觉得没问题，但验证过吗？

Nanobot Runner的28个版本，每个版本都经过了完整的开发-测试-发布流程。这不是"形式主义"，而是**保证质量的必要手段**。

本章将带你了解：
- 六类角色与核心指令
- 项目Skill主链路与全局Skill嵌入
- 准入准出规则与异常处理
- 单人开发的流程裁剪指南

## 3.1 六类角色与核心指令

### 3.1.1 为什么需要角色划分？

在团队开发中，不同人有不同职责：产品经理定方向，架构师设计方案，开发工程师写代码，测试工程师找bug。

但即使是一个人，这套角色体系依然有价值。**角色不是人，而是职责的抽象**。我可以在不同阶段扮演不同角色：

- 写第一章时，我是**产品经理**：确定愿景、规划路线图
- 写第二章时，我是**架构师**：设计Schema、选择技术栈
- 写代码时，我是**开发工程师**：TDD、单元测试、代码优化
- 提测前，我是**测试工程师**：制定测试策略、验证功能
- 发版时，我是**发布运维**：CICD、自动化发布、监控

### 3.1.2 六类角色职责

| 角色 | 核心职责 | 关键交付物 |
|------|----------|------------|
| **产品经理** | 产品规划，确定方向 | 产品规划方案 |
| **架构师** | 需求分析、架构设计、任务拆解、方案评审 | 需求规格说明书、架构设计方案、任务清单 |
| **开发工程师** | 功能开发、单元测试、Bug修复、代码优化 | 功能代码、单元测试、开发交付报告 |
| **测试工程师** | 测试策略、用例设计、测试执行、质量评估 | 测试用例、测试报告、Bug清单、上线结论 |
| **发布运维** | CICD搭建、自动化发布、FIX-CI、监控配置 | 流水线配置、发布报告、运维文档 |
| **代码评审工程师** | 代码质量评审、架构一致性检查、安全审计 | 代码评审报告 |

## 3.2 项目Skill主链路

### 3.2.1 全链路协作流程

```
产品经理（产品规划）
        ↓
架构师（需求分析 → 架构设计 → 架构评审 → 任务拆解）
        ↓
开发工程师（功能开发 → 单元测试 → 本地验证）
        ↓
代码评审工程师（代码评审）
        ↓
测试工程师（测试策略 → 测试执行 → Bug反馈）
        ↓
开发工程师（Bug修复）
        ↓
测试工程师（回归测试 → 上线结论）
        ↓
发布运维（文档更新 → CICD验证 → 自动化发布 → 文档归档）
```

### 3.2.2 为什么单人开发也要走全链路？

你可能会想：我一个人开发，这些流程是不是太形式化了？

让我用Nanobot Runner的实际经历告诉你答案。

**v0.20.0的教训**

v0.20.0是"ML增强预测"的大版本。我当时觉得功能已经完成了，就直接跳过了测试阶段，开始做v0.21。结果v0.21开发到一半，发现v0.20的预测模块有bug——VDOT预测误差超过了预期。

**被迫回滚**，重新修复bug、补测试，浪费了两天时间。

**v0.25.0的正面案例**

吸取教训后，v0.25.0（自适应进化引擎）严格按照流程走：

```
v0.25.0开发流程：
Day 1: 需求确认 → 架构设计 → 任务拆解
Day 2-5: 功能开发 + 单元测试（TDD）
Day 6: 代码评审
Day 7: 测试策略 → 测试执行 → Bug修复
Day 8: 回归测试 → 上线结论
Day 9: 文档更新 → CICD验证 → 发布
```

结果：**一次上线，零bug遗留**，后续版本基于v0.25开发，基础非常扎实。

**结论**：流程不是负担，而是**质量保障的护栏**。跳过流程 = 欠债，迟早要还。

## 3.3 全局Skill嵌入：四大方法论

### 3.3.1 TDD：测试驱动开发

**什么时候用？**

编写任何功能代码或修复代码时，必须先写测试。

**三步循环**：

```
RED（红）→ GREEN（绿）→ REFACTOR（重构）
 ↓          ↓          ↓
写失败测试   写最少实现   重构优化
```

**Nanobot Runner的实践**：

```python
# 例子：VDOT计算器的TDD流程

# Step 1: RED - 先写失败测试
def test_vdot_calculation_below_minimum_distance():
    """距离小于1500m时，VDOT应为0"""
    calculator = VDOTCalculator()
    result = calculator.calculate_vdot(1000, 300)  # 1000米，5分钟
    assert result == 0.0

# Step 2: GREEN - 写最少实现
class VDOTCalculator:
    def calculate_vdot(self, distance_m: float, time_s: float) -> float:
        if distance_m < 1500:
            return 0.0
        # 简化实现
        return 0.0

# Step 3: GREEN - 补充完整实现
class VDOTCalculator:
    def calculate_vdot(self, distance_m: float, time_s: float) -> float:
        if distance_m < 1500:
            return 0.0
        # 完整Jack Daniels公式
        time_min = time_s / 60.0
        pace_min_per_km = time_min / (distance_m / 1000.0)
        vdot = 85.46 - 7.08 * pace_min_per_km + 0.15 * pace_min_per_km**2
        return round(max(0.0, vdot), 2)

# Step 4: REFACTOR - 优化（添加批量计算）
def test_vdot_batch_calculation():
    """向量化批量计算"""
    df = pl.DataFrame({
        "distance": [5000, 10000, 42195],
        "time": [1500, 3000, 12600]
    })
    calculator = VDOTCalculator()
    result = calculator.calculate_vdot_batch(df)
    assert len(result) == 3
```

### 3.3.2 systematic-debugging：系统化调试

**什么时候用？**

Bug根因不明或测试失败时。

**四步法**：

```
Phase 1: 收集证据 → 测试用例、日志、错误信息
Phase 2: 形成假设 → 可能的原因（≥3个）
Phase 3: 验证假设 → 逐一排除，定位根因
Phase 4: 修复验证 → 确保修复有效，不引入新bug
```

**Nanobot Runner的实践**：

```python
# 例子：VDOT预测误差异常的调试

# Phase 1: 收集证据
"""
测试失败日志：
FAILED test_vdot_predictor.py::test_vdot_mae
AssertionError: MAE 0.52 > 0.05

错误上下文：
- 最近7天数据正常
- 最近30天数据异常
- 训练集正常，测试集异常
"""

# Phase 2: 形成假设
"""
假设1: 特征工程有问题（最近30天）
假设2: 数据分布漂移（训练集 vs 测试集）
假设3: 模型过拟合（只拟合了最近数据）
假设4: 特征缺失（缺少某个关键特征）
"""

# Phase 3: 验证假设
"""
验证假设1: 检查最近30天的特征计算
结果: 特征工程正常，否定了假设1

验证假设2: 检查数据分布
结果: 发现测试集VDOT均值比训练集高2.3
结论: 假设2成立，根因是数据分布漂移

验证假设3: 检查过拟合
结果: 过拟合指标正常，排除假设3

验证假设4: 检查特征
结果: 发现缺少"训练类型"特征
结论: 假设4也成立，但主因是假设2
"""

# Phase 4: 修复验证
"""
修复方案: 添加数据归一化 + 增加"训练类型"特征
修复后: MAE 0.03 < 0.05 ✓
无新bug引入 ✓
"""
```

### 3.3.3 verification-before-completion：完成前验证

**什么时候用？**

声称完成、修复、通过之前，必须有新鲜验证证据。

**禁止的行为**：

- ❌ "应该没问题"（没有证据）
- ❌ "之前测过可以"（证据过期）
- ❌ "我检查过了"（缺少客观证据）

**正确的做法**：

- ✅ 运行pytest，通过截图作为证据
- ✅ 运行coverage，获得覆盖率报告
- ✅ 运行lint/mypy，获得质量报告

**Nanobot Runner的实践**：

```bash
# 每次commit前必须运行的验证命令
uv run ruff format src/ tests/     # 代码格式化
uv run ruff check src/ tests/      # 代码质量检查
uv run mypy src/ --ignore-missing-imports  # 类型检查
uv run pytest tests/unit/          # 单元测试
uv run pytest --cov=src --cov-report=term-missing  # 覆盖率报告

# 只有全部通过，才能commit
```

### 3.3.4 brainstorming：头脑风暴

**什么时候用？**

需求不明确或新功能设计时。

**四步法**：

```
Step 1: 探索用户意图 → 真正要解决什么问题？
Step 2: 收集约束条件 → 技术限制、时间限制、资源限制
Step 3: 评估方案选项 → 至少3个方案
Step 4: 确定最终方案 → 选择最优方案
```

**Nanobot Runner的实践**：

```markdown
# v0.21数字孪生引擎的brainstorming

## 问题定义
用户问："如果我按照这个训练计划跑4周，VDOT能提升多少？"

## 约束条件
- 只有2年历史数据
- VDOT计算基于Jack Daniels公式
- 需要支持What-If推演
- 性能要求：4周推演<10秒

## 方案选项

### 方案A: 纯规则模拟
- 优点: 简单、性能快
- 缺点: 精度低、无法处理复杂情况

### 方案B: 机器学习预测
- 优点: 精度高
- 缺点: 需要大量训练数据、解释性差

### 方案C: 混合方案（最终选择）
- 用Banister IR模型模拟训练响应
- 用ML模型校准参数
- 用规则处理边界情况
- 优点: 精度+解释性平衡

## 最终决策
选择方案C，设计状态向量+What-If推演引擎
```

## 3.5 异常处理机制

### 3.5.1 常见异常场景

| 异常场景 | 处理方式 | 时效要求 |
|----------|----------|----------|
| 产品规划方向不明确 | 重新定义产品愿景 | 1工作日内调整规划 |
| 需求理解存在歧义 | 逐个提问澄清，产出设计文档 | 1工作日内澄清 |
| 本地调试多次失败 | 输出调试报告，同步排查 | 持续排查 |
| Bug修复3次仍失败 | 停止修复，质疑架构设计 | 立即暂停 |
| CICD流水线连续失败 | 同步分析，FIX-CI | 2工时内修复 |
| 架构评审不通过 | 按评审报告整改后重新提交 | 1工作日内整改 |
| 代码评审不通过 | 按评审报告整改后重新提交 | 4工时内整改 |
| 测试通过率<95% | 优先修复失败用例，达标后进入发布 | 持续修复 |

### 3.5.2 关键原则

**Bug修复3次仍失败的应对**：

```
Phase 1: 验证问题确实存在
Phase 2: 重新分析根因（systematic-debugging）
Phase 3: 如果仍然失败，质疑架构设计
Phase 4: 如果架构有问题，考虑重构或回退
```

**CICD流水线失败的应对**：

```
配置型错误（YAML语法、路径错误等）:
  → 直接修复（FIX-CI Skill）

代码型错误（类型错误、测试失败等）:
  → systematic-debugging定位根因
  → 修复后重新提交
```

## 3.6 单人开发流程裁剪指南

### 3.6.1 裁剪原则

1. **方法论不可省** — TDD、systematic-debugging、verification-before-completion始终执行
2. **文档按需产出** — 根据变更影响面决定文档完整度
3. **角色合并执行** — 同一人连续执行多角色，减少形式化切换

### 3.6.2 按任务类型裁剪

| 任务类型 | 执行流程 | 文档产出 |
|----------|----------|----------|
| **小优化/Bug修复**（≤半天）| systematic-debugging → TDD → verification | 仅代码+测试，可选Bug修复报告 |
| **中等功能**（半天~2天）| brainstorming（轻量）→ TDD → verification → 代码评审 | 代码+测试+交付报告 |
| **大功能/架构变更**（>2天）| 完整流程：brainstorming → 需求分析 → 架构设计 → 任务拆解 → TDD → 代码评审 → 测试执行 | 全套文档 |
| **紧急Hotfix** | systematic-debugging（加速Phase 2-3）→ TDD → verification → 回归测试 | 修复代码+测试+回归报告 |

### 3.6.3 流程步骤合并建议

```
完整链路（12步）：
  产品规划 → 需求分析 → 架构设计 → 架构评审 → 任务拆解 → 
  writing-plans → 功能开发 → 代码评审 → 测试策略 → 测试执行 → 
  回归测试 → 发布

轻量链路（5步，适用大多数迭代）：
  需求确认 → [TDD] 功能开发 → [verification] 代码评审 → 
  测试执行 → 发布
```

**合并规则**：

- 产品规划 + 需求分析 → **需求确认**（一条记录即可）
- 架构设计 + 架构评审 → 仅重大变更时执行
- 任务拆解 + writing-plans → 仅复杂任务保留
- 测试策略 → 仅首次或测试框架变更时
- 文档归档 → 发布时顺带完成

## 3.8 经验教训

### 教训1：流程不是负担，是护栏

v0.20的教训告诉我：跳过测试阶段的"节省"时间，迟早要还。**宁可慢一点，也要稳一点**。

### 教训2：方法论要内化，不能形式化

TDD不是"先写测试"，而是"测试驱动设计"。如果TDD变成了形式，代码质量不会提升。**理解方法论的本质，而不是机械执行步骤**。

### 教训3：单人开发更需要流程

没有团队互相review，更容易陷入"自己骗自己"的陷阱。**流程是给自己的，不是给别人的**。

### 教训4：验证证据是最好的保护

"应该没问题"是最大的谎言。每次声称完成前，必须有**新鲜的、可复现的验证证据**。截图、日志、测试报告，都是保护自己的方式。

### 教训5：裁剪要合理，不能任性

裁剪是为了**提高效率**，不是**偷懒**。小功能可以跳步骤，但TDD、verification不能跳。**核心方法论必须保留**。
