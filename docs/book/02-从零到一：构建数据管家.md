## 2.1 FIT文件解析：从运动手表到结构化数据

### 2.1.1 为什么选择FIT格式？

在开始写代码之前，我需要解决第一个问题：**数据从哪里来？**

现代运动手表（Garmin、COROS、Apple Watch等）都使用ANT+ FIT协议存储运动数据。FIT是一种二进制格式，具有以下优势：

- **体积小**：相比CSV，二进制格式节省80%+存储空间
- **字段丰富**：支持GPS、心率、功率、海拔等上百个字段
- **元数据完整**：包含设备信息、活动摘要、圈数记录等

我选择了`fitparse`库来解析FIT文件。这个库能自动识别FIT文件的结构，并将二进制数据转换为Python字典。

### 2.1.2 解析器设计

```python
# src/core/storage/parser.py
class FitParser:
    """FIT文件解析器"""
    
    def parse(self, filepath: Path) -> dict[str, Any]:
        """
        解析单个FIT文件
        
        Args:
            filepath: FIT文件路径
            
        Returns:
            dict: 包含元数据和记录的字典
        """
        with fitparse.FitFile(str(filepath), data_processor=fitparse.StandardUnitsDataProcessor()) as fit_file:
            # 获取活动元数据
            metadata = self._extract_metadata(fit_file)
            
            # 获取秒级记录
            records = self._extract_records(fit_file)
            
            return {
                "metadata": metadata,
                "records": records
            }
    
    def _extract_metadata(self, fit_file) -> dict[str, Any]:
        """提取活动元数据"""
        return {
            "timestamp": self._get_field(fit_file, "timestamp"),
            "total_distance": self._get_field(fit_file, "total_distance"),
            "total_timer_time": self._get_field(fit_file, "total_timer_time"),
            "total_calories": self._get_field(fit_file, "total_calories"),
            "avg_heart_rate": self._get_field(fit_file, "avg_heart_rate"),
            "max_heart_rate": self._get_field(fit_file, "max_heart_rate"),
            "serial_number": self._get_field(fit_file, "serial_number"),
        }
    
    def _extract_records(self, fit_file) -> list[dict[str, Any]]:
        """提取秒级记录"""
        records = []
        for record in fit_file.get_messages("record"):
            record_data = {}
            for field in record.fields:
                record_data[field.name] = field.value
            records.append(record_data)
        return records
```

**关键设计决策**：
- 使用`StandardUnitsDataProcessor`自动转换单位（英里→公里，华氏度→摄氏度）
- 元数据与记录分离：元数据用于session级别聚合，记录用于详细分析
- 支持字段容错：某个字段不存在时不报错，使用默认值

|------|-----|---------|

| 生态支持 | 通用 | Polars原生支持 |
| 写入性能 | 快 | 快（追加模式） |

## 2.2 Parquet存储：按年分片策略

### 2.2.1 为什么按年分片？

早期的数据存储方案我尝试过两种设计：
1. **单文件存储**：所有数据存在一个文件中
2. **按月分片**：每月一个文件

最终我选择了**按年分片**，理由如下：

- **管理简单**：每年最多12个文件，文件列表一目了然
- **查询高效**：大多数分析查询都有时间范围，直接定位年份文件
- **备份友好**：可以单独备份某一年的数据
- **性能均衡**：2000条/年 vs 20000条/年，单文件性能差异可接受

### 2.2.2 存储管理器实现

StorageManager负责所有Parquet文件操作：

```python
class StorageManager:
    """Parquet存储管理器"""
    
    def __init__(self, data_dir: Path | None = None) -> None:
        # 数据目录：~/.nanobot-runner/data/
        self.data_dir = data_dir or Path.home() / ".nanobot-runner" / "data"
        self.data_dir.mkdir(parents=True, exist_ok=True)
    
    def append_activities(
        self, 
        dataframe: pl.DataFrame, 
        primary_key: str = "activity_id"
    ) -> dict[str, Any]:
        """
        批量追加活动数据（按年份分片）
        
        自动按年份分片并增量写入，性能提升 50%+。
        """
        # 按年份分组
        dataframe = dataframe.with_columns(
            pl.col("timestamp").dt.year().alias("_year")
        )
        year_groups = dataframe.partition_by("_year", as_dict=True)
        
        total_added = 0
        total_updated = 0
        
        for year_key, year_df in year_groups.items():
            year_int = int(year_key[0]) if isinstance(year_key, tuple) else int(year_key)
            year_df = year_df.drop("_year")
            
            # 增量写入单个年份文件
            result = self.save_to_parquet_incremental(
                year_df, year_int, primary_key
            )
            total_added += result["records_added"]
            total_updated += result["records_updated"]
        
        return {
            "success": True,
            "total_added": total_added,
            "total_updated": total_updated,
        }
```

### 2.2.3 增量写入：SHA256去重

数据去重是存储模块的核心功能之一。我使用**SHA256哈希**作为主键：

```python
def save_to_parquet_incremental(
    self,
    dataframe: pl.DataFrame,
    year: int,
    primary_key: str = "activity_id",
) -> dict[str, Any]:
    """增量写入 Parquet 文件，使用主键去重"""
    
    filename = f"activities_{year}.parquet"
    filepath = self.data_dir / filename
    
    if filepath.exists():
        existing_df = pl.read_parquet(filepath)
        existing_ids = set(existing_df[primary_key].to_list())
        new_ids = set(dataframe[primary_key].to_list())
        
        # 统计新增和更新
        records_added = len(new_ids - existing_ids)
        records_updated = len(new_ids & existing_ids)
        
        # 合并并去重
        combined_df = pl.concat([existing_df, dataframe])
        combined_df = combined_df.unique(
            subset=[primary_key], 
            keep="last", 
            maintain_order=True
        )
        
        combined_df.write_parquet(filepath, compression="snappy")
    else:
        dataframe.write_parquet(filepath, compression="snappy")
    
    return {
        "success": True,
        "records_added": records_added,
        "records_updated": records_updated,
    }
```

**性能优化技巧**：
- 使用`set`进行O(1)去重判断
- `keep="last"`确保最新数据优先
- `compression="snappy"`平衡压缩率和速度

## 2.3 Polars向量化计算：告别循环

### 2.3.1 为什么选择Polars？

在数据处理领域，Pandas是绝对的主流。但我选择了Polars，原因如下：

**1. 性能优势**

Polars使用Rust编写，在大多数场景下比Pandas快5-10倍：

```python
# Pandas: 逐行循环
for idx, row in df.iterrows():
    pace = row['duration'] / row['distance']
    df.at[idx, 'pace'] = pace

# Polars: 向量化操作
df = df.with_columns(
    (pl.col("duration") / pl.col("distance")).alias("pace")
)
```

**2. 延迟求值（LazyFrame）**

这是Polars最优雅的设计。查询不会立即执行，而是构建一个优化计划：

```python
# 延迟求值示例
lf = storage.read_parquet()  # 不加载数据
lf = lf.filter(pl.col("distance") > 1000)  # 不加载数据
lf = lf.with_columns(...)  # 不加载数据
df = lf.collect()  # 真正加载数据并执行优化后的计划
```

**3. 内存效率**

Polars使用Apache Arrow作为内存格式，支持零拷贝读取。

### 2.3.2 VDOT计算器：向量化实现

VDOTCalculator是第一个让我感受到Polars强大的模块：

```python
class VDOTCalculator:
    """VDOT跑力值计算器"""
    
    def calculate_vdot(self, distance_m: float, time_s: float) -> float:
        """单次计算VDOT"""
        if distance_m < 1500:  # Jack Daniels公式要求距离≥1500m
            return 0.0
        
        time_min = time_s / 60.0
        pace_min_per_km = time_min / (distance_m / 1000.0)
        
        vdot = 85.46 - 7.08 * pace_min_per_km + 0.15 * pace_min_per_km**2
        return round(max(0.0, vdot), 2)
    
    def calculate_vdot_batch(
        self, 
        df: pl.DataFrame,
        distance_col: str = "session_total_distance",
        duration_col: str = "session_total_timer_time",
    ) -> pl.Series:
        """
        批量计算VDOT（向量化版本）
        
        性能提升 30%+，代码更简洁。
        """
        result_df = df.with_columns(
            [
                pl.when(
                    (pl.col(distance_col) > 0)
                    & (pl.col(duration_col) > 0)
                    & (pl.col(distance_col) >= 1500)
                )
                .then(
                    (
                        85.46
                        - 7.08 * ((pl.col(duration_col) / 60.0) / (pl.col(distance_col) / 1000.0))
                        + 0.15 * ((pl.col(duration_col) / 60.0) / (pl.col(distance_col) / 1000.0)) ** 2
                    )
                    .clip(lower_bound=0.0)
                    .round(2)
                )
                .otherwise(0.0)
                .alias("vdot")
            ]
        )
        return result_df["vdot"]
```

**关键设计**：
- `pl.when().then().otherwise()`链式条件表达式
- `.clip(lower_bound=0.0)`防止负数
- `.round(2)`保留两位小数

### 2.3.3 SessionRepository：仓储层设计

SessionRepository封装了Session级别的聚合查询：

```python
class SessionRepository:
    """Session数据仓储，封装Session级别的数据聚合查询"""
    
    def _build_session_lazy(
        self,
        start_date: datetime | None = None,
        end_date: datetime | None = None,
        min_distance: float | None = None,
        descending: bool = True,
    ) -> pl.LazyFrame:
        """构建Session聚合查询的LazyFrame链"""
        
        lf = self.storage.read_parquet()
        
        # 按session_start_time聚合
        session_lf = lf.group_by("session_start_time").agg(
            [
                pl.col("session_start_time").first().alias("session_start"),
                pl.col("session_total_distance").first().alias("distance"),
                pl.col("session_total_timer_time").first().alias("duration"),
                pl.col("session_avg_heart_rate").first().alias("avg_hr"),
                # ...
            ]
        )
        
        # 时间范围过滤
        if start_date and end_date:
            session_lf = session_lf.filter(
                pl.col("session_start").is_between(start_date, end_date)
            )
        
        # 距离过滤
        if min_distance:
            session_lf = session_lf.filter(pl.col("distance") >= min_distance)
        
        # 排序
        session_lf = session_lf.sort("session_start", descending=descending)
        
        return session_lf
```

**仓储层的好处**：
- 隐藏Polars细节，提供业务友好的API
- 统一查询逻辑，消除重复代码
- 支持LazyFrame优化，所有过滤在最终输出前执行

## 2.4 依赖注入：AppContext架构

### 2.4.1 什么是依赖注入？

依赖注入（Dependency Injection）是一种设计模式，通过"注入"而不是"创建"来管理对象依赖。好处是：

- **松耦合**：组件不直接依赖具体实现
- **易测试**：可以注入Mock对象进行单元测试
- **可配置**：运行时决定使用哪个实现

### 2.4.2 AppContext实现

AppContext是这个项目依赖注入的核心：

```python
@dataclass
class AppContext:
    """
    应用上下文
    
    集中管理所有核心组件的实例，支持依赖注入和测试。
    """
    
    config: ConfigManager
    storage: StorageManager
    indexer: IndexManager
    parser: FitParser
    importer: ImportService
    analytics: AnalyticsEngine
    profile_engine: ProfileEngine
    session_repo: SessionRepository
    report_service: ReportService
    plan_manager: PlanManager
    
    # 扩展组件（延迟加载）
    _extensions: dict = field(default_factory=dict)
    
    def get_extension(self, name: str) -> Any | None:
        """获取扩展组件"""
        return self._extensions.get(name)
    
    def set_extension(self, name: str, instance: Any) -> None:
        """设置扩展组件"""
        self._extensions[name] = instance
```

**延迟加载模式**：

```python
@property
def prediction_engine(self) -> PredictionEngine:
```

## 2.7 经验教训

### 教训1：Schema设计要一步到位

早期版本的数据Schema比较混乱，后来花了很大力气才统一。**建议**：在开始存储数据之前，先花时间设计好Schema，包括字段名、类型、默认值。

### 教训2：LazyFrame要尽早用

有段时间我直接在DataFrame上操作，每次查询都全量加载数据。后来改成LazyFrame后，性能提升明显。**建议**：所有查询都从`read_parquet()`开始返回LazyFrame，只在最终展示时才`collect()`。

### 教训3：CLI要早做用户体验

早期CLI输出很简陋，用户体验很差。**建议**：从第一天就用Rich美化输出，用Typer自动生成帮助文档。

### 教训4：单元测试不能省

基础设施模块出bug影响范围大。**建议**：每个核心模块都要有对应的单元测试，核心逻辑覆盖率要达到80%+。
