这是一个**非常可行且优雅**的设计方案。将 `~/.nanobot-runner` 作为 `nanobot-ai` 的 workspace（工作空间），并利用 `user.md` 作为用户画像和记忆的存储载体，能够最大程度地发挥 nanobot-ai 的上下文理解能力，同时保持架构的清晰度。
以下是详细的可行性分析与实施建议：
### 1. 设计可行性分析
#### ✅ 优势：分层解耦，职责明确
这种设计将“底层框架配置”与“业务数据”完美分离：
*   **`~/.nanobot` (框架层)**：存放 `nanobot-ai` 的通用配置（如 LLM API Key、模型选择、全局代理等）。这是框架的“全局环境”。
*   **`~/.nanobot-runner` (应用层/Workspace)**：存放业务数据（训练记录 Parquet、计划 JSON）和 Agent 上下文（`user.md`）。这是 Agent 的“工作台”。
#### ✅ 优势：利用 Agent 原生能力，降低开发成本
`nanobot-ai` 等 Agent 框架通常具备文件读取和 RAG（检索增强生成）能力。
*   **传统方案**：应用维护数据库 -> 封装 API Tool -> Agent 调用 Tool -> 解析返回结构化数据。链路长，易出错。
*   **您的方案**：应用维护 `user.md` -> Agent 直接阅读 workspace 下的文件。Agent 像“读自己的笔记”一样理解用户画像，更符合 LLM 的处理逻辑，且无需额外的序列化/反序列化开销。
#### ✅ 优势：解决“长短期记忆”结合问题
*   **`user.md` (长期记忆/画像)**：存储静态画像（VDOT、历史伤病、偏好）、用户目标。
*   **对话上下文 (短期记忆)**：存储在 Agent 的 Session 中。
两者结合，构成了完整的个性化交互基础。

### 2. 核心实施建议：`user.md` 的维护策略
虽然设计可行，但关键在于**谁来维护 `user.md`**。直接让 Agent 随意写入可能导致数据混乱，建议采用 **“应用计算 + Agent 补充”** 的策略。
#### 建议的文件结构 (`~/.nanobot-runner/user.md`)
```markdown
# 用户画像
> 📅 最后更新时间: 2026-03-18 14:30 (系统自动更新)
## 1. 核心体能数据
<!-- 此部分由 RunnerProfileEngine 计算后自动覆盖写入 -->
- **当前 VDOT**: 45.2 (趋势: 📈 上升)
- **体能评分**: 68/100
- **周跑量**: 35 km
## 2. 训练偏好与习惯
- **偏好时段**: 早晨 06:00
- **一致性评分**: 85/100
- **常用装备**: Nike Vaporfly Next%
## 3. 伤病与恢复记录
- **伤病风险等级**: 低
- **历史伤病**: 右脚踝曾扭伤 (2025年)
- **恢复能力**: 平均恢复时间 24 小时
## 4. Agent 观察笔记
<!-- 此部分允许 Agent 在交互中追加或修改，用于存储非结构化信息 -->
- **2026-03-17**: 用户提到最近工作压力大，建议降低本周训练强度。
- **2026-03-18**: 用户反馈新跑鞋鞋带系法不舒服，下次提醒注意。
```
#### 维护流程设计
1.  **应用主导更新**：
    *   `RunnerProfileEngine` 在每日晨报生成或新数据导入后，根据算法计算结果，**覆写** `user.md` 中的“核心体能数据”部分。
    *   这保证了数据的准确性和时效性（如 VDOT 变化）。
2.  **Agent 辅助更新**：
    *   在飞书交互过程中，Agent 发现了新信息（如用户说“我膝盖不舒服”），通过调用 `update_memory` 工具，在“Agent 观察笔记”中追加条目。
    *   这实现了非结构化信息的记忆。

#### 3.1 修正 FR-001：画像持久化逻辑
```python
# core/profile.py
class RunnerProfileEngine:
    def __init__(self, storage: StorageManager):
        self.workspace_path = storage.config.base_dir # ~/.nanobot-runner
        self.profile_file = self.workspace_path / "user.md"
    def update_profile(self, activity_data: Dict) -> None:
        """计算画像并更新 user.md"""
        # 1. 计算最新数据
        new_stats = self._calculate_stats(activity_data)
        
        # 2. 读取现有的 user.md 内容
        existing_content = self._read_user_md()
        
        # 3. 智能合并（保留 Agent 手写的笔记，更新计算数据）
        updated_content = self._merge_profile_to_md(existing_content, new_stats)
        
        # 4. 写回文件
        self.profile_file.write_text(updated_content, encoding='utf-8')
        
    def get_profile_context(self) -> str:
        """提供给 Agent 的上下文接口"""
        return self.profile_file.read_text(encoding='utf-8')
```

### 4. 潜在风险与应对
| 风险点 | 描述 | 应对策略 |
| :--- | :--- | :--- |
| **文件过大** | 随着时间推移，`user.md` 可能包含过多历史记录，超出上下文窗口。 | 1. 应用在写入时进行“摘要压缩”<br>2. 仅保留最近 N 条 Agent 笔记<br>3. 旧数据归档到 `history.md` (nanobot 可索引但不常读) |
| **并发冲突** | Agent 正在写笔记时，应用正在更新统计数据。 | 使用文件锁（`portalocker`）或队列机制，确保写操作串行。 |
| **格式破坏** | Agent 误删了统计数据块。 | 将统计数据放在 Markdown 的 HTML 注释块 `<!-- ... -->` 中，或使用 JSON 块格式，并在 System Prompt 中明确告知 Agent “禁止修改特定区域”。 |
### 5. 总结
**此设计完全可行，且强烈推荐。**
它将 `nanobot-ai` 作为一个真正的“知识工作者”来看待：给它一个办公桌，给它一份用户档案，它就能基于这些上下文开展智能工作。这比单纯把 Agent 当作“API 调用器”要高明得多，也更符合 Agent 时代的软件工程范式。
**建议落地的关键点**：
在需求文档中明确 `user.md` 的 Markdown 模板规范，并界定“应用计算数据”与“Agent 记忆数据”的边界。
