## 二、阻塞项（必须修正）

### 🔴 C-01: SessionRepository 方法名错误 — 影响 §5.2 / §7.1 / §4.5

**问题**: 架构文档多处引用 `session_repo.query_by_date_range(start, end)`，但 [SessionRepository](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/src/core/storage/session_repository.py#L319) 中**不存在此方法**。实际方法为：

```python
# 实际方法签名
def get_sessions_by_date_range(
    self, start_date: datetime, end_date: datetime
) -> list[SessionSummary]:
```

**影响范围**:
- §5.2 数据导出序列图：`Engine->>Storage: query_by_date_range(start, end)` — 方法名错误
- §7.1 计算字段分批导出代码：`self.session_repo.query_by_date_range(...)` — 方法名错误
- §4.5 ExportEngine 伪代码：同上

**额外问题**: `get_sessions_by_date_range` 返回 `list[SessionSummary]`（dataclass），而非 `list[dict]`。`SessionSummary` 的字段名与 §7.1 代码中使用的 `session_total_distance`、`session_avg_heart_rate` 等原始 Parquet 列名**不一致**。需要确认 `SessionSummary` 是否包含这些字段，或改用 `StorageManager.query_by_date_range()` 直接获取原始 dict。

**修正建议**:
```diff
# §5.2 序列图
- Engine->>Storage: query_by_date_range(start, end)
- Storage-->>Engine: List[Dict] 原始记录
+ Engine->>SessionRepo: get_sessions_by_date_range(start, end)
+ SessionRepo-->>Engine: List[SessionSummary]

# §7.1 代码
- records = self.session_repo.query_by_date_range(
-     start_date=config.start_date,
-     end_date=config.end_date,
- )
+ records = self.session_repo.get_sessions_by_date_range(
+     start_date=start_dt,
+     end_date=end_dt,
+ )
```

同时需明确：`SessionSummary` 是否包含 `avg_heart_rate` 字段用于 TSS 计算。若不包含，需改用 `StorageManager.query_by_date_range()` 获取原始记录。

### 🔴 C-02: 数据获取路径不一致 — §5.2 序列图 vs §7.1 代码

**问题**: 同一数据流在两处描述不一致：

| 位置 | 数据源 | 调用方式 |
|------|--------|---------|
| §5.2 序列图 | `StorageManager` | `Engine->>Storage: query_by_date_range()` |
| §7.1 代码 | `SessionRepository` | `self.session_repo.query_by_date_range()` |

这两个是不同的对象，返回类型也不同：
- `StorageManager.query_by_date_range()` → `list[dict[str, Any]]`（原始 Parquet 行）
- `SessionRepository.get_sessions_by_date_range()` → `list[SessionSummary]`（聚合 dataclass）

**修正建议**: 统一为一种路径。推荐使用 `StorageManager.query_by_date_range()` 获取原始 dict（因为 §7.1 代码需要 `session_total_distance`、`session_avg_heart_rate` 等原始字段名），或扩展 `SessionSummary` 使其包含心率字段。

### 🔴 C-03: ChartRenderer Protocol 缺少降级行为定义

**问题**: §2.3 明确指出 plotext 对堆叠柱状图"⚠️ 需验证"、热力图"❌ 需备选"，但 §4.1 的 `ChartRenderer` Protocol 仍要求实现 `render_stacked_bar_chart()`。如果 `PlotextRenderer` 无法实现该方法，应该：

1. 抛出 `NotImplementedError`？
2. 返回降级文本？
3. 在 Protocol 中标记为可选？

当前架构未定义此行为，开发时会产生歧义。

**修正建议**: 在 §4.1 ChartRenderer Protocol 中增加注释：
```python
def render_stacked_bar_chart(
    self, data: ChartData, config: ChartConfig
) -> str:
    """渲染堆叠柱状图

    注意: plotext 对堆叠柱状图支持有限。
    若渲染后端不支持，应降级为 Rich Table 文本展示，
    不应抛出异常阻塞调用方。
    """
    ...
```

并在 §7.2 降级策略中补充堆叠柱状图的 fallback 示例。

def _generate_report_with_charts(
    self,
    report_data: WeeklyReportData | MonthlyReportData,
    report_type: ReportType,
    console: Console,
) -> None:
    """生成报告（含图表增强）

## 四、亮点

1. **Protocol 抽象设计优秀**: `ChartRenderer` 和 `DataExporter` 两个 Protocol 实现了渲染后端和导出格式的可替换性，符合开闭原则
2. **降级容错策略完善**: §7.2 的 `_render_with_fallback` 和 §8.2 的 try/except 降级确保了图表渲染失败不阻塞核心功能
3. **终端自适应**: §7.3 的 `get_chart_width()` 考虑了 Windows 终端兼容性
4. **路径安全校验**: §7.4 的 `validate_export_path` 防止了路径穿越攻击
5. **数据源映射表**: §8.1 清晰列出了每个需求对应的具体方法和模块，可追溯性强
6. **AppContext 扩展方式一致**: 使用 `@property` + `_extensions` 懒加载模式，与现有 `training_reminder_manager` 等组件完全一致

## 五、风险补充

架构文档 §10 已识别 5 项风险，建议补充以下 2 项：

| 风险 | 等级 | 触发条件 | 缓解措施 |
|------|------|----------|---------|
| SessionSummary 缺少心率字段 | 中 | `get_sessions_by_date_range` 返回的 dataclass 不含 `avg_heart_rate` | 验证 SessionSummary 字段；若不包含则改用 StorageManager 原始查询 |
| plotext 与 Rich Console 输出冲突 | 低 | plotext 使用 `print()` 而 Rich 使用 `console.print()`，输出可能乱序 | plotext 的 `plt.build()` 返回字符串，统一通过 Rich Console 输出 |
