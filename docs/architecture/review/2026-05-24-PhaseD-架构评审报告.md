## 二、底座升级设计文档评审

### 2.1 API 兼容性分析（正面结论）

通过对比 `nanobot/agent/hook.py` 与 RunFlowAgent 的 `DecisionLogHook`，核心 API 兼容性评估如下：

| API | 0.1.5.post2 | 0.2.0 | Breaking? | RunFlowAgent 影响 |
|-----|-----------|-------|-----------|-------------------|
| `AgentHook.wants_streaming()` | 有 | 有 | 无 | 无影响 |
| `AgentHook.before_iteration()` | 有 | 有 | 无 | 无影响 |
| `AgentHook.on_stream()` | 有 | 有 | 无 | 无影响 |
| `AgentHook.on_stream_end()` | 有 | 有 | 无 | 无影响 |
| `AgentHook.before_execute_tools()` | 有 | 有 | 无 | 无影响 |
| `AgentHook.after_iteration()` | 有 | 有 | 无 | 无影响 |
| `AgentHook.finalize_content()` | 有 | 有 | 无 | 无影响 |
| `AgentHook.emit_reasoning()` | **无** | **有** | **无（新增）** | 自动继承，无适配成本 |
| `AgentHook.emit_reasoning_end()` | **无** | **有** | **无（新增）** | 自动继承，无适配成本 |
| `AgentHookContext.streamed_reasoning` | 无 | **有** | 无（新增字段） | 向后兼容 |
| `ToolCallRequest` 数据结构 | 有 | 有 | 无 | DecisionLogHook 兼容 |
| `LLMResponse` 数据结构 | 有 | 有 | 无 | 核心接口不变 |

**结论**：nanobot-ai 0.2.0 对 AgentHook 接口**纯增量扩展，无破坏性变更**。DecisionLogHook 作为 AgentHook 的直接子类，可无缝兼容。风险从"中"下调为"低"。

### 2.2 🔴 需修正：GoalState 实现方式描述不准确

**问题** (§4.1)：文档写"在 Agent 工具中集成 `long_task` 工具"，这与实际机制存在偏差。

**修正**：

> **实现方式**：
> - 在 Agent system prompt（SOUL.md）中注入 GoalState 使用指导，引导 LLM 在长时间任务时使用 `long_task` Skill 持久化目标
> - 在 `DecisionLogHook` 的 `after_iteration` 中通过 `goal_state_raw(context.metadata)` 读取当前活跃目标，关联到 DecisionLog
> - 进化引擎的 `EvolutionController` 在月度复盘触发时，检查当前 GoalState 是否活跃并纳入进化上下文

### 2.3 🔴 需修正：推理可见化的实现方式过于笼统

**问题** (§4.2)：缺少 DecisionLogHook 如何重写 emit_reasoning 的具体说明。

**修正**：

> **实现方式**：
> - 在 `DecisionLogHook` 中重写 `emit_reasoning()` 方法，将推理片段追加到内部缓冲区
> - 在 `finalize_content()` 中，将推理缓冲区内容写入 DecisionLog 的上下文快照
> - 飞书/WebSocket 渠道的推理流式展示由 nanobot-ai 原生 Channel 层自动处理，RunFlowAgent 仅需确保 `show_reasoning: true`（已有默认值）

### 2.4 🔴 需修正：Model Presets 的 CLI 命令设计有歧义

**问题** (§4.3)：`/model <preset>` 不是 RunFlowAgent 需要实现的，是 nanobot-ai 内置的 chat command。

**修正**：验收标准第3条改为：

> 3. 飞书/WebUI 中可通过 `/model <preset>` 切换预设（nanobot-ai 内置命令，RunFlowAgent 仅负责提供预设配置）

### 2.5 🟡 建议补充：v0.26.0 缺少具体的风险评估清单

建议补充以下风险项：

| 风险项 | 说明 | 排查方法 |
|--------|------|----------|
| `pyproject.toml` 间接依赖冲突 | nanobot-ai 0.2.0 可能引入冲突的依赖版本 | `uv pip install --dry-run` 预检 |
| config.json Schema 兼容 | 0.2.0 新增字段是否被现有 config.json 接受 | 用 0.2.0 的 Config loader 测试加载现有 config.json |
| ToolRegistry 注册机制变更 | 工具注册 API 是否有变更 | 检查 `tools.py` 和 `tools_evolution.py` |

### 2.6 🟡 建议补充：缺少性能基线对比要求

建议 v0.26.0 验收标准新增：

> 5. 核心 CLI 命令响应时间不退化 >20%（以 v0.25.0 为基线）

## 三、产品规划方案评审

### 3.1 版本节奏：建议压缩为4版本

| 当前版本 | 可压缩方案 | 理由 |
|----------|-----------|------|
| v0.26 + v0.27 | 合并为 **v0.26.0 底座升级+新特性** | API 基本无破坏性变更，三项新特性（GoalState/推理可见化/Model Presets）均不涉及 UI，可一个版本闭环 |
| v0.28 + v0.29 | 保持分离 | v0.28 WebUI 基础是"0→1"，v0.29 数据可视化是"1→N"，分离合理 |
| v0.31 | **降级为评估任务，不占版本号** | FallbackProvider 和 MCP Resources/Prompts 本质是评估，可能结论是"暂不采纳" |

**推荐方案**：压缩为 **4 个版本**（v0.26 → v0.27 → v0.28 → v0.29），v0.31 的评估工作并入 v0.29。

### 3.2 🟡 WebUI 品牌自定义路径不够具体

建议明确 config.json 配置示例和 SOUL.md/AGENTS.md 模板适配方案。

### 3.3 🟡 WebUI 数据可视化（v0.29.0）技术方案待细化

需明确三点：
1. API 端点设计：在 nanobot-ai aiohttp server 注册新路由 vs 独立 FastAPI 服务？
2. 前端组件扩展：fork WebUI 源码 vs iframe/widget 嵌入独立前端？
3. 图表库选型：Recharts vs ECharts 的体积和功能对比？
