## 2. 评审维度详细分析

### 2.1 技术选型评审

| 评审项 | 状态 | 分析 |
|--------|------|------|
| 无新增第三方依赖 | ✅ | v0.19.0 所有计算基于 Polars/numpy 现有能力，无需引入新库 |
| 与现有技术栈兼容 | ✅ | Python 3.11+ / Polars 0.20+ / nanobot-ai Latest，完全一致 |
| 计算方案可行性 | ✅ | 静息心率趋势、心率恢复率、疲劳度加权评分均可基于现有数据实现 |
| HRV估算方案可行性 | ⚠️ | RMSSD/SDNN 需 RR 间期数据，基于心率数据估算精度有限（见风险章节） |

**评审意见**: 技术选型策略正确——"零新依赖"是个人项目的最佳实践，降低维护负担。HRV 估算标注"非医疗级精度"是负责任的披露。

---

### 2.2 架构模式评审

| 评审项 | 状态 | 分析 |
|--------|------|------|
| 模块划分合理性 | ✅ | 5个子模块职责清晰：HRVAnalyzer(分析)、FatigueAssessor(评估)、RecoveryMonitor(监控)、BodySignalEngine(编排)、ComparisonAnalyzer(对比) |
| 分层架构一致性 | ✅ | CLI → Handler → Core(body_signal) → Storage，与现有分层完全一致 |
| 依赖注入模式 | ✅ | AppContext 懒加载属性模式，与 `plan_execution_repo` 等现有扩展一致 |
| 复用现有组件 | ✅ | HeartRateAnalyzer、TrainingLoadAnalyzer、SmartAdviceEngine、InjuryRiskAnalyzer 均被正确引用 |
| 目录结构规范 | ✅ | `src/core/body_signal/` 遵循现有子模块命名和结构约定 |

**评审意见**: 架构模式与现有代码库高度一致，开发团队无需学习新模式。BodySignalEngine 作为编排层的设计合理，避免了 CLI/Agent 直接调用多个分析器的耦合。

## 3. 问题清单

### 3.1 必改项（进入开发前必须解决）

| ID | 问题 | 严重度 | 建议方案 |
|----|------|--------|----------|
| **Q1** | **数据缺失降级策略未定义**：当用户无近期训练数据、无心率数据、或仅有个位数记录时，各分析器应返回什么？ | 🔴 高 | 为每个分析方法定义 `empty_state` 返回值。例如：`HRVAnalysisResult` 增加 `data_quality: DataQuality` 字段（SUFFICIENT/INSUFFICIENT/EMPTY），CLI 层根据 quality 展示不同文案 |
| **Q2** | **边界条件处理未定义**：静息心率趋势仅1个数据点时、疲劳度各维度权重之和≠100%时、TSB 极端值（如±50）时如何处理？ | 🔴 高 | 补充边界条件矩阵：①单点数据→趋势显示"N/A"而非折线 ②权重校验→初始化时 assert sum=100% ③TSB 截断→限制在[-50, 50]区间避免评分失真 |
| **Q3** | **配置 Schema 未定义**：架构文档称"权重可配置"，但未定义配置项的数据结构、默认值位置、环境变量覆盖规则 | 🔴 高 | 在 `src/core/config/schema.py` 中新增 `BodySignalConfig`：包含 `fatigue_weight_atl`、`fatigue_weight_hr`、`fatigue_weight_consecutive`、`fatigue_weight_subjective`、`hr_spike_threshold_pct`、`overtraining_tsb_threshold` 等字段，遵循现有 Pydantic-Settings 模式 |
| **Q4** | **RPE 数据输入路径缺失**：疲劳度模型依赖主观疲劳度(RPE)，但未定义用户如何提供该数据 | 🟡 中 | 方案A（推荐）：从训练记录的 `perceived_effort` 字段读取（若 FIT 文件包含）。方案B：CLI 命令 `analysis fatigue --rpe 6` 支持手动输入。方案C：RPE 缺失时该维度权重自动分配给其他三维度 |
| **Q5** | **测试策略未定义**：架构文档未包含 body_signal 模块的测试策略 | 🟡 中 | 补充测试策略：①单元测试覆盖每个分析器的核心计算逻辑（mock SessionRepository）②集成测试覆盖 BodySignalEngine 编排流程③边界测试覆盖空数据/单点数据/极端值场景④性能测试验证 <2s 目标 |
| **Q6** | **`status` 命令组与 `analysis` 命令组职责边界模糊**：`status today` 与 `analysis fatigue` 功能重叠，用户可能困惑 | 🟡 中 | 明确职责划分：`status` = 快速摘要（一句话+三色灯，<500ms），`analysis` = 详细分析（完整数据+图表+趋势，<2s）。在 CLI help 中注明"快速查看用 status，深入分析用 analysis" |

### 3.2 建议改进项（可在开发中逐步完善）

| ID | 建议 | 优先级 |
|----|------|--------|
| **S1** | HRV 分析结果增加 `data_source` 字段，标注"RR间期直接计算"或"心率数据估算"，让用户了解数据可信度 | 低 |
| **S2** | BodySignalEngine 增加缓存机制：同一自然日内多次查询复用计算结果，避免重复计算 | 低 |
| **S3** | `status weekly` 输出增加与上周的对比（"静息心率较上周↓2bpm，恢复趋势向好"） | 低 |
| **S4** | 考虑将 `RecoveryStatus` 枚举从 `body_signal/models.py` 提升到 `core/models/` 共用，因为 `InjuryRiskAnalyzer` 也可能需要引用 | 低 |

## 4. 风险清单

| ID | 风险 | 等级 | 根因 | 缓解措施 | 残留风险 |
|----|------|------|------|----------|----------|
| **R1** | HRV 指标(RMSSD/SDNN)计算不准确 | 🔴 高 | 基于心率数据估算而非 RR 间期直接计算，物理精度上限低 | ①明确标注"非医疗级精度，仅供参考" ②优先展示静息心率趋势和恢复率（这两个指标更可靠） ③若 FIT 数据无 RR 间期，RMSSD/SDNN 返回 None | 用户可能忽略精度提示，过度解读 HRV 数值 |
| **R2** | 疲劳度模型不适用于所有跑者 | 🟡 中 | 固定权重模型无法适配个体差异（如有人对心率偏差更敏感） | ①权重可配置 ②提供"校准模式"：用户反馈"我感觉比评分更累/更轻松"，系统据此调整权重 | 校准需要用户持续反馈，初期准确性有限 |
| **R3** | FIT 数据中 RR 间期缺失 | 🟡 中 | 部分设备（如早期 Garmin、入门级心率带）不记录 RR 间期 | ①检测 RR 间期数据是否存在 ②不存在时 HRV 指标返回 None ③降级提示"当前设备不支持 HRV 精确计算" | 部分用户无法使用 HRV 功能 |
| **R4** | 用户过度依赖量化指标 | 🟡 中 | 量化评分给人以"客观准确"的错觉 | ①UI 中强调"倾听身体" ②异常预警附带"如有不适请咨询医生"免责声明 | 用户行为改变需要时间 |
| **R5** | 需求依赖链断裂风险 | 🟡 中 | REQ-0.19-03 依赖 REQ-0.19-01 和 REQ-0.19-02，若前序需求延期则后续阻塞 | ①BodySignalEngine 设计为松耦合：即使子分析器返回空结果，引擎仍可输出降级摘要 ②开发顺序：HRV→疲劳度→引擎，但引擎骨架可并行搭建 | 若 HRV 分析器延期，引擎的预警功能受限 |
