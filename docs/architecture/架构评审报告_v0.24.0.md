### 3.2 MEDIUM级别问题

#### MEDIUM-1: CalibrationEngine的bias+scale双重修正可能过度修正

**问题描述**: 架构设计Section 8.3.5.2校准流程同时计算new_scale和new_bias:
- `new_scale = 1.0 / (1.0 + mean_error)` 修正比例偏差
- `new_bias = -mean(actual - predicted) / n` 修正加法偏差

两者从同一组数据计算，本质上修正的是同一系统性偏差的不同表达形式。同时应用可能导致过度修正。例如: VDOT预测值46.5，实际值45.0，mean_error=3.3%:
- scale修正: 46.5 * (1/1.033) = 45.02 (已接近实际值)
- 再加bias修正: 45.02 + (-1.5) = 43.52 (过度修正)

**影响**: 校准后预测误差可能反而增大，违背校准目标。

**建议整改**:
1. 在ADR-008中明确bias和scale的修正分工:
   - scale: 修正比例性偏差(预测值与实际值的比值偏离1.0)
   - bias: 修正加法性偏差(预测值与实际值的绝对差)
2. 增加互斥修正策略: 当scale偏离1.0超过2%时，仅使用scale修正；否则使用bias修正
3. 或简化为仅使用scale修正(更简单，且幅度限制已提供安全网):
   - `corrected = raw * scale` (移除bias项)
   - 理由: VDOT值范围40-60，比例修正已足够；bias对小数值的修正效果有限

**整改优先级**: MEDIUM

#### MEDIUM-4: CalibrationEngine new_scale公式在极端mean_error下不稳健

**问题描述**: 架构设计Section 8.3.5.2校准流程中:
```
new_scale = 1.0 / (1.0 + mean_error)
```
当mean_error < -1.0(即模型持续低估超过100%)时，new_scale变为负值，导致校准后预测值符号反转。虽然幅度限制(+-10%)可以兜底，但公式本身缺乏数学稳健性。

**影响**: 在极端数据下产生无意义的中间计算结果，增加调试难度。

**建议整改**:
1. 增加mean_error范围检查:
   ```python
   mean_error = clamp(mean_error, -0.5, 0.5)  # 限制在合理范围
   new_scale = 1.0 / (1.0 + mean_error)
   ```
2. 或改用更稳健的公式:
   ```python
   new_scale = mean(actual) / mean(predicted)  # 直接比例修正
   ```
3. 在_detect_bias()中增加异常值检测，排除极端配对数据

**整改优先级**: MEDIUM

## 5. 风险评估

### 5.1 架构设计已识别风险复查

| 风险(Section 8.3.11) | 等级 | 评审意见 |
|----------------------|------|----------|
| 校准偏差方向错误 | 中 | 缓解措施充分(10条门槛+EMA+幅度限制+reset)，认可 |
| 训练响应性样本不足 | 低 | 缓解措施充分(每组>=5+data_sufficient标识)，认可 |
| 强度偏差计算不准 | 中 | 缓解措施部分(向后兼容)，但来源未明确(见MEDIUM-2) |
| 校准与PredictionEngine耦合 | 低 | 无侵入设计合理，认可 |
| Banister IR参数调整过度 | 中 | 缓解措施部分(5%限制+MAE验证)，但持久化缺失(见HIGH-3) |
| 用户反馈稀疏 | 高 | 缓解措施合理(隐式反馈+AskUserConfirmManager)，认可 |

### 5.2 评审新增风险

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| VDOT变化量数据稀疏 | 高 | ResponseAnalyzer可用配对极少，分析结果不可靠 | 利用runner_state.vdot(自动采集)替代actual_vdot(需手动触发)；降低fidelity门槛至0.5增加配对数 |
| prediction_snapshot结构不统一 | 高 | CalibrationEngine无法可靠提取预测值 | 标准化prediction_snapshot Schema(见HIGH-2整改) |
| 校准与进化循环依赖 | 中 | CalibrationEngine依赖EvolutionStore，ModelEvolver依赖CalibrationEngine | 依赖链单向: Store → CE → ME，无循环，认可 |
| 校准触发频率过低 | 中 | 用户不主动执行evolution calibration，校准数据持续过时 | v0.25增加自动触发；v0.24在evolution status中提示校准建议 |

## 8. 评审结论

### 8.1 总体评价

v0.24.0个性化学习模块的架构设计整体质量良好:
- 需求覆盖完整，P0/P1全部AC均有对应设计

主要不足集中在**数据流实现细节**和**参数持久化机制**两个维度，均为可修复的设计补充项，非根本性设计缺陷。
