## 1. 评审概述

### 1.1 评审对象

v0.25.0 在 `src/core/evolution/` 模块递增式添加 3 个子组件：

| 子组件 | 职责 | 对应需求 |
|--------|------|----------|
| EvolutionController | 进化触发器（4种规则 + 异步执行） | REQ-0.25-01 (P0) |

新增 3 个 CLI 命令（`evolution triggers/report/tune`）、3 个 Agent 工具、4 个数据模型（EvolutionAction/TriggerCheckResult/PromptTuningParams/EvolutionReport）、2 个 ADR（ADR-009/010）。

### 1.2 评审方法

基于架构设计说明书 v12.0.1 与需求规格说明书 v10.0 的交叉验证，结合现有代码实现（v0.23 + v0.24 已开发完成），从技术可行性、架构一致性、安全风险、性能风险、可维护性、向后兼容性 6 个维度进行评审。

## 2. 评审发现

### 2.1 CRITICAL 级别

#### C-01: daemon 线程中模型进化操作的数据一致性风险

**维度**: 安全风险

**问题描述**:

EvolutionController 通过 `threading.Thread(daemon=True)` 异步执行进化动作。架构设计 Section 8.4.5.1 中 `execute_action()` 方法在 daemon 线程中执行以下非原子操作序列：

1. 调用 `ModelEvolver.evolve_model()` -> 修改 BanisterIRModel 实例属性（即时生效）
2. 调用 `EvolutionStore.save_model_params()` -> 持久化参数到 JSON（原子写入）

若步骤 1 完成后、步骤 2 执行前主进程退出，daemon 线程被强制终止，将导致：
- 内存中的模型参数已修改（即时生效），但持久化文件未更新
- 下次启动时从 JSON 加载旧参数，内存状态与持久化状态不一致
- 用户看到的进化效果丢失，且无任何错误提示

**影响**: 进化结果静默丢失，校准-进化闭环断裂，违反 NFR-09（模型回滚安全）

**整改要求**:

方案 A（推荐）：调整执行顺序为「先持久化，后生效」：
1. `execute_action()` 先调用 `EvolutionStore.save_model_params()` 持久化参数
2. 持久化成功后，再修改 BanisterIRModel 实例属性
3. 若持久化失败，不修改实例属性，进化动作标记为 failed

方案 B：为 daemon 线程添加优雅退出机制：
1. 使用 `threading.Event` 作为退出信号
2. 主进程退出前发送信号，等待 daemon 线程完成当前操作（超时 5 秒）
3. 在 `atexit` 中注册清理函数

**验收标准**: 进化动作执行过程中主进程退出，持久化文件与内存状态一致

#### C-02: check_triggers() 性能可能违反 NFR-05

**维度**: 性能风险

**问题描述**:

架构设计声称 `check_triggers()` 仅做条件判断（轻量，延迟 <10ms），但实际实现需要：

1. `_check_vdot_error_trigger()` -> 调用 `EvolutionStore.get_prediction_actual_pairs("vdot", min_count=3)` -> 内部调用 `get_decision_outcome_pairs()` -> 扫描所有 decisions/ 和 outcomes/ Parquet 文件
2. `_check_rejection_trigger()` -> 调用 `EvolutionStore.get_decision_outcome_pairs()` -> 同上
3. `_check_new_data_trigger()` -> 调用 `EvolutionStore.query_decisions(limit=10000)` -> 扫描所有 decisions/ Parquet 文件

`get_decision_outcome_pairs()` 当前实现（[evolution_store.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.25.0/src/core/evolution/evolution_store.py#L591-L646)）会 collect 全部 outcomes 后再与 decisions join，数据量增长后延迟可能远超 10ms。

`check_triggers()` 在 `DecisionLogHook.after_iteration()` 中同步调用，若延迟超过 100ms，将违反 NFR-05（Hook 接入延迟 <100ms），直接影响 Agent 交互体验。

**影响**: Agent 交互延迟增加，用户体验退化

**整改要求**:

1. 为 `check_triggers()` 添加性能预算：总延迟 <50ms（留 50ms 余量给其他 Hook 逻辑）
2. 优化 `get_decision_outcome_pairs()` 查询：
   - 添加 `days` 参数限制扫描范围（如仅扫描最近 90 天）
   - 使用 LazyFrame join 替代 Python 层关联
3. 为 `_check_new_data_trigger()` 使用 `trigger_state.json` 缓存上次检查的记录数，避免每次全量扫描
4. 添加性能监控日志：`check_triggers()` 执行时间超过 50ms 时输出 warning

**验收标准**: check_triggers() 在 1000 条决策数据下延迟 <50ms

### 2.2 HIGH 级别

#### H-01: EvolutionController 绕过 EvolutionEngine 直接调用子组件

**维度**: 架构一致性

**问题描述**:

EvolutionController 的 `execute_action()` 方法直接调用 `self._model_evolver.evolve_model()`、`self._prompt_tuner.auto_adjust_on_rejection()`、`self._evolution_reporter.generate_report()`，绕过了 EvolutionEngine 的编排层。

而 EvolutionEngine 又提供了 `check_evolution_triggers()` 委托给 EvolutionController，形成不一致的调用路径：

```
CLI/Agent -> EvolutionEngine.check_evolution_triggers() -> EvolutionController.check_triggers()
CLI/Agent -> EvolutionEngine.execute_evolution_action() -> EvolutionController.execute_action()
DecisionLogHook -> EvolutionController.check_triggers() -> EvolutionController.execute_pending_actions()
```

DecisionLogHook 直接持有 EvolutionController 引用，跳过了 EvolutionEngine 编排层，违反了「薄编排层」设计原则。

**影响**: EvolutionEngine 无法感知进化动作的执行，`get_evolution_status()` 中进化动作统计可能不准确

**整改要求**:

1. DecisionLogHook 应持有 EvolutionEngine 引用（而非 EvolutionController），通过 `EvolutionEngine.check_evolution_triggers()` 间接调用
2. EvolutionController.execute_action() 的执行结果应回调通知 EvolutionEngine，更新进化状态
3. 或者：EvolutionController 作为 EvolutionEngine 的内部实现细节，不直接暴露给外部

**验收标准**: 所有进化相关操作通过 EvolutionEngine 编排层进行，EvolutionEngine 可感知全部进化动作

#### H-02: incremental_learn 动作缺乏部分失败处理

**维度**: 安全风险

**问题描述**:

EvolutionController.execute_action() 中 `incremental_learn` 动作遍历 3 种模型类型执行进化：

```python
for model_type in ["vdot", "injury", "training_response"]:
    try:
        r = self._model_evolver.evolve_model(model_type)
        results.append(f"{model_type}: MAE {r.mae_before:.4f}->{r.mae_after:.4f}")
    except ValueError:
        results.append(f"{model_type}: 数据不足跳过")
```

问题：
1. vdot 进化成功后，injury 进化失败（非 ValueError，如 RuntimeError），vdot 的进化结果已生效但 injury 未进化，系统处于不一致状态
2. 进化结果仅记录在 `execution_result` 字符串中，无结构化数据，无法追溯哪些模型进化成功
3. EvolutionAction 是 frozen dataclass，执行结果通过创建新实例记录，但原始 action 的 `executed=True` 不区分部分成功

**影响**: 增量学习部分失败时系统状态不一致，且难以追溯

**整改要求**:

1. 为 incremental_learn 动作添加逐模型结果记录：
   ```python
   @dataclass(frozen=True)
   class IncrementalLearnResult:
       model_type: str
       success: bool
       mae_before: float | None
       mae_after: float | None
       error: str | None
   ```
2. EvolutionAction.execution_result 改为结构化数据（或新增 execution_details 字段）
3. 部分失败时 action 标记为 `executed=True` 但 `execution_result` 包含详细的部分成功/失败信息

**验收标准**: incremental_learn 动作部分失败时，可追溯每个模型的进化结果

---

#### H-03: PromptTuner.auto_adjust_on_rejection() 未考虑参数下限保护

**维度**: 安全风险

**问题描述**:

`auto_adjust_on_rejection()` 在连续拒绝时降低 aggressive 和 data_driven：

```python
updated = current.with_updates(
    aggressive=current.recommendation_aggressiveness - self._config.tuning_adjustment_step,
    data_driven=current.data_driven_weight - self._config.tuning_adjustment_step * 0.5,
)
```

虽然 `with_updates()` 会 clamp 到 [0.0, 1.0]，但存在以下问题：

1. 连续多次拒绝（如 5 次）后，aggressive 可能从 0.5 降到 0.25，再降到 0.20，最终趋近 0.0。当 aggressive=0.0 时，推荐将极度保守，可能永远不再触发激进推荐，形成「保守陷阱」
2. data_driven 持续降低会导致系统越来越依赖经验驱动，与数据驱动进化的设计目标相悖
3. 没有反弹机制：一旦参数降到极低值，即使后续用户接受推荐，`auto_adjust_on_feedback()` 的步长仅 0.05，恢复缓慢

**影响**: 提示调优参数可能陷入极端值，难以自然恢复

**整改要求**:

1. 添加参数下限保护：aggressive 最低不低于 0.1，data_driven 最低不低于 0.2
2. 在 `auto_adjust_on_rejection()` 中添加日志：当参数接近下限时输出 warning
3. 考虑添加反弹机制：当用户接受推荐时，aggressive 的恢复步长应大于降低步长（如恢复 0.08 vs 降低 0.05）

**验收标准**: 连续 10 次拒绝后，aggressive 不低于 0.1，data_driven 不低于 0.2

### 2.3 MEDIUM 级别

#### M-01: EvolutionAction.trigger_condition 缺乏 Schema 约束

**维度**: 可维护性

**问题描述**: `trigger_condition: dict[str, Any]` 无结构化定义，4 种触发条件产生的数据结构各不相同，下游代码难以类型安全地访问。

**整改建议**: 为每种触发条件定义 frozen dataclass（如 VdotErrorCondition/RejectionCondition/NewDataCondition/MonthlyReviewCondition），trigger_condition 改为 `dict[str, Any]` 但在文档中明确每种触发条件的 Schema，或使用 Union 类型。

---

#### M-02: EvolutionReporter 个性化程度计算公式偏差

**维度**: 技术可行性

**问题描述**: `_get_personalization_degree()` 中校准偏离计算 `|1.0 - avg_scale|` 取各模型 scale 绝对偏差的平均值。当 vdot scale=0.97、injury scale=1.02 时，avg_scale=0.995，偏离仅 0.005，但实际两个模型都有明显校准。应使用 `avg(|1.0 - scale|)` 而非 `|1.0 - avg(scale)|`。

**整改建议**: 修正公式为 `calibration_degree = sum(|1.0 - scale|) / len(scales)`，当前代码逻辑实际已正确（先取绝对值再求和再除以数量），但架构设计文档描述有歧义，需统一。

---

#### M-03: trigger_state 持久化机制未完整定义

**维度**: 技术可行性

**问题描述**: `_check_new_data_trigger()` 中调用 `self._load_last_incremental_count()` 获取上次增量学习时的记录数，但该方法在架构设计中未定义实现。EvolutionStore 新增了 `save_trigger_state()/load_trigger_state()` 方法，但 EvolutionController 如何使用这些方法管理触发器状态未明确。

**整改状态**: ✅ 已整改 (v12.0.2) — _load_last_incremental_count()实现逻辑已补充，trigger_state.json Schema已明确({"last_incremental_count": int, "last_monthly_report": str})，incremental_learn分支添加save_trigger_state更新

**整改建议**: 在架构设计中补充 `_load_last_incremental_count()` 的实现逻辑：
1. 首次调用时从 `trigger_state.json` 加载，若无则返回 0
2. 增量学习执行成功后，通过 `save_trigger_state("last_incremental_count", total_count)` 更新
3. 明确 trigger_state.json 的 Schema：`{"last_incremental_count": int, "last_monthly_report": str}`

---

#### M-04: 提示调优参数缺乏版本历史，回滚粒度不足

**维度**: 可维护性

**问题描述**: 需求规格 REQ-0.25-02 AC-04 要求"优化策略存储为配置文件，可回滚(reset_to_default())"，但 `reset_to_default()` 仅能回滚到初始默认值（全部 0.5），无法回滚到上一次调整前的状态。若用户手动调优到满意状态后，自动微调破坏了手动设置，无法恢复。

**整改建议**: 
1. 在 PromptTuner 中添加参数快照机制：每次更新前保存当前参数到 `tuning/params_history.jsonl`（追加写入）
2. 新增 `rollback_last_change()` 方法，从历史记录中恢复上一次参数
3. 保留 `reset_to_default()` 作为终极回滚手段

---

#### M-05: EvolutionEngine.get_evolution_status() 输出结构变化缺乏版本管理

**维度**: 向后兼容性

**问题描述**: v0.25 在 `get_evolution_status()` 返回值中新增 `evolution_status` 嵌套字段，输出结构从扁平变为嵌套。虽然新增字段不影响现有字段，但 CLI `evolution status` 命令和 Agent 工具可能依赖此输出结构。

**整改建议**: 在 `get_evolution_status()` 的 docstring 中明确输出结构的版本演进，并在 CLI handler 中添加版本感知的解析逻辑。

### 2.4 LOW 级别

#### L-01: EvolutionReport.prediction_accuracy_trend 缺乏结构化定义

**维度**: 可维护性

**问题描述**: `prediction_accuracy_trend: list[dict[str, Any]]` 无明确 Schema，下游代码无法类型安全地访问趋势数据。

**整改建议**: 定义 `AccuracyTrendPoint` dataclass（date/mae/direction），替换 `dict[str, Any]`。

---

#### L-02: EvolutionController 与 EvolutionReporter 间接循环依赖

**维度**: 架构一致性

**问题描述**: EvolutionController 依赖 EvolutionReporter（generate_report 动作），EvolutionReporter 依赖 PromptTuner 和 CalibrationEngine，PromptTuner 被 EvolutionController 持有。虽然不是直接循环依赖，但依赖链较长。

**整改建议**: 当前可接受，但需在代码中添加注释说明依赖关系。若后续 EvolutionReporter 需要访问 EvolutionController 的状态，需重构为事件驱动模式。

---

#### L-03: CLI `evolution tune --data-driven` 参数名与 Python 命名规范不一致

**维度**: 可维护性

**问题描述**: CLI 参数 `--data-driven` 含连字符，Typer 会自动转换为 `data_driven`，但架构设计文档中 Agent 工具参数使用 `data_driven`（下划线），两者命名风格不一致。

**整改建议**: 统一文档描述，CLI 使用 `--data-driven`（符合 CLI 惯例），Agent 工具使用 `data_driven`（符合 Python 惯例），在文档中明确说明转换关系。

### 2.5 建议项

#### S-01: 为 daemon 线程添加执行队列

当前架构每次触发进化动作都创建新的 daemon 线程，若短时间内多次触发（如连续 3 次 VDOT 误差触发 + 连续 2 次拒绝触发），可能创建多个并发线程，导致：
- 同一模型被并发进化，参数写入冲突
- 资源浪费

**建议**: 使用 `queue.Queue + 单 daemon 线程` 模式，进化动作入队后由单线程串行执行，避免并发问题。

#### S-02: 为 check_triggers() 添加节流机制

DecisionLogHook.after_iteration() 每次迭代都调用 check_triggers()，即使触发条件不太可能变化（如 1 分钟内连续 3 次迭代）。建议添加时间节流：距上次检查不足 5 分钟时跳过。

#### S-03: PromptTuningParams 添加 last_auto_adjust_reason 字段

当前 PromptTuningParams 仅记录 update_count，无法追溯调整原因。建议新增 `last_adjust_reason: str` 字段（如 "rejection"/"feedback"/"manual"），便于 EvolutionReporter 生成更详细的调优摘要。

#### S-04: EvolutionAction 添加 cancelled 状态

当前 EvolutionAction 仅有 executed=True/False，缺少"用户手动取消"状态。建议在 CLI `evolution triggers` 命令中支持跳过某个动作，将 action 标记为 cancelled。

## 3. 评审结论

### 3.1 总体评价

v0.25.0 架构设计在以下方面表现良好：

1. **递增式添加模式**：严格遵循 v0.23/v0.24 建立的递增式添加模式，3 个新子组件以可选注入方式接入，无侵入原则执行到位
2. **数据模型规范**：4 个新数据模型均使用 frozen dataclass，提供 to_dict()/from_dict() 序列化方法，与现有规范一致
3. **依赖注入模式**：AppContext 扩展、EvolutionEngine 构造函数扩展、DecisionLogHook 可选注入，均与 v0.23/v0.24 模式一致
4. **ADR 决策记录**：ADR-009（规则引擎 + 异步执行）和 ADR-010（4 维连续参数空间）决策有据，选型合理
5. **风险缓解**：6 项风险缓解措施覆盖了主要风险场景
6. **测试策略**：6 项测试覆盖范围明确，Mock 策略合理

### 3.2 需整改项

| 级别 | 数量 | 必须整改 | 建议整改 |
|------|------|---------|---------|
| CRITICAL | 2 | C-01, C-02 | - |
| HIGH | 3 | H-01, H-02, H-03 | - |
| MEDIUM | 5 | M-03 | M-01, M-02, M-04, M-05 |
| LOW | 3 | - | L-01, L-02, L-03 |

### 3.3 结论

**有条件通过**

- C-01（daemon 线程数据一致性）和 C-02（check_triggers 性能）必须在开发前完成整改方案设计
- H-01（编排层一致性）、H-02（部分失败处理）、H-03（参数下限保护）必须在开发中解决
- M-03（trigger_state 定义）必须在开发前补充设计
- 其余 MEDIUM/LOW 项建议整改，不阻塞开发
