## 3. 问题清单

### 3.1 HIGH 级问题（必须整改）

---

#### [HIGH] #1: Subagent 无法直接访问 RunnerTools，数据分析能力受限

- **文件**: `v0.17.0_架构设计说明书.md` — ADR-002
- **关联需求**: REQ-0.17-02（Subagent 基础验证）
- **问题描述**: 
  ADR-002 明确指出 "Subagent 工具集固定（文件操作 + Exec + Web），无法直接使用 RunnerTools"。但 DataAnalystAgent 的职责定义为"分析 VDOT 趋势、计算训练负荷（ATL/CTL/TSB）、检测心率漂移、评估训练效果"，这些操作**全部依赖 RunnerTools**（`get_vdot_trend`、`get_training_load`、`get_hr_drift_analysis` 等）。如果 Subagent 无法调用 RunnerTools，它将无法完成其核心职责。
- **影响评估**: 
  - REQ-0.17-02 的核心验收标准 AC-01（主 Agent 成功调用 DataAnalystAgent 并获取数据分析结果）可能无法达成
  - Subagent 验证将退化为"仅验证调用链通顺"，无法验证实际业务能力
- **整改建议**（三选一）:
  1. **方案A（推荐）**: 研究底座 SubagentManager 是否支持自定义工具集注入，将 RunnerTools 的部分只读方法注册到 Subagent
  2. **方案B**: 主 Agent 在 spawn Subagent 时，将所需数据作为上下文传入（通过 task 参数携带序列化数据），Subagent 仅做数据解释
  3. **方案C**: 调整 Subagent 职责描述，明确其仅做"LLM 层面的数据解释"，数据查询由主 Agent 完成后再传入
- **责任人**: 架构师

---

#### [HIGH] #2: `on_tool_error` Hook 方法与架构设计不一致

- **文件**: `PRD_v0.17.0_底座能力全面激活.md` REQ-0.17-04 vs `v0.17.0_架构设计说明书.md` 4.1.4
- **关联需求**: REQ-0.17-04（CLI 错误处理增强）
- **问题描述**: 
  PRD 明确要求"实现 `on_tool_error` Hook 方法，捕获工具执行错误"。但架构设计中的 ErrorHandlingHook 使用的是 `after_iteration(context)` + 检查 `context.error`。架构文档的 AgentHook 生命周期图（2.2.1节）中**没有 `on_tool_error` 方法**。需要确认底座 AgentHook 是否提供 `on_tool_error` 回调。
- **影响评估**: 
  - 如果底座有 `on_tool_error` 但架构未使用，错误捕获时机可能不准确（`after_iteration` 在工具执行完成后触发，可能遗漏某些错误场景）
  - PRD 与架构文档的术语不一致，可能导致开发工程师困惑
- **整改建议**:
  1. 确认 nanobot-ai AgentHook 是否有 `on_tool_error` 方法
  2. 如果有，ErrorHandlingHook 应优先使用 `on_tool_error`，并更新生命周期图
  3. 如果没有，在架构文档中增加明确说明："底座 AgentHook 不提供 `on_tool_error` 回调，改用 `after_iteration` + `context.error` 实现等价功能"，并同步更新 PRD 中的术语
- **责任人**: 架构师

### 3.2 MEDIUM 级问题（建议整改）

#### [MEDIUM] #3: CronService 的 `on_job` 回调机制未在接口规范中定义

- **文件**: `v0.17.0_架构设计说明书.md` — 4.3.6 & 5.3 & 7.3
- **关联需求**: REQ-0.17-03（Cron 训练提醒深度集成）
- **问题描述**: 
  架构文档指出当前 `gateway.py` 中 CronService 未传入 AgentLoop，导致 CronTool 未注册。数据流图（5.3）展示了 `Cron->>Agent: on_job(job) → process_direct(...)` 的流程，但接口规范（第7节）中**没有给出 `cron.on_job` 回调的具体设置代码**。风险表中也标注了此风险，但缓解措施只说"Gateway 启动时设置"，缺少具体方案。
- **影响评估**: 开发工程师可能不清楚如何正确连接 CronService 和 AgentLoop，导致 Cron 任务触发但无响应
- **整改建议**: 在接口规范（7.3节）中补充 `cron.on_job` 回调的注册代码示例：
  ```python
  async def on_cron_job(job):
      await agent.process_direct(job.payload.message)
  cron.on_job = on_cron_job
  ```
- **责任人**: 架构师

#### [MEDIUM] #4: `ask_user` 工具的实现方案存在架构模糊性

- **文件**: `v0.17.0_架构设计说明书.md` — 4.6
- **关联需求**: REQ-0.17-07（ask_user 训练确认）
- **问题描述**: 
  架构文档指出"当前底座不直接支持同步等待用户输入"，CLI 模式下降级为 Rich `Prompt.ask`/`Confirm.ask`。但这意味着 Agent 调用 `ask_user` 工具时，CLI 循环需要**拦截该工具调用**并暂停 Agent 执行、等待用户输入、再将输入传回 Agent。架构文档没有说明这个"拦截-暂停-恢复"机制如何实现。
- **影响评估**: ask_user 的实现可能需要在 AgentLoop 外部做大量胶水代码，架构复杂度被低估
- **整改建议**:
  1. 补充 ask_user 工具在 CLI 模式下的完整交互时序图
  2. 明确是否需要自定义工具来替代底座的 `ask_user`，还是通过 Hook 机制拦截
  3. 评估是否将 REQ-0.17-07 降级为 P2 或标记为"实验性功能"
