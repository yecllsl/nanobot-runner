## 四、方案设计问题

### 4.1 `ReportGenerator` 拆分方案不完整

文档声称 `ReportGenerator` "已存在，需迁移"，但实际 `ReportGenerator` 有 **1023+ 行**，本身就是一个需要拆分的大类。文档将其原封不动放入拆分后方案，未给出进一步拆分策略。

**修正建议**：要么在 v0.9.0 中一并拆分 `ReportGenerator`，要么明确标注为 v0.9.1/v0.9.2 任务。

### 4.2 依赖注入方案缺乏迁移路径

`AppContext` + `AppContextFactory` 方案本身合理，但缺少从当前模式到目标模式的**渐进式迁移路径**。特别是：

- `ProfileEngine` 内部延迟导入 `AnalyticsEngine` 的问题如何解决？
- `ReportGenerator` 和 `ReportService` 是否也需要注入 `AppContext`？
- `RunnerTools` 当前是 `BaseTool` 的工厂，它是否持有 `AppContext`？

**修正建议**：补充迁移路径描述，至少给出每个受影响模块的改造方案。

### 4.3 `PaceAnalyzer` 定义无现实依据

`AnalyticsEngine` 拆分方案中的 `PaceAnalyzer` 预估 ~150 行，列出方法 `get_pace_distribution()`、`get_best_pace()`、`get_pace_trend()`。但需确认这些方法在当前 `AnalyticsEngine` 中是否实际存在。如果不存在，这就是**虚构的拆分目标**。

### 4.4 适配器模式与直接删除的矛盾

文档第六章风险应对中提出用适配器模式保留 `AnalyticsEngine` 旧接口（标记 `@deprecated`），但第五章 Sprint 1.1 又说"删除 IntentParser 及相关模型"。同一个重构方案中，对死代码（`IntentParser`）选择直接删除，对拆分后旧代码（`AnalyticsEngine` 旧接口）选择适配器保留，策略不一致。

**修正建议**：明确何时用适配器、何时直接删除的决策标准。对于 `IntentParser` 这种零引用的死代码，直接删除合理；对于 `AnalyticsEngine` 拆分后的旧接口，适配器过渡也合理，但需明确过渡期和最终删除时间。

### 4.5 `SessionRepository` 与 `StatisticsAggregator` 职责重叠

文档同时定义了：
- `StatisticsAggregator`：数据聚合、统计摘要、会话聚合
- `SessionRepository`：统一会话聚合

两者都有"会话聚合"的职责，边界不清。

**修正建议**：明确 `SessionRepository` 是数据访问层（封装 Parquet 读写 + 基础聚合），`StatisticsAggregator` 是业务计算层（基于 `SessionRepository` 返回的数据做进一步统计计算）。

## 五、技术准确性问题

### 5.1 `SEC-001` 行号不准确

文档声称 fitparse monkey-patching 位于 `parser.py:16-86`，经验证**行号准确**。但描述为"Monkey-patching fitparse，升级风险高"，严重程度标为🟡中。实际上直接修改第三方库的全局行为在运行时生效，升级 `fitparse` 后如果内部实现变化会导致**静默失败**，风险应评为🔴高。

### 5.2 `SEC-002` 描述不够精确

文档说"飞书App Secret明文存储"在 `config.py:79`。实际该行是默认模板中的空字符串 `""`，并附有 `# nosec B105` 压制了安全扫描。**真正的问题**是运行时敏感凭据以明文 JSON 存储在 `~/.nanobot-runner/config.json`。文档应更精确地描述问题位置和影响。

### 5.3 `DATA-004` 严重程度应升级

`IndexManager._save_index` 中 `self.index["metadata"]["updated"] = str(Path.home())` 将用户主目录路径（如 `C:\Users\yecll`）写入 `updated` 时间戳字段。这不是一般 bug，而是**数据语义完全错误**——metadata 中的 `updated` 字段永远不可能被正确使用。严重程度应从🔴高提升，或至少在描述中明确指出这不是一个"off-by-one"级别的小 bug。

## 六、合规性与一致性问题

### 6.1 文档声称的"参考文档"与实际不符

### 6.2 量化指标缺乏基线测量

重构目标表中声称"查询性能提升≥30%"、"导入性能提升≥50%"，但未给出当前基线值。第七章的性能测试基准提供了部分目标值（如"TSS 计算 100 次跑步: 500ms → ≤350ms"），但与目标表中的百分比提升没有一一对应关系。

**修正建议**：要么在目标表中给出基线值，要么在性能基准表中确保与目标表的百分比一致。

文档声称新增测试路径为 `tests/unit/core/analytics/*.py`，但实际测试目录中 analytics 测试位于 `tests/unit/test_analytics.py`（单文件 132KB）。重构后是保持单文件还是拆分？如果拆分，需要明确。

文档 Sprint 1.2 说"`__init__.py` 改为 0.8.0"，但整个项目正在进行的是 **v0.9.0 重构**。版本号应统一为 **0.9.0**，而非同步到旧版本 0.8.0。
