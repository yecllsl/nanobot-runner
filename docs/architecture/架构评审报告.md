**[CRITICAL] 文档不一致：ML技术栈描述存在矛盾**

- **涉及文档**:

  - [架构设计说明书 v6.0.0](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/docs/architecture/架构设计说明书.md) ADR-001

- **问题描述**:
  产品规划方案和需求规格说明书中多处描述为 "ML时间序列模型（LSTM/Prophet）"，但架构设计说明书 ADR-001 明确选择了 scikit-learn（GradientBoostingRegressor/Classifier），并明确拒绝了 Prophet（依赖过重~200MB）和 PyTorch/TensorFlow（过重）。

- **影响**:

  1. 按需求文档引入 Prophet/PyTorch 等重依赖
  2. 实现方向与架构决策偏离
  3. 返工和工期延误

- **整改方案**:
  1. 将产品规划方案 v8.0 第3.1节中 "LSTM/Prophet" 统一修改为 "scikit-learn（GradientBoosting + 时序特征工程）"
  2. 将需求规格说明书 v7.0 第2.1节中 "LSTM/Prophet" 统一修改为 "scikit-learn（GradientBoosting + 时序特征工程）"
  3. 在需求规格说明书第2.1节增加对 ADR-001 的引用，确保技术决策可追溯

**[CRITICAL] AppContext中prediction_engine违反依赖注入规范**

- **文件**: [架构设计说明书](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/docs/architecture/架构设计说明书.md) 第6.7节

- **问题描述**:
  `prediction_engine` 属性的延迟加载代码中直接实例化了核心组件：
  ```python
  feature_engine = FeatureEngine(
      training_load_analyzer=TrainingLoadAnalyzer(),  # ❌ 直接实例化
      vdot_calculator=VDOTCalculator(),                # ❌ 直接实例化
  )
  vdot_predictor = VDOTPredictor(
      race_engine=RacePredictionEngine(),              # ❌ 直接实例化
  )
  injury_predictor = InjuryPredictor(
      injury_analyzer=InjuryRiskAnalyzer(),            # ❌ 直接实例化
  )
  ```
  违反了项目 AGENTS.md 铁律：**"禁止直接实例化核心组件：必须通过 get_context() 获取应用上下文"**。

- **影响**:
  1. 绕过依赖注入，导致测试时无法 Mock 这些组件
  2. 与项目现有模式不一致（body_signal_engine 通过 session_repo 等上下文属性注入）
  3. 可能导致多个实例并存，状态不一致

- **整改方案**:
  1. `TrainingLoadAnalyzer` 和 `VDOTCalculator` 应作为 AppContext 的延迟属性暴露（或复用现有 analytics 引擎中的实例）
  2. `RacePredictionEngine` 和 `InjuryRiskAnalyzer` 应通过 AppContext 属性获取
  3. 参考 body_signal_engine 的实现模式，所有依赖通过 AppContext 属性注入
  4. 建议新增以下 AppContext 属性：
     ```python
     @property
     def training_load_analyzer(self) -> TrainingLoadAnalyzer: ...
     @property
     def vdot_calculator(self) -> VDOTCalculator: ...
     @property
     def race_prediction_engine(self) -> RacePredictionEngine: ...
     @property
     def injury_risk_analyzer(self) -> InjuryRiskAnalyzer: ...
     ```

**[HIGH] 缺少 PredictionConfig 配置 Schema**

- **文件**: [架构设计说明书](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/docs/architecture/架构设计说明书.md) 第6章

- **问题描述**:
  v0.19 的 body_signal 模块有完整的 `BodySignalConfig` 配置 Schema（5.6节），支持环境变量覆盖和权重校验。v0.20 的 prediction 模块缺少对应的 `PredictionConfig`，以下参数散落在各处未统一管理：
  - 模型超参数（GradientBoosting 的 n_estimators、learning_rate、max_depth）
  - 数据充足度阈值（18个月、400条、心率完整度80%）
  - SHAP 采样参数（max_evals=100）
  - 增量学习触发阈值（50条）
  - 风险预警阈值（概率>60%）
  - 赛前状态修正幅度（疲劳下调2-5%、恢复上调1-3%）

- **影响**:
  1. 魔法数字散落代码，不符合项目"配置驱动"原则
  2. 用户无法根据个人情况调整阈值
  3. 测试时需要修改源码而非配置

- **整改方案**:
  1. 新增 `src/core/prediction/config.py`，定义 `PredictionConfig` frozen dataclass
  2. 遵循 `BodySignalConfig` 的模式：`@dataclass(frozen=True)` + `__post_init__` 校验
  3. 环境变量前缀：`NANOBOT_PREDICTION_`
  4. 在 `config.json` 中新增 `prediction` 字段
  5. 在架构文档中补充配置 Schema 定义章节

**[HIGH] 缺少预测结果缓存机制**

- **文件**: [架构设计说明书](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/docs/architecture/架构设计说明书.md) 第6章

- **问题描述**:
  body_signal_engine 有同日缓存机制（5.3.4节），同一自然日内多次查询复用计算结果。prediction 模块没有类似设计，但预测计算（特征提取 + ML推理 + SHAP分析）比身体信号分析更重（目标<5秒 vs <2秒）。

- **影响**:
  1. Agent 对话中连续多次调用预测工具时重复计算
  2. CLI 中用户连续查询不同天数预测时无法复用特征提取结果
  3. 性能目标<5秒在无缓存时可能难以稳定达成

- **整改方案**:
  1. PredictionEngine 增加同日缓存：同一自然日内相同预测类型复用结果
  2. FeatureEngine 增加特征矩阵缓存：特征提取结果按 (days, 数据截止日期) 缓存
  3. 缓存失效策略：新数据导入时（import 命令）清除所有预测缓存
  4. 在架构文档中补充缓存机制设计

**[HIGH] 模型存储路径命名不一致**

- **文件**: [架构设计说明书](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/docs/architecture/架构设计说明书.md) 第6.5.6节

- **问题描述**:
  架构文档定义模型存储路径为 `~/.nanobotrun/models/`，但项目现有数据目录为 `~/.nanobot-runner/data/`（注意 `nanobot-runner` 中有连字符，而 `nanobotrun` 没有）。命名不一致会导致：
  - 用户目录下出现两个不同的隐藏文件夹
  - 备份策略需要覆盖两个目录

- **影响**:
  1. 目录结构混乱，用户困惑
  2. 备份命令可能遗漏模型文件

- **整改方案**:
  1. 统一使用 `~/.nanobot-runner/` 作为根目录（与现有 data 目录一致）
  2. 模型存储路径改为 `~/.nanobot-runner/models/`
  3. 或在 AppConfig 中统一管理 `models_dir` 路径，由 ConfigManager 读取

**[HIGH] 缺少模型预热/冷启动策略**

- **文件**: [架构设计说明书](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/docs/architecture/架构设计说明书.md) 第6.5.3节

- **问题描述**:
  预测流程中，如果 ML 模型不存在，会在首次预测时自动训练（6.5.3节流程图）。模型训练目标<60秒，加上特征提取<3秒 + SHAP<5秒，首次预测总耗时可能达到 60-70 秒，远超<5秒的性能目标。

- **影响**:
  1. 首次用户体验极差，等待超过1分钟
  2. Agent 对话中可能触发 LLM 超时
  3. 用户可能以为程序卡死

- **整改方案**:
  1. 在 `predict status` 命令中增加"一键训练"选项，让用户主动触发训练
  2. 或在 `system init` 完成后提示用户可运行 `predict model train --all` 预训练
  3. 首次自动训练时显示 Rich 进度条，明确告知用户正在训练
  4. 在架构文档中补充冷启动策略章节

- **文件**: [架构设计说明书](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/docs/architecture/架构设计说明书.md) 第6.12节

  2. 或使用 sklearn 内置的 `feature_importances_` 作为快速降级方案
  3. 在架构文档中补充 SHAP 降级策略

## 6. 架构模式评审详情

### 6.1 分层架构（✅ 通过）

```
CLI (predict.py) → Handler (predict_handler.py) → PredictionEngine → Predictors → FeatureEngine/DataAssessor
```

| 模块 | 职责 | 评审 |
|------|------|------|
| `prediction_engine.py` | 编排层，统一入口 | ✅ 单一职责 |
| `vdot_predictor.py` | VDOT趋势预测 | ✅ 单一职责 |
| `race_predictor.py` | 个人化比赛预测 | ✅ 单一职责 |
| `injury_predictor.py` | ML伤病风险预测 | ✅ 单一职责 |
| `feature_engine.py` | 特征工程 | ✅ 单一职责 |
| `data_assessor.py` | 数据充足度评估 | ✅ 单一职责 |

### 6.3 依赖关系（⚠️ 警告）

| 依赖方向 | 评审 | 说明 |

## 8. 风险识别补充

架构文档已识别的风险较全面（6.14节），以下为补充风险：

| 风险 | 等级 | 说明 | 建议 |
|------|------|------|------|
| numpy 版本兼容性 | 🟡 中 | scikit-learn≥1.3.0 与 numpy≥2.4.2 的兼容性需验证。numpy 2.x 有 Breaking Changes | 在开发环境先验证 `pip install scikit-learn>=1.3.0` 不冲突 |
| 循环依赖风险 | 🟡 中 | prediction_engine 依赖 body_signal_engine，两者都是 AppContext 延迟属性 | 确保 prediction_engine 的加载不触发 body_signal_engine 的提前初始化 |
| 模型文件损坏 | 🟡 中 | joblib 文件可能因磁盘故障损坏 | 模型加载时增加 try/except，损坏时自动重训 |
| 特征提取失败 | 🟡 中 | 依赖多个上游模块（TrainingLoadAnalyzer/BodySignalEngine/HRVAnalyzer），任一失败导致特征缺失 | FeatureEngine 增加容错：单个特征提取失败时跳过该特征，记录 warning |
