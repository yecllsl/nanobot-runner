## 1. 风险概述

本文档记录 Nanobot Runner 项目全生命周期中的已知风险、应对措施及监控状态。

### 1.1 风险等级定义

| 等级 | 标识 | 定义 |
|------|------|------|
| **高** | 🔴 | 可能导致项目重大延期或失败，需立即处理 |
| **中** | 🟡 | 可能影响项目进度或质量，需持续监控 |
| **低** | 🟢 | 影响较小，可按计划处理 |

### 1.2 风险状态定义

| 状态 | 说明 |
|------|------|
| **已识别** | 风险已发现，正在评估 |
| **处理中** | 风险应对措施正在执行 |
| **已缓解** | 风险已得到控制，持续监控 |
| **已关闭** | 风险已消除或不再适用 |
| **已发生** | 风险已转化为问题，正在解决 |

## 2. 已关闭风险（历史记录）

### R-001: CI/CD mypy类型检查失败

| 属性 | 内容 |
|------|------|
| **风险ID** | R-001 |
| **风险描述** | GitHub Actions CI流程中mypy类型检查失败，导致构建阻断 |
| **风险等级** | 🟡 中 |
| **发生时间** | v0.3.0开发期 |
| **根本原因** | 1. 代码中使用`any`而非`Any`<br>2. Decimal与float运算类型不匹配<br>3. polars复杂类型未处理 |
| **应对措施** | 1. 修复indexer.py中的类型注解<br>2. 添加float()显式转换<br>3. 添加`# type: ignore`注释 |
| **当前状态** | ✅ 已关闭 |
| **经验教训** | 本地开发需先运行mypy检查，避免提交后才发现 |

### R-002: E2E测试uv命令不兼容

| 属性 | 内容 |
|------|------|
| **风险ID** | R-002 |
| **风险描述** | E2E测试中使用`uv run python`命令，GitHub Actions Linux环境无uv |
| **风险等级** | 🟡 中 |
| **发生时间** | v0.3.0开发期 |
| **根本原因** | 假设开发环境与CI环境一致，未考虑uv的可用性 |
| **应对措施** | 将`uv run python`替换为直接使用`python`命令，添加sys.path处理 |
| **当前状态** | ✅ 已关闭 |
| **经验教训** | E2E测试需考虑跨平台兼容性，避免依赖特定工具 |

### R-003: 文档重复度高，维护困难

| 属性 | 内容 |
|------|------|
| **风险ID** | R-003 |
| **风险描述** | 需求文档、测试报告等多版本重复，内容同步困难 |
| **风险等级** | 🟢 低 |
| **发生时间** | v0.2.0→v0.3.0 |
| **根本原因** | 缺乏文档版本管理规范，历史文档未归档 |
| **应对措施** | 1. 归档历史版本文档<br>2. 明确文档差异化定位<br>3. 建立文档导航索引 |
| **当前状态** | ✅ 已关闭 |
| **经验教训** | 建立文档生命周期管理规范，定期清理过期文档 |

### R-005: 版本号管理不一致

| 属性 | 内容 |
|------|------|
| **风险ID** | R-005 |
| **风险描述** | pyproject.toml、AGENTS.md、README.md版本号不一致 |
| **风险等级** | 🟢 低 |
| **发生时间** | v0.3.0开发期 |
| **根本原因** | 版本号分散在多个文件，手动更新易遗漏 |
| **应对措施** | 建立版本发布Checklist，统一更新所有版本号位置 |
| **当前状态** | ✅ 已关闭 |
| **经验教训** | 考虑使用bumpversion等工具自动化版本管理 |

## 3. 活跃风险（持续监控）

### R-004: 网络不稳定导致Git推送失败

| 属性 | 内容 |
|------|------|
| **风险ID** | R-004 |
| **风险描述** | 国内网络环境访问GitHub不稳定，导致代码推送失败 |
| **风险等级** | 🟡 中 |
| **发生概率** | 高（偶发） |
| **影响范围** | 发布流程、代码同步 |
| **根本原因** | 网络基础设施限制，非技术问题 |
| **应对措施** | 1. 多次重试推送<br>2. 使用代理或镜像<br>3. 本地备份重要提交 |
| **预防措施** | 1. 避免在截止时间前才推送<br>2. 提前完成代码提交 |
| **监控指标** | 推送成功率、平均重试次数 |
| **当前状态** | ⚠️ 已缓解（持续监控） |
| **责任人** | @发布运维工程师 |

## 4. 潜在风险（预防为主）

### R-006: FIT文件格式兼容性（新设备）

| 属性 | 内容 |
|------|------|
| **风险ID** | R-006 |
| **风险描述** | 不同厂商（Garmin、Wahoo、Polar等）FIT文件字段定义有差异，可能导致解析失败 |
| **风险等级** | 🔴 高 |
| **发生概率** | 中 |
| **影响范围** | 核心功能（数据导入） |
| **根本原因** | 1. FIT协议允许厂商扩展<br>2. 测试数据集单一<br>3. 缺乏多厂商设备测试 |
| **应对措施** | **短期**: 收集多厂商测试数据，扩展解析器<br>**中期**: 实现容错解析，未知字段标记但不阻断<br>**长期**: 建立设备兼容性数据库 |
| **预防措施** | 1. 建立FIT文件样本库<br>2. 实现字段级容错处理<br>3. 增加兼容性测试用例 |
| **监控指标** | FIT解析失败率、新设备支持数量 |
| **当前状态** | 📝 已识别 |
| **责任人** | @测试工程师 @开发工程师 |
| **计划完成** | v0.4.0 |

### R-007: 数据量增长导致性能下降

| 属性 | 内容 |
|------|------|
| **风险ID** | R-007 |
| **风险描述** | 随着跑步记录增加，查询响应时间可能超过3秒目标 |
| **风险等级** | 🟡 中 |
| **发生概率** | 高 |
| **影响范围** | 用户体验、查询性能 |
| **根本原因** | 1. 当前未实现数据分区<br>2. 缺乏查询缓存<br>3. 架构设计未考虑大数据量 |
| **应对措施** | 1. 实现数据分区（按月/季度）<br>2. 添加查询缓存机制<br>3. 性能基准测试自动化 |
| **预防措施** | 1. 定期性能测试（每月）<br>2. 监控查询响应时间<br>3. 设置性能告警阈值 |
| **监控指标** | 查询响应时间P95、内存占用峰值 |
| **当前状态** | 📝 已识别 |
| **责任人** | @架构师 @开发工程师 |
| **计划完成** | v0.4.0 |

### R-008: 依赖库版本冲突

| 属性 | 内容 |
|------|------|
| **风险ID** | R-008 |
| **风险描述** | 依赖库更新可能导致版本冲突或API不兼容 |
| **风险等级** | 🟡 中 |
| **发生概率** | 中 |
| **影响范围** | 构建稳定性、功能正确性 |
| **根本原因** | 依赖版本未锁定，自动更新可能引入破坏性变更 |
| **应对措施** | 1. 使用uv.lock锁定依赖版本<br>2. 建立依赖更新流程<br>3. 兼容性测试矩阵 |
| **预防措施** | 1. 定期审查依赖更新<br>2. 在隔离环境测试更新<br>3. 渐进式更新策略 |
| **监控指标** | 依赖更新频率、构建失败率 |
| **当前状态** | 📝 已识别 |
| **责任人** | @发布运维工程师 |
| **计划完成** | 持续进行 |
