## 📋 归档规则

### 归档时机

- 版本发布成功后立即归档（OPS-05 文档归档）
- 在发布准出验证通过后执行

### 归档内容

每个版本的归档目录应包含以下子目录：

#### 1. test/ - 测试相关文档

- `测试报告_*.md` - 测试执行报告
- `Bug清单_*.md` - Bug列表和修复状态
- `回归报告_*.md` - 回归测试报告
- `上线结论_*.md` - 上线决策文档
- `E2E测试报告_*.md` - E2E测试报告
- `测试策略_*.md` - 测试策略文档

#### 2. devops/ - 运维相关文档

- `流水线执行报告_*.md` - CI/CD流水线验证报告
- `发布报告_*.md` - 版本发布报告
- `故障排查报告_*.md` - 故障排查文档

#### 3. development/ - 开发相关文档

- `交付报告_*.md` - 开发交付报告
- `Bug修复报告_*.md` - Bug修复报告
- `代码优化报告_*.md` - 代码优化报告

#### 4. review/ - 评审相关文档

- `代码评审报告_*.md` - 代码评审报告
- `架构评审报告_*.md` - 架构评审报告

## 🔍 查找指南

### 如何查找历史版本文档？

1. **按版本号查找**：直接访问对应版本目录，如 `docs/archive/v0.9.0/`
2. **按文档类型查找**：访问版本目录下的子目录，如 `docs/archive/v0.9.0/test/`
3. **按时间查找**：查看版本索引表中的发布日期

### 常见查找场景

- **查找Bug修复记录**：`docs/archive/{版本号}/development/Bug修复报告_*.md`
- **查找测试报告**：`docs/archive/{版本号}/test/测试报告_*.md`
- **查找发布记录**：`docs/archive/{版本号}/devops/发布报告_*.md`
- **查找代码评审**：`docs/archive/{版本号}/review/代码评审报告_*.md`

## 📝 归档操作

### 自动归档（推荐）

执行 `OPS-05 文档归档` 指令，系统会自动完成以下操作：

1. 创建版本归档目录
2. 移动版本相关文档
3. 清理项目目录
4. 更新本索引文件

### 手动归档

如需手动归档，请按以下步骤操作：

```bash
# 1. 创建版本归档目录
mkdir -p docs/archive/v{版本号}/{test,devops,development,review}

# 2. 移动测试文档
mv docs/test/reports/*_v{版本号}.md docs/archive/v{版本号}/test/

# 3. 移动运维文档
mv docs/devops/*_v{版本号}.md docs/archive/v{版本号}/devops/

# 4. 移动开发文档
mv docs/development/*_v{版本号}.md docs/archive/v{版本号}/development/

# 5. 移动评审文档
mv docs/review/*_v{版本号}.md docs/archive/v{版本号}/review/

# 6. 更新归档索引
# 编辑本文件，添加新版本信息
```

## ⚠️ 注意事项

1. **不要删除归档文档**：归档文档是项目历史的重要记录，应永久保留
2. **不要修改归档文档**：归档文档应保持原始状态，如需修改请创建新版本
3. **及时归档**：版本发布后应立即归档，避免文档堆积影响项目目录整洁
4. **更新索引**：归档后必须更新本索引文件，确保版本信息准确
