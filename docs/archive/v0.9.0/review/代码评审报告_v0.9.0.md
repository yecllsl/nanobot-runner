## 四、代码质量评审

### 4.1 命名规范 ✅

- 函数/变量 snake_case：`calculate_vdot_batch`, `get_sessions` ✅
- 常量 UPPER_SNAKE_CASE：`VDOT_COEFFICIENT`, `ATL_TIME_CONSTANT` ✅

### 4.2 类型注解 ⚠️

核心模块类型注解覆盖率整体良好，但存在以下问题：

```
[P1] SessionRepository.storage 缺少类型注解
File: src/core/session_repository.py:30
Issue: def __init__(self, storage) -> None: 中 storage 参数没有类型注解，
       应为 storage: StorageManager。公共函数缺少类型注解违反项目规范。
Fix: 添加类型注解：def __init__(self, storage: StorageManager) -> None:
```

```
[P2] SessionRepository 返回类型使用 List[Dict[str, Any]]
File: src/core/session_repository.py:54-160
Issue: get_recent_sessions, get_sessions_by_date_range, get_sessions_by_distance
       等方法返回 List[Dict[str, Any]]，过度使用 Any 不利于类型安全。
       建议定义 TypedDict 或 dataclass 作为返回类型。
Fix: 定义 SessionSummary dataclass 替代 Dict[str, Any]，
     提升类型安全性和 IDE 补全体验。
```

### 4.3 异常处理 ⚠️

```
[P1] agent.py 裸 Exception + continue 静默吞没异常
File: src/cli/commands/agent.py:160
Issue: try/except Exception: continue 模式静默吞没所有异常，
       包括 KeyboardInterrupt 等不应被捕获的异常。
       Bandit 标记为 B112 (try_except_continue)。
Fix: 捕获具体异常（如 FileNotFoundError, OSError），
     或至少添加日志记录：except (FileNotFoundError, OSError) as e: logger.debug(...)
```

```
[P2] config.py 使用 assert 进行类型缩窄
File: src/core/config.py:140
Issue: assert ConfigManager._cache is not None 在优化模式下会被移除，
       可能导致 None 访问异常。Bandit 标记为 B101 (assert_used)。
Fix: 使用 if/raise 模式替代：
     if ConfigManager._cache is None:
         raise RuntimeError("配置缓存不应为 None")
     return ConfigManager._cache.copy()
```

### 4.4 代码复杂度与重复 ❌

```
[P1] SessionRepository 严重代码重复
File: src/core/session_repository.py:54-160
Issue: get_recent_sessions, get_sessions_by_date_range, get_sessions_by_distance
       三个方法包含几乎完全相同的行迭代逻辑（distance/duration/pace 计算），
       违反 DRY 原则。每新增一个查询方法都需要复制粘贴相同的转换代码。
Fix: 提取私有方法 _format_session_row() 封装行转换逻辑，
     或使用 Polars 表达式直接在 DataFrame 层完成转换，
     消除 Python 层的 iter_rows 循环。
```

```
[P2] SessionRepository 使用 iter_rows 而非 Polars 表达式
File: src/core/session_repository.py:54-160
Issue: 多个方法使用 for row in session_df.iter_rows(named=True) 逐行处理，
       这是 C 风格循环，违反项目"Polars 规范"要求。
       应使用 Polars 表达式（with_columns + alias）批量完成计算。
Fix: 使用 Polars 表达式替代 iter_rows 循环：
     df.with_columns([
         (pl.col("distance") / 1000).round(2).alias("distance_km"),
         (pl.col("duration") / 60).round(1).alias("duration_min"),
         ...
     ])
```

### 4.5 Pythonic 模式 ⚠️

```
[P2] AnalyticsEngine._calculate_ewma 使用 Python 循环
File: src/core/analytics.py:118-137
Issue: _calculate_ewma 方法使用 Python for 循环 + enumerate(reversed())

       AnalyticsEngine 中此方法是遗留代码，与重构后的向量化方向不一致。
Fix: 删除 AnalyticsEngine._calculate_ewma，

```

### 6.2 性能问题

```
[P1] SessionRepository 过早 collect() 导致全量内存加载
File: src/core/session_repository.py:48
Issue: get_sessions() 在 group_by + agg 后立即调用 .collect()，
       后续的 filter/sort/limit 操作均在内存中执行。
       随着数据量增长（多年跑步数据），内存占用将持续增加。
Fix: 保持 LazyFrame 链式操作，仅在最终返回前 collect()：
     lf = self.storage.read_parquet()
     result = (
         lf.group_by("session_start_time").agg([...])
         .filter(...)  # LazyFrame filter
         .sort(...)
         .limit(...)
     )
     return result.collect()  # 仅此处 collect
```

```
[P2] AnalyticsEngine.get_vdot_trend 使用 iter_rows 逐行计算
File: src/core/analytics.py:271
Issue: get_vdot_trend 方法使用 for row in df.iter_rows(named=True) 逐行
       调用 calculate_vdot()，而 VDOTCalculator 已提供向量化版本
       calculate_vdot_batch()。逐行调用无法利用 Polars 向量化优势。
Fix: 使用 VDOTCalculator.calculate_vdot_batch() 替代逐行调用：
     vdot_series = self.vdot_calculator.calculate_vdot_batch(df)
```
