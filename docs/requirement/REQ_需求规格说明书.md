# 需求规格说明书

## 1. 引言
### 1.1 项目背景
Nanobot Runner 是一款基于 nanobot-ai 底座的本地化 AI 跑步助理。核心目标是解决数据隐私与深度分析的矛盾，通过 Parquet 列式存储与 Polars 高性能计算引擎，为技术型跑者提供企业级 BI 能力。

## 2. 功能性需求规格

### 2.1 核心功能模块 (MVP)

#### 2.1.1 数据导入与管理 (P0)
*   **FR-001 FIT文件解析**：
    *   底层解析 `.fit` 格式，提取心率、步频、功率、轨迹等元数据。
    *   *验收标准*：成功解析 Garmin 标准格式，错误捕获率 100%，异常数据标记清晰。
*   **FR-002 高效存储与去重**：
    *   **存储格式**：采用 **.parquet** 列式存储格式。数据按“年-月”或“活动ID”分片存储，兼顾查询性能与文件管理。
    *   **指纹去重**：基于 `Serial Number + Time Created + Total Distance` 生成哈希指纹。导入前先扫描存量 Parquet 元数据，确保幂等性。
    *   *验收标准*：导入流程需支持断点续传；重复导入同一目录，数据记录零增长。

#### 2.1.2 CLI 交互模块 (P0)
*   **FR-003 命令行控制台**：
    *   `nanobotrun import <path>`：触发解析与 Parquet 落盘。
    *   `nanobotrun query "<natural language>"`：触发 Agent 进行数据分析。
*   **FR-004 性能要求**：
    *   CLI 启动速度 < 1s；百万级数据行聚合查询响应 < 3s（基于 Polars 引擎）。

#### 2.1.3 深度数据分析引擎 (P0)
*   **FR-005 计算逻辑**：
    *   **体能指标**：TSS、ATL、CTL 的滚动窗口计算。
    *   **跑力追踪**：VDOT 动态计算与历史趋势拟合。
    *   **数据清洗**：心率漂移识别算法，配速异常值过滤。
*   **FR-006 技术实现**：
    *   所有统计计算基于 **Polars** 库实现，利用其 Lazy API 进行查询优化，降低内存峰值。

### 2.2 扩展功能模块 (V1.1)
*   **FR-007 飞书集成**：基于 Webhook 的消息推送（晨报、导入通知）。

## 3. 非功能性需求
*   **NFR-001 数据隐私**：原始数据与 Parquet 文件均存储于本地，默认零外联。
*   **NFR-002 存储效率**：Parquet 格式需启用压缩，存储空间占用需比原始 CSV/JSON 减少 70% 以上。
*   **NFR-003 兼容性**：支持 Windows/macOS/Linux 三端运行。

## 4. 数据规模与性能指标
*   **数据规模**：支持管理 10,000+ 条活动记录，约 500MB - 2GB 的 Parquet 数据集。
*   **并发能力**：支持同时进行 1 路写入（导入）与 1 路读取（Agent分析），采用文件锁或写时复制策略避免冲突。
