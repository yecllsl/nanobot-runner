### 1. 核心架构评审：飞书机器人交互链路 (关键改进点)
您特别询问了飞书机器人是与 `nanobot agent` 交互还是与 `Runner Tools` 交互。
**评审意见：**
当前文档在架构设计上存在**逻辑矛盾**。
*   **架构图 (7.1)** 描绘为：`Feishu Bot` -> `RunnerTools` -> `Agent交互`。这暗示机器人先调工具，再调 Agent。
*   **代码规格 (FR-004-1)** 描绘为：`FeishuBotHandler` 同时持有 `agent_loop` 和 `runner_tools`。在处理文本消息时调用了 `self.agent.process_direct`，处理快捷指令时调用了 `self.tools.execute`。
**改进建议：**
应明确 **“飞书机器人作为 Channel（通道），Agent 作为 Brain（大脑），Tools 作为 Hands（手脚）”** 的架构原则。
**建议架构调整为：**
**飞书机器人 $\rightarrow$ Nanobot Agent $\rightarrow$ Runner Tools**
**理由：**
1.  **智能化统一入口**：用户通过飞书发送的消息（无论是自然语言还是类似 `/stats` 的指令）都应视为意图。`nanobot-ai` Agent 的核心能力是意图识别与规划。如果机器人直接调用 Tools，就绕过了 Agent 的上下文记忆和推理能力，导致“个性化”失效。
2.  **上下文一致性**：用户可能在查询完 `/stats` 后追问“为什么比上周低？”。如果 `/stats` 由机器人直接调工具返回，Agent 将丢失这次交互上下文，无法回答后续追问。
3.  **职责分离**：`FeishuBotHandler` 应只负责“协议转换”（将飞书事件转为 Agent 的输入格式，将 Agent 的输出转为飞书消息卡片），不应承担路由业务逻辑。
**修改建议：**
*   **FR-004-1 代码重构**：移除 `FeishuBotHandler` 对 `runner_tools` 的直接依赖。所有消息（包括快捷指令）统一发送给 `agent_loop`。
*   **Agent 系统提示词优化**：在 Agent 的 System Prompt 中注册指令映射，例如：“用户输入 `/stats` 时，请调用 `get_running_stats` 工具”。
*   **架构图修正**：

### 2. 功能性需求评审
#### 2.1 用户画像系统 (FR-001)
*   **数据新鲜度缺失**：规格中提到了 `last_updated`，但未定义“画像失效策略”。如果用户停止训练 1 个月，VDOT 仍然是旧值，会导致预测失真。
    *   **建议**：在 `RunnerProfile` 中增加 `stale_status` 或 `confidence_level`。若数据超过 7 天未更新，应在建议中提示“画像数据陈旧，建议重新校准”。
*   **增量更新逻辑**：`update_profile` 仅接收 `activity_data`，未说明如何处理“异常数据”（如 GPS 漂移导致的超高配速）。
    *   **建议**：增加数据清洗层，明确异常值过滤规则（如：配速 < 2:00/km 的记录不计入 VDOT 计算）。
#### 2.2 训练计划生成 (FR-002)
*   **动态调整输入缺失**：`adjust_plan` 的入参只有 `completion_rate` 和 `current_tsb`。这不足以支撑“个性化调整”。
    *   **问题**：如果用户完成了 100% 的量但心率漂移严重（强度过大），单纯看完成率会误判。
    *   **建议**：入参增加 `avg_hr_drift` 或 `rpe`（主观疲劳度）。
*   **硬编码配置**：`PHASE_CONFIG` 中将各阶段比例硬编码（如基础期 40%）。不同水平跑者的周期划分不同（新手基础期需更长）。
    *   **建议**：`generate_weekly_schedule` 应引入 `fitness_level` 参数，动态调整 `weeks_ratio`。
#### 2.3 飞书日历同步 (FR-003)
*   **双向交互闭环缺失**：文档提到了“双向交互”，但 FR-003 仅详述了 `sync_plan`（系统 -> 飞书）。如果用户在飞书日历上直接修改了训练时间（飞书 -> 系统），系统如何感知？
    *   **建议**：
        1. 明确 `subscribe_events` (FR-003-3) 的回调处理逻辑。
        2. 新增 `handle_calendar_update` 方法：当用户在飞书改时间时，更新本地 `TrainingPlan` 中的 `planned_time`，并重新计算当天的提醒逻辑。
*   **冲突处理策略**：`check_conflicts` 只返回冲突列表，未定义解决策略。
    *   **建议**：定义策略（P0/P1 需求）。是自动顺延、直接覆盖，还是发送飞书消息询问用户？（建议：低强度自动顺延，高强度/长距离跑发送卡片询问）。
#### 2.4 比赛预测 (FR-005)
*   **VDOT 插值算法风险**：`VDOT_TIME_TABLE` 仅提供了 40-50 的示例数据。实际跑者 VDOT 可能是 35 或 55，且距离是连续值。
    *   **建议**：明确说明使用 Jack Daniels 官方公式的多项式拟合算法进行插值，而非仅依赖有限 Lookup Table，以提高预测精度。

### 3. 非功能性需求与实施评审
#### 3.1 性能指标
*   **内存占用 < 800MB**：对于 Python 本地应用略显宽松。考虑到支持 10,000+ 条记录，建议优化为按需加载（Lazy Loading），将内存占用控制在 **300MB 以内** 更佳。
#### 3.2 实施计划
*   **依赖关系缺失**：`T005 飞书机器人处理器` 标注“无依赖”是不准确的。
    *   **纠正**：如果采用上述架构建议（机器人依赖 Agent），则 T005 必须依赖 `T001 用户画像`（Agent 需要画像作为 System Prompt 上下文）和 `T003 训练计划`（Agent 需要调用计划工具）。

### 4. 总结与修订建议清单
| 编号 | 问题项 | 改进建议 | 优先级 |
| :--- | :--- | :--- | :--- |
| **Arch-01** | **飞书交互架构混乱** | **确立 "Bot -> Agent -> Tools" 链路，Bot 仅做消息转发，Agent 负责所有决策与工具调用。** | **P0** |
| FR-002 | 计划调整依据不足 | 动态调整算法引入心率漂移/主观疲劳度参数。 | P1 |
| FR-003 | 缺少反向同步逻辑 | 补充“用户在飞书修改日程 -> 更新本地计划”的处理流程。 | P1 |
| FR-001 | 画像数据有效期 | 增加画像“保鲜期”机制，过期数据降权或提示更新。 | P2 |
| FR-005 | 预测算法精度 | 使用公式拟合替代查表法，覆盖更广 VDOT 区间。 | P2 |
**总体评价：**
该文档业务价值清晰，功能模块划分合理。只要修正核心交互架构（Arch-01），确保 Agent 处于核心调度地位，并补全飞书双向同步逻辑，即可进入开发阶段。建议在开发前针对 Agent 的 Prompt 工程编写一份技术设计文档，明确 Agent 如何调用新工具。
