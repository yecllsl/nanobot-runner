## 1. 项目定位

### 1.1 产品愿景

Nanobot Runner 是一款运行于个人电脑本地环境的**深度数据分析与智能训练规划中枢**。为技术型跑者提供企业级的数据洞察深度，同时确保所有隐私数据不出本地，打造专属"数字教练"。

### 1.2 核心价值主张

| 价值维度 | 描述 |
|---------|------|
| **数据隐私** | 本地存储，零外联设计，数据完全用户可控 |
| **深度分析** | 基于 Parquet + Polars 实现企业级 BI 能力 |
| **智能交互** | 自然语言交互，Agent 自动调用分析工具 |
| **多端协同** | CLI（命令行）+ 飞书双端协同 |

## 2. 目标用户画像

### 2.1 主要用户群体

#### 👨‍💻 技术极客/开发者

- **特征**: 习惯终端操作，追求数据的颗粒度与脚本化控制能力
- **痛点**: 商业跑步 App 数据封闭，无法自定义分析
- **期望**: 命令行工具、API 接口、可编程的数据处理

#### 🏃 硬核跑者

- **特征**: 关注 TSS（训练压力分数）、VDOT（跑力值）等进阶指标
- **痛点**: 需要专业分析工具，但不想上传敏感身体数据到云端
- **期望**: 深度数据分析、训练负荷管理、成绩预测

#### 🔒 隐私主义者

- **特征**: 拒绝将身体数据上传至商业云服务的用户
- **痛点**: 担心运动数据被商业利用或泄露
- **期望**: 完全本地化的数据处理与存储

## 3. 业务场景与用例

### 3.1 场景一：新设备数据迁移

**用户**: 从 Garmin 手表导出历年数据的硬核跑者

**场景描述**:
1. 用户将包含 500 个 `.fit` 文件的文件夹拖入本地目录
2. 在终端执行批量导入命令
3. 系统自动识别重复文件，只导入新数据
4. 导入完成后，用户通过自然语言查询最早的运动记录

**用户价值**:
- 一次性完成历史数据迁移
- 避免重复导入造成数据混乱
- 快速验证数据完整性

### 3.2 场景二：训练状态监控

**用户**: 正在备战马拉松的严肃跑者

**场景描述**:
1. 用户每日通过 CLI 导入新的跑步记录
2. 系统自动计算 TSS、ATL、CTL 等指标
3. 用户询问 Agent："我现在的训练状态如何？"
4. Agent 返回训练负荷趋势图和疲劳度评估
5. 飞书每日推送晨报，提醒今日训练建议

**用户价值**:
- 实时掌握训练负荷与恢复状态
- 科学规划训练强度，避免过度训练
- 预防运动损伤

### 3.3 场景三：跑步表现分析

**用户**: 追求 PB（个人最好成绩）的进阶跑者

**场景描述**:
1. 用户完成一次长距离训练后导入数据
2. 询问 Agent："分析这次跑步的心率漂移"
3. Agent 分析心率曲线，识别有氧能力变化
4. 用户查询 VDOT 趋势，了解跑力值变化
5. 系统预测全马完赛时间

**用户价值**:
- 深度了解自身运动表现
- 科学评估训练效果
- 制定合理的成绩目标

### 3.4 场景四：数据资产管理

**用户**: 重视数据隐私的技术用户

**场景描述**:
1. 用户确认所有数据存储在本地目录
2. 通过 CLI 导出数据备份
3. 配置飞书 Webhook，仅发送摘要消息
4. 原始 `.fit` 文件绝不离开本地环境

**用户价值**:
- 完全掌控个人运动数据
- 避免隐私泄露风险
- 支持数据导出和迁移

## 4. 功能期望

### 4.1 数据导入与管理

| 功能 | 用户期望 | 业务价值 |
|------|---------|----------|
| **FIT 文件导入** | 支持 Garmin、Wahoo 等设备导出的标准 FIT 文件 | 兼容主流运动设备 |
| **批量导入** | 支持目录批量扫描和导入 | 提高数据迁移效率 |
| **智能去重** | 自动识别重复文件，避免数据污染 | 保证数据仓库清洁 |
| **增量更新** | 多次导入同一目录，只处理新文件 | 支持持续数据同步 |

### 4.2 CLI 交互体验

| 功能 | 用户期望 | 业务价值 |
|------|---------|----------|
| **命令行操作** | 熟悉的终端交互方式 | 符合技术用户习惯 |
| **自然语言查询** | 直接输入问题，无需记忆命令 | 降低使用门槛 |
| **快速响应** | 命令秒级响应，查询结果即时返回 | 流畅的使用体验 |
| **报告生成** | 支持导出 PDF/HTML 分析报告 | 便于分享和存档 |

### 4.3 数据分析能力

| 功能 | 用户期望 | 业务价值 |
|------|---------|----------|
| **体能指标** | TSS、ATL、CTL、TSB 等进阶指标 | 科学评估训练状态 |
| **VDOT 追踪** | 跑力值动态计算和趋势分析 | 量化有氧能力提升 |
| **心率漂移** | 识别有氧能力变化和疲劳信号 | 预防过度训练 |
| **成绩预测** | 基于 VDOT 预测全马/半马成绩 | 制定合理比赛目标 |

### 4.4 飞书集成

| 功能 | 用户期望 | 业务价值 |
|------|---------|----------|
| **每日晨报** | 定时推送训练建议和身体状态 | 养成科学训练习惯 |
| **导入通知** | 批量导入完成后发送汇总消息 | 及时了解数据处理状态 |
| **轻量查询** | 在飞书中快速查询关键指标 | 移动端便捷访问 |

## 6. 用户旅程地图

### 6.1 首次使用旅程

```
发现产品 → 安装配置 → 首次导入 → 数据验证 → 探索功能 → 持续使用
   ↓           ↓           ↓           ↓           ↓           ↓
 了解功能    环境准备    批量导入    查询确认    深度分析    日常管理
```

**关键触点**:
1. **发现**: 通过 README 了解产品定位
2. **安装**: 使用 uv 快速安装依赖
3. **导入**: 执行 `nanobotrun import` 完成首次数据迁移
4. **验证**: 通过自然语言查询确认数据完整性
5. **探索**: 尝试 VDOT 计算、心率漂移分析等功能
6. **持续**: 每日导入新数据，飞书接收晨报

### 6.2 日常使用旅程

```
完成跑步 → 导入数据 → 查看统计 → 分析表现 → 调整计划
   ↓           ↓           ↓           ↓           ↓
 设备同步    自动去重    即时反馈    深度洞察    科学训练
```

## 7. 竞品分析

### 7.1 市场定位对比

| 产品 | 定位 | 优势 | 劣势 |
|------|------|------|------|
| **TrainingPeaks** | 专业训练平台 | 功能全面，生态成熟 | 云端存储，订阅费用高 |
| **Strava** | 社交跑步平台 | 社区活跃，分享便捷 | 隐私担忧，分析深度有限 |
| **Garmin Connect** | 设备配套软件 | 与设备深度集成 | 数据封闭，分析能力一般 |
| **Nanobot Runner** | 本地 AI 助理 | 隐私优先，深度分析，开源 | 需技术背景，生态初期 |

### 7.2 差异化优势

1. **隐私优先**: 唯一完全本地化的专业跑步分析工具
2. **技术开放**: 开源，支持自定义分析和扩展
3. **AI 驱动**: Agent 自然语言交互，降低使用门槛
4. **企业级分析**: Parquet + Polars 提供 BI 级数据处理能力

## 9. 风险与假设

### 9.1 风险识别

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 技术门槛高 | 限制用户群体 | 提供详细的 CLI 使用指南 |
| 设备兼容性 | 部分 FIT 文件解析失败 | 持续完善解析器，错误友好提示 |
| 用户习惯 | 用户习惯商业 App | 强调隐私优势和深度分析能力 |

### 9.2 关键假设

1. 目标用户具备基础的命令行操作能力
2. 用户愿意为了隐私牺牲一定的便利性
3. 用户有持续记录跑步数据的习惯
