## 2. 整改明细

### 2.1 CRITICAL 级问题

| 编号 | 问题 | 整改状态 | 整改说明 |
|------|------|---------|---------|
| C-01 | 存储格式 JSONL → Parquet | ✅ 已修复 | 重写 EvolutionStore，实现 Parquet 按月分片存储 |

**C-01 整改详情**:

- 重写 `evolution_store.py`（约 630 行），移除全部 JSONL 逻辑
- 定义 `_DECISIONS_PARQUET_SCHEMA`（10 字段）和 `_OUTCOMES_PARQUET_SCHEMA`（11 字段），与架构文档一致
- 存储路径: `{data_dir}/decisions/YYYY-MM/decisions_YYYY-MM.parquet` 和 `{data_dir}/outcomes/YYYY-MM/outcomes_YYYY-MM.parquet`
- `_append_to_parquet()`: 原子写入（临时文件 + replace），确保数据安全
- `query_decisions()`: 使用 `pl.scan_parquet()` → `filter()` → `sort()` → `limit()` → `collect()`（LazyFrame 优先）
- `query_outcomes()`: 同上，LazyFrame 管道查询
- `_get_parquet_files_in_range()`: 根据日期范围裁剪分片文件，避免全量扫描
- `update_decision()`: 读取 → 替换匹配行 → 原子写入
- `get_decision_outcome_pairs()`: LazyFrame 扫描 + 内存关联
- 复杂字段（runner_state、tool_call_chain、prediction_snapshot）使用 JSON 字符串存储，读写时自动序列化/反序列化

### 2.2 HIGH 级问题

| 编号 | 问题 | 整改状态 | 整改说明 |
|------|------|---------|---------|
| H-01 | runner_state 全 None → 注入 TwinEngine | ✅ 已修复 | DecisionLogHook 注入 DigitalTwinEngine |
| H-02 | 全量加载 → LazyFrame 查询 | ✅ 随 C-01 解决 | Parquet + scan_parquet 实现 |
| H-03 | update 竞态 → 原子写入 | ✅ 随 C-01 解决 | 临时文件 + replace 原子写入 |
| H-04 | PlanExecutionDataAdapter NotImplementedError | ✅ 已修复 | 实现真实逻辑 |

**H-01 整改详情**:

- `DecisionLogHook.__init__()` 新增 `twin_engine: DigitalTwinEngine | None = None` 参数
- `_build_runner_state()` 优先调用 `twin_engine.get_current_snapshot()` 获取 5 维度实际数值
- 提取字段: `vdot`, `vdot_trend`, `ctl`, `atl`, `tsb`, `acwr`, `fatigue_score`, `recovery_status`, `injury_risk_7d`, `injury_risk_28d`, `overtraining_risk`, `weekly_volume_km`, `long_run_frequency`, `snapshot_date`, `data_quality`
- 异常时降级为字段名 + None 值，日志记录 warning

**H-04 整改详情**:

- 实现 `PlanExecutionDataAdapter.get_execution_data()` 真实逻辑
- 从 `PlanManager.get_plan()` 获取计划详情，计算计划跑量和时长
- 从 `PlanExecutionRepository.get_plan_execution_stats()` 获取执行统计
- 异常时降级为 0 值，日志记录 warning
