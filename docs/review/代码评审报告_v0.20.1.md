### [CRITICAL-1] VDOTPredictor 训练/推理特征维度不匹配，导致 ML 推理运行时 ValueError

**文件**: [vdot_predictor.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.20.1/src/core/prediction/vdot_predictor.py) 第 188-233 行（训练）vs 第 375 行（推理）

**问题描述**:

`_build_training_data()` 方法自行构建 **5 个特征**：
```python
# vdot_predictor.py:188-233 — 训练特征（5维）
features_list.append([dist_km, dur_min, hr_val, day_of_week, month])
# 特征: [distance_km, duration_min, avg_hr, day_of_week, month]
```

但 `_run_ml_inference()` 使用 `FeatureEngine.extract_vdot_features()` 返回 **12 个特征**：
```python
# vdot_predictor.py:375 — 推理特征（12维）
matrix = self._feature_engine.extract_vdot_features(days=days)
# 特征: [weekly_volume_km, volume_change_rate, sin_month, cos_month,
#        ctl_value, tsb_value, atl_ctl_ratio, load_ramp_rate,
#        high_intensity_pct, avg_intensity_factor, fatigue_score,
#        resting_hr_deviation]
```

**影响**: 模型训练时 `n_features_in_=5`，推理时传入 12 维特征，`model.predict()` 将抛出 `ValueError: X has 12 features, but GradientBoostingRegressor is expecting 5 features as input`。

**根因**: `_build_training_data()` 未使用 `FeatureEngine` 提取训练特征，而是自行构建了简化的 5 维特征向量。

**修复方案**:
1. `_build_training_data()` 应调用 `self._feature_engine.extract_vdot_features()` 为每个训练样本提取 12 维特征
2. 或修改 `_run_ml_inference()` 使用与训练一致的 5 维特征提取逻辑
3. **推荐方案1**：使用 FeatureEngine 统一训练和推理的特征提取，确保特征维度一致

**验证方法**: 编写集成测试，完整走通 train_model() → save_model() → load_model() → predict() 全链路

### [CRITICAL-2] InjuryPredictor 训练/推理特征语义不匹配，导致 ML 推理结果无意义

**文件**: [injury_predictor.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.20.1/src/core/prediction/injury_predictor.py) 第 243-288 行（训练）vs 第 358 行（推理）

**问题描述**:

`_build_training_data()` 构建 **8 个特征**（基于单次 session 的原始指标）：
```python
# injury_predictor.py:243-288 — 训练特征（8维）
feat = np.array([
    float(tss),                                    # 0: tss
    float(tss) / max(float(duration_s), 1.0) * 3600, # 1: intensity
    float(distance_m) / 1000.0,                     # 2: distance_km
    float(avg_hr),                                  # 3: avg_hr
    float(max_hr) - float(avg_hr),                  # 4: hr_reserve
    float(elevation_gain),                          # 5: elevation_gain
    float(duration_s) / 3600.0,                     # 6: duration_hours
    float(tss) * 0.3,                               # 7: load_proxy
])
```

但 `_run_ml_inference()` 使用 `FeatureEngine.extract_injury_features()` 返回 **8 个完全不同语义的特征**：
```python
# feature_engine.py:148-189 — 推理特征（8维，语义完全不同）
features = np.array([
    atl_ctl_ratio,              # 0: 急慢性负荷比
    weekly_load_change_pct,     # 1: 周负荷变化率
    tsb_consecutive_low_days,   # 2: TSB连续低位天数
    tsb_trend_slope,            # 3: TSB趋势斜率
    resting_hr_deviation_pct,   # 4: 静息心率偏离百分比
    resting_hr_7d_trend,        # 5: 静息心率7日趋势
    hrv_rmssd_trend,            # 6: HRV RMSSD趋势
    hrv_sdnn_deviation,         # 7: HRV SDNN偏离
])
```

**影响**: 虽然维度数相同（都是 8），但特征语义完全不同。模型在训练时学习的是"单次 session 原始指标 → 伤病风险"的映射，推理时却输入"负荷比/HRV/心率趋势"特征，预测结果完全无意义。

**根因**: 同 CRITICAL-1，`_build_training_data()` 未使用 `FeatureEngine` 统一特征提取。

**修复方案**:
1. `_build_training_data()` 应调用 `self._feature_engine.extract_injury_features()` 为每个训练样本提取 8 维特征
2. 需要为每个历史 session 构造对应时间点的特征上下文（如该 session 日期前的 ATL/CTL/TSB/HRV 状态）
3. **推荐方案1**：统一使用 FeatureEngine，训练时按 session 日期回溯计算特征

**验证方法**: 同 CRITICAL-1，编写全链路集成测试

| [training_response_predictor.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.20.1/src/core/prediction/training_response_predictor.py) | 149, 160 | `_get_tsb()`, `_get_weekly_tss()` |
| [vdot_predictor.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.20.1/src/core/prediction/vdot_predictor.py) | 422 | `_run_ml_inference()` 趋势斜率计算 |

**问题描述**: 所有 10 处均使用 `except Exception: pass` 模式，违反项目编码规范"禁止裸 Exception、禁止静默吞没异常"。虽然每处都有默认返回值作为降级策略，但异常被完全静默，问题排查困难。

**修复方案**: 将 `pass` 替换为 `logger.debug(f"获取xxx失败，使用默认值: {e}")`，使用 DEBUG 级别避免干扰正常日志输出。

### [HIGH-2] context.py 新增属性返回类型使用 `Any`，降低类型安全性

**文件**: [context.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.20.1/src/core/base/context.py) 第 296-350 行

**问题描述**: 4 个新增 lazy property 的返回类型均为 `Any`：
```python
@property
def training_load_analyzer(self) -> Any:  # 应为 TrainingLoadAnalyzer
    ...

@property
def vdot_calculator(self) -> Any:  # 应为 VDOTCalculator
    ...

@property
def race_prediction_engine(self) -> Any:  # 应为 RacePredictionEngine
    ...

@property
def injury_risk_analyzer(self) -> Any:  # 应为 InjuryRiskAnalyzer
    ...
```

**影响**: 架构要求核心模块类型注解覆盖率 ≥ 80%，使用 `Any` 降低了 IDE 自动补全和类型检查的有效性。

**修复方案**: 使用 `TYPE_CHECKING` 延迟导入具体类型：
```python
from __future__ import annotations
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from src.core.calculators.training_load_analyzer import TrainingLoadAnalyzer
    from src.core.calculators.vdot_calculator import VDOTCalculator
    from src.core.calculators.race_prediction import RacePredictionEngine
    from src.core.calculators.injury_risk_analyzer import InjuryRiskAnalyzer
```

### [MEDIUM-1] ModelManager 中 Windows symlink 兼容性风险

**文件**: [model_manager.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.20.1/src/core/prediction/model_manager.py) 第 218-220, 256-258 行

**问题描述**: `save_model()` 和 `rollback()` 方法使用 `Path.symlink_to()` 创建 `current → version_dir` 的符号链接。在 Windows 上，创建符号链接默认需要管理员权限或开启开发者模式，否则抛出 `OSError`。

```python
# model_manager.py:218-220
current_link = model_dir / "current"
if current_link.exists() or current_link.is_symlink():
    current_link.unlink()
with contextlib.suppress(OSError):  # 静默忽略失败
    current_link.symlink_to(version_dir, target_is_directory=True)
```

**影响**: 虽然使用了 `contextlib.suppress(OSError)` 兜底，但 symlink 创建失败后 `current` 指向丢失，`load_model()` 将无法找到当前版本。

**修复方案**: symlink 失败时，降级为写入一个 `current_version.txt` 文本文件记录当前版本号，`load_model()` 优先读 symlink，失败则读文本文件。
