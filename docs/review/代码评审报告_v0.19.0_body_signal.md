#### C1. [CRITICAL] `get_recovery_trend()` 传入空 DataFrame 导致每行数据相同
[recovery_monitor.py:L188-L196](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/src/core/body_signal/recovery_monitor.py#L188-L196)

```python
for row in df.iter_rows(named=True):

```

**问题描述**：在遍历每一行时，`calculate_training_load_from_dataframe` 始终接收空的 `pl.DataFrame()`，导致每次循环都返回相同的全局 TSB/CTL 值。所有 `RecoveryPoint` 的 `tsb` 和 `ctl` 完全相同，使恢复趋势数据完全无意义。

**影响**：恢复趋势功能完全失效 —— `get_recovery_trend()` 返回值不反映按日的真实恢复变化。进而影响 `check_alerts()` 中的过度训练检测和 `compare_training_periods()` 中的周期对比。

**修复建议**：`calculate_training_load_from_dataframe` 应接收按日期过滤后的数据；或者直接使用已有的 `TrainingLoadAnalyzer` 获取最新的全局 TSB/CTL，循环内仅按日期记录。

#### C2. [CRITICAL] `get_resting_hr_trend()` 中死代码条件永远不成立
[hrv_analyzer.py:L108-L120](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/src/core/body_signal/hrv_analyzer.py#L108-L120)

```python
trend: list[RestingHRPoint] = []  # L108: trend 初始化为 []
```

**问题描述**：`trend` 在第 108 行初始化为空列表 `[]`，后续没有任何向其添加元素的操作，因此 `len(trend)` 在第 112 行始终为 0，条件永远不成立。应检查的是 `len(resting_hrs) == 1`（数据点数量）。

**影响**：虽然单数据点场景下 fallthrough 代码也能正确计算出 `deviation_pct=0.0`（因为 mean_hr == hr），但死代码是代码质量问题，且如果后续逻辑变更，此分支的行为将与预期不符。

#### C3. [CRITICAL] `analyze_hr_recovery()` 和 `check_hr_drift()` 分析范围错误
[hrv_analyzer.py:L141-L206](file:///d:/yecll/Documents/LocalCode/RunFlowAgent/src/core/body_signal/hrv_analyzer.py#L141-L206)

**问题描述**：
- `analyze_hr_recovery()` 加载全部 Parquet 数据（不按时间过滤），取全局最后一条记录的 `heart_rate` 返回为 `hr_end`。心率恢复分析应该是针对**单次训练后**的心率下降速率（如训练结束后 1 分钟、2 分钟的心率差值），不应是全局最后一条心率值的简单查询。
- `check_hr_drift()` 同样加载全部数据，用全局第一条和最后一条心率的差值计算漂移率。心率漂移应针对**单次训练中恒定配速下**的心率上升趋势，而非跨所有训练记录。

**影响**：`hr-recovery` 和 `hr-drift` 两个 CLI 命令的输出结果无实际业务意义，给用户提供的是错误的分析结论。

**修复建议**：这两个方法需要按 `session_id` 分组或按 `session_start_time` 过滤，分析单次训练内的心率数据。实现此类分析需要读取原始时序心率数据（可能来自 FIT 文件的 record 消息），而非聚合后的 Parquet 数据。
