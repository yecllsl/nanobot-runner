## 4. 代码质量评审

### 4.1 优点

1. **方法提取得当**: 将飞书通道和 WebSocket 通道配置构建分别提取为 `_build_feishu_channel_config()` 和 `_build_websocket_channel_config()`，职责清晰，符合单一职责原则
2. **类型注解完整**: 所有新增公共方法均有完整的类型注解，包括参数和返回值
3. **防御性编程**: `get_websocket_config()` 对 websocket 配置节类型异常做了防御（非 dict 类型返回空 dict），浅拷贝避免修改原始配置
4. **常量提取**: `WS_ENV_KEY_MAPPING`、`BOOL_KEYS`、`INT_KEYS` 提取为模块级常量，复用 `_cast_env_value()` 类型转换逻辑
5. **keyword-only 参数**: `webui_enabled` 使用 keyword-only 参数（`*` 分隔），避免位置参数误用
6. **文档字符串完整**: 新增方法均有中文文档字符串，说明参数、返回值和行为

### 4.2 问题清单

```
[Important] ruff format 格式化未通过
File: src/core/config/manager.py, src/core/provider_adapter.py, src/cli/commands/gateway.py
Issue: ruff format --check 报告 3 个文件需要格式化，代码格式不符合项目规范
Fix: 运行 uv run ruff format src/core/config/manager.py src/core/provider_adapter.py src/cli/commands/gateway.py
```

```
[Important] AppConfig Schema 未包含 websocket 字段定义
File: src/core/config/schema.py
Issue: AppConfig.FIELD_TYPES 和 AppConfig 属性中未定义 websocket 字段。
       虽然当前 validate() 方法对未知字段采取忽略策略（不报错），
       但这意味着 config.json 中的 websocket 配置节不会被 Schema 验证。
       如果用户在 websocket 节中写了错误的类型（如 port: "abc"），
       AppConfig.validate() 不会捕获。
       不过，_cast_env_value() 对 port 的 int 转换有 ValueError 回退，
       且 WebSocket 配置最终由 nanobot-ai 的 WebSocketConfig 校验，
       所以实际风险可控。
Fix: 建议在 AppConfig 中新增 websocket: dict[str, Any] | None = None 字段，
     并在 FIELD_TYPES 中添加对应类型定义，确保配置验证的完整性。
     优先级为 Important（非阻塞），可在后续迭代中补充。
```

```
[Minor] load_config_with_env_override() 与 get_websocket_config() 存在 WebSocket 环境变量覆盖的重复逻辑
File: src/core/config/manager.py:345-356 vs :454-478
Issue: load_config_with_env_override() 和 get_websocket_config() 都独立实现了
       WebSocket 环境变量覆盖逻辑。两处代码结构相似但独立维护，
       如果 WS_ENV_KEY_MAPPING 发生变更，需要同步修改两处。
Fix: 建议让 load_config_with_env_override() 在处理 WebSocket 环境变量时
     复用 get_websocket_config() 的逻辑，或将 WebSocket 环境变量覆盖
     提取为独立的私有方法 _apply_ws_env_override()。
```

```
[Minor] gateway.py 中 WebUI 信息显示重复调用 get_websocket_config()
File: src/cli/commands/gateway.py:383-398 vs :432-443
Issue: 在 channels.enabled_channels 判断和 webui 交互信息区块中，
       两次调用 context.config.get_websocket_config()。
       虽然 get_websocket_config() 内部有缓存机制（load_config 有缓存），
       但仍存在冗余调用。
Fix: 建议在首次调用后将 ws_config 赋值给局部变量，后续复用。
```

```
[Minor] _build_websocket_channel_config 中 token_issue_path 默认值与架构设计不一致
File: src/core/provider_adapter.py:331
Issue: 架构设计说明书 10.5.1 节定义 token_issue_path 默认值为 "/token"，
       但 _build_websocket_channel_config() 中默认值为空字符串 ""。
       config.example.json 中也为空字符串 ""。
       这可能导致用户未配置时，token 签发端点不生效。
Fix: 将 ws_config.get("token_issue_path", "") 改为
     ws_config.get("token_issue_path", "/token")，
     与架构设计保持一致。同时更新 config.example.json 中的默认值。
```

```
[Minor] config.example.json 中使用 _comment 后缀作为注释
File: config.example.json:45-79
Issue: JSON 标准不支持注释，使用 _comment 后缀字段是一种常见变通方案，
       但这些额外字段会被 JSON 解析器读取为合法配置项，
       可能被传递到 nanobot-ai 的 WebSocketConfig 中导致意外行为。
Fix: 当前方案可接受（nanobot-ai 的 WebSocketConfig 应忽略未知字段），
     但建议在文档中明确说明 _comment 字段不会被程序使用，
     或改用 JSONC 格式（如 .jsonc 扩展名）。
```

```
[Minor] gateway.py 中条件判断过长
File: src/cli/commands/gateway.py:431
Issue: `if webui and channels.enabled_channels and "websocket" in channels.enabled_channels and context is not None:`
       单行条件判断包含 4 个 and 条件，可读性不佳。
Fix: 将条件拆分为多行或提取为语义化的局部变量：
     ws_channel_active = (webui and channels.enabled_channels
                          and "websocket" in channels.enabled_channels
                          and context is not None)
```

Issue: 当 websocket_requires_token=True 但 token_issue_path="" 时，
