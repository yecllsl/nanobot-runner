## 3. 问题清单

### 3.1 CRITICAL 级问题

---

[CRITICAL] **存储格式偏离架构设计：JSONL → 应使用 Parquet**

- **文件**: [evolution_store.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.23.0/src/core/evolution/evolution_store.py#L1-L5) (全文件)
- **Issue**: 架构设计说明书 Section 8.2.5 明确规定 EvolutionStore 使用 **Parquet 按月分片存储**，Parquet Schema 在架构文档中已详细定义（decisions 表 10 字段、outcomes 表 11 字段）。实际实现使用了 **JSONL（JSON Lines）** 格式，存储路径也完全不同：
  - 架构设计路径: `~/.nanobot-runner/decisions/YYYY-MM/decisions_YYYY-MM.parquet`
  - 实际代码路径: `data_dir/evolution/decisions/{year}/{year}-{month}.jsonl`
  
  这不仅仅是格式变更，更涉及：
  1. **违反项目核心约束**：AGENTS.md 明确 "存储: Parquet按年分片"，pyproject.toml 技术栈含 Polars+pyarrow
  2. **丧失 LazyFrame 查询能力**：JSONL 需要全量 `_load_all_decisions()` 加载到内存再过滤，与 "LazyFrame 优先" 铁律直接冲突
  3. **丧失列式存储优势**：无法按列过滤、无法利用 Polars 的谓词下推和分区裁剪
  4. **决策日志查询性能退化**：1000 条记录时全量 JSON 解析 → 全表扫描，Parquet 方案只需扫描目标分片文件
  5. **update_execution_status 实现脆弱**：`decision_logger.py:108-124` 通过逐行读写 JSONL 文件更新单条记录（读全文件 → 修改一行 → 重写全文件），不是原子操作，存在数据丢失风险

- **Fix**: 
  1. 将 `EvolutionStore` 重写为基于 Polars Parquet 的存储实现
  2. 文件路径改为 `{data_dir}/decisions/YYYY-MM/decisions_YYYY-MM.parquet`
  3. `save_decision()`: 使用 `pl.DataFrame([row]).write_parquet()` 追加模式
  4. `query_decisions()`: 使用 `pl.scan_parquet()` → `filter()` → `sort()` → `limit()` → `collect()`（LazyFrame 优先）
  5. `update_execution_status()`: 通过 Polars 的 `with_columns` + `when/then` 原地更新
  6. 同步更新 `decision_logger.py` 和 `evolution_engine.py` 中所有引用 JSONL 路径和逻辑的代码

### 3.2 HIGH 级问题

---

[HIGH] **DecisionLogHook._build_runner_state() 返回全部 None 值，runner_state 快照形同虚设**

- **文件**: [decision_log_hook.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.23.0/src/core/evolution/decision_log_hook.py#L196-L213)
- **Issue**: `_build_runner_state()` 方法从配置获取字段名列表后，将所有字段值设为 `None`。这意味着每一条 DecisionLog 的 `runner_state` 都是 `{"vdot": None, "ctl": None, "atl": None, "tsb": None, "fatigue_score": None}`。架构设计说明书 Section 8.2.4 明确要求 runner_state 存储 5 个关键维度的**实际数值**，这些数据是 v0.24 训练响应性分析的输入基础。当前实现使得决策日志的 runner_state 完全无法用于后续分析。

- **Fix**:
  1. 在 `DecisionLogHook.__init__()` 中注入 `DigitalTwinEngine` 实例（通过 `evolution_engine` 传递或构造函数注入）
  2. 在 `before_iteration()` 中调用 `twin_engine.get_current_snapshot()` 获取实际 RunnerStateVector
  3. 提取 5 个关键字段的实际值：`vdot`, `ctl`, `atl`, `tsb`, `fatigue_score`
  4. 异常时降级为空 dict（而非全 None dict），并在日志中记录 warning

---

[HIGH] **EvolutionStore 查询实现全量加载，违反 LazyFrame 优先原则**

- **文件**: [evolution_store.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.23.0/src/core/evolution/evolution_store.py#L213-L247)
- **Issue**: `query_decisions()` 和 `query_outcomes()` 均调用 `_load_all_decisions()` / `_load_all_outcomes()` 加载**全部历史数据**到内存，然后逐条过滤。这与项目 "Polars LazyFrame 优先，仅在最终输出时 collect()" 的铁律直接冲突。随着决策日志积累（预计 ~500条/年），每次查询都全量加载会持续退化。

- **Fix**: 切换到 Parquet 后自然解决此问题（使用 `pl.scan_parquet()` 的 LazyFrame 管道）。如果短期内必须保持 JSONL，至少应按年月分片加载（根据 `start_date`/`end_date` 确定需要扫描的月份文件），而非遍历全部 `*.jsonl`。

---

[HIGH] **DecisionLogger.update_execution_status() 存在数据竞态和原子性风险**

- **文件**: [decision_logger.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.23.0/src/core/evolution/decision_logger.py#L93-L127)
- **Issue**: `update_execution_status()` 采用 "读全文件 → 修改一行 → 写全文件" 策略：
  1. 非原子操作：如果在写入过程中进程崩溃，整个月份文件可能损坏
  2. 并发不安全：如果两个请求同时更新不同 decision_id，后写入的会覆盖先写入的变更
  3. 违反封装：`_get_decisions_file_path()` 直接访问 `self._store._data_dir`（私有属性），破坏 EvolutionStore 的封装

- **Fix**:
  1. 切换到 Parquet 后，使用 Polars 的 `read_parquet()` → `with_columns(when().then())` → `write_parquet()` 实现原地更新
  2. 短期如果保持 JSONL，使用原子写入模式：写临时文件 → `os.replace()` 原子替换
  3. 将 `_get_decisions_file_path()` 提升为 EvolutionStore 的公共方法或移除对私有属性的访问

---

[HIGH] **PlanExecutionDataAdapter.get_execution_data() 为 NotImplementedError 占位实现**

- **文件**: [outcome_collector.py](file:///d:/yecll/Documents/LocalCode/RunFlowAgent.worktrees/feature-0.23.0/src/core/evolution/outcome_collector.py#L70-L75)
- **Issue**: `PlanExecutionDataAdapter.get_execution_data()` 直接 `raise NotImplementedError`，注释为 "当前为占位实现，由子类或Mock覆盖"。这意味着 `check_plan_execution()` 的核心功能（计算执行忠实度）在真实环境中无法工作——`fidelity` 始终为 `None`。架构设计说明书 Section 8.2.5 明确要求该适配器封装 PlanManager + PlanExecutionRepository 的调用。

- **Fix**:
  1. 实现 `get_execution_data()` 的真实逻辑：从 plan_manager 获取计划数据，从 execution_repo 获取执行统计
  2. 映射为 `PlanExecutionData`：`recommended_volume`, `actual_volume`, `recommended_duration`, `actual_duration`
  3. 如果当前 PlanManager/PlanExecutionRepository API 不足以支持批量查询，在代码中标注 TODO 并记录技术债务
