# 协作链路

本文档定义智能体协作流程、准入准出规则和异常处理机制。

---

## 1. 协作链路总览

```
产品及项目经理（产品规划→需求管理→项目管理→进度跟踪）→ 
架构师（需求分析→架构设计→任务拆解→架构评审）→ 
开发工程师（功能开发→单元测试→本地验证）→ 
代码评审工程师（代码评审）→ 
测试工程师（测试策略→测试执行→Bug反馈）→ 
开发工程师（Bug修复）→ 
测试工程师（回归测试→上线结论）→ 
发布运维（CICD验证→自动化发布）
```

---

## 2. 分阶段协作规范

### 阶段0：产品规划与项目管理（产品及项目经理主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 |
|------|------|------|------|----------|
| 产品及项目经理 | PM-01 产品规划 | 市场调研、业务目标 | `/docs/product/产品规划方案.md` | 产品愿景清晰，路线图合理 |
| 产品及项目经理 | PM-02 需求管理 | 需求来源 | `/docs/product/需求池.md` + `/docs/product/需求分析报告.md` | 需求无歧义，优先级合理 |
| 产品及项目经理 | PM-03 项目管理 | 任务清单、资源配置 | `/docs/planning/项目计划.md` + `/docs/planning/风险管理计划.md` | 资源分配均衡，风险识别全面 |
| 产品及项目经理 | PM-04 进度跟踪 | 项目计划、任务状态 | `/docs/planning/进度报告_*.md` + 项目看板 | 进度数据准确，偏差分析深入 |

### 阶段1：需求与架构（架构师主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 |
|------|------|------|------|----------|
| 架构师 | ARC-01 需求分析 | 业务诉求 | `/docs/requirements/REQ_*.md` | 需求无歧义，验收标准可量化 |
| 架构师 | ARC-02 架构设计 | 需求规格 | `/docs/architecture/架构设计说明书.md` | 技术栈适配，接口规范完整 |
| 架构师 | ARC-03 任务拆解 | 架构方案 | `/docs/planning/task_list_*.md` | 任务粒度适配迭代周期 |
| 架构师 | ARC-04 架构评审 | 架构方案、需求规格 | `/docs/architecture/架构评审报告.md` | 技术可行性验证，风险识别完整 |

### 阶段2：开发实现（开发工程师主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 |
|------|------|------|------|----------|
| 开发工程师 | DEV-01 功能开发 | 任务清单、架构方案 | 代码 + `/tests/unit/` + `/docs/development/交付报告_*.md` | 覆盖率≥80%，测试通过率100% |
| 代码评审工程师 | REV-01 代码评审 | 代码仓库、架构方案 | `/docs/review/代码评审报告_*.md` | 核心代码无架构偏离，代码规范达标 |

### 阶段3：测试验证（测试工程师主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 |
|------|------|------|------|----------|
| 测试工程师 | TST-01 测试策略 | 需求规格、架构方案 | `/docs/test/strategy_*.md` | 覆盖MVP需求 |
| 测试工程师 | TST-02 测试执行 | 代码仓库、测试用例 | `/docs/test/测试报告_*.md` + `/docs/test/Bug清单_*.md` | 核心流程通过率≥95% |
| 开发工程师 | DEV-02 Bug修复 | Bug清单 | 修复代码 + `/docs/development/Bug修复报告_*.md` | P0/P1优先修复 |
| 测试工程师 | TST-03 回归测试 | Bug修复报告 | `/docs/test/回归报告_*.md` + `/docs/test/上线结论_*.md` | P0/P1 Bug 100%验证 |

### 阶段4：自动化发布（发布运维主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 |
|------|------|------|------|----------|
| 发布运维 | OPS-01 CICD验证 | Git分支、构建命令 | `/docs/devops/流水线执行报告_*.md` | 流水线无报错 |
| 发布运维 | OPS-02 自动化发布 | 上线结论、版本号 | 版本Tag + `/docs/devops/发布报告_*.md` | 必须有上线结论 |

## 3. 准入准出规则

### 3.1 架构准入

- [ ] 产品规划方案已确认
- [ ] 需求池已建立并优先级排序
- [ ] 项目计划已制定

### 3.2 开发准入

- [ ] 需求规格说明书已确认
- [ ] 架构设计方案已确认
- [ ] 任务清单已拆解
- [ ] 架构评审通过（ARC-04）

### 3.3 测试准入

- [ ] 开发交付报告已提交
- [ ] 单元测试覆盖率≥80%
- [ ] 本地核心场景验证通过
- [ ] 代码评审通过（REV-01）

### 3.4 发布准入

- [ ] 测试上线结论为"建议上线"
- [ ] P0/P1级Bug 100%修复验证
- [ ] CICD流水线验证通过
- [ ] 版本号已更新

### 3.5 发布准出

- [ ] 版本Tag创建成功
- [ ] GitHub Release发布成功
- [ ] 线上服务验证通过

## 4. 异常处理

| 异常场景 | 处理方式 | 责任人 |
|---------|---------|--------|
| 产品规划方向不明确 | 召开产品评审会议，重新定义产品愿景 | 产品及项目经理 |
| 项目进度延期超过20% | 召开项目复盘会议，调整项目计划 | 产品及项目经理 |
| 本地调试多次失败 | 输出调试报告，同步架构师排查 | 开发工程师 |
| CICD流水线连续失败 | 同步架构师、测试工程师共同分析 | 发布运维 |
| 架构评审不通过 | 按评审报告整改后重新提交 | 架构师 |
| 代码评审不通过 | 按评审报告整改后重新提交 | 开发工程师 |
| 测试通过率<95% | 优先修复失败用例，达标后进入发布 | 开发工程师 |

## 5. 时效规则

| 场景 | 响应时效 | 处理时效 |
|------|---------|---------|
| 产品方向重大变更 | 立即 | 1工作日内调整规划 |
| 项目进度偏差超过10% | 4工时内 | 1工作日内调整计划 |
| 架构评审反馈 | 2工时内 | 1工作日内整改 |
| 代码评审反馈 | 2工时内 | 4工时内整改 |
| 测试反馈P0/P1级Bug | 2工时内 | 4工时内修复 |
| CICD流水线失败 | 1工时内 | 2工时内修复 |
| 安全测试高危漏洞 | 立即 | 4工时内修复 |
