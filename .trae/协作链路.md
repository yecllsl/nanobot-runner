# 协作链路

本文档定义智能体协作流程、准入准出规则和异常处理机制，并明确全局Skill在各阶段的嵌入方式与协作模式。

***

## 1. 协作链路总览

### 1.1 项目Skill主链路

```
产品及项目经理（产品规划）→ 
架构师（需求分析→架构设计→架构评审→任务拆解）→ 
开发工程师（功能开发→单元测试→本地验证）→ 
代码评审工程师（代码评审）→ 
测试工程师（测试策略→测试执行→Bug反馈）→ 
开发工程师（Bug修复）→ 
测试工程师（回归测试→上线结论）→ 
发布运维（文档更新→CICD验证→自动化发布→文档归档）
```

### 1.2 全局Skill嵌入链路

全局Skill以串联叠加、跨切面嵌入的方式贯穿项目Skill主链路：

| 全局Skill | 触发条件 | 嵌入位置 | 嵌入模式 |
|-----------|---------|---------|---------|
| brainstorming | 需求不明确、新功能设计、重大变更 | 阶段0/1 前置 | 串联 |
| writing-plans | 任务拆解完成，需编写实施计划 | 阶段1→2 桥接 | 串联 |
| test-driven-development | 编写功能代码或修复代码 | 阶段2/3 内嵌 | 嵌入 |
| systematic-debugging | Bug根因不明、测试失败、异常行为 | 阶段3 前置 | 串联 |
| verification-before-completion | 任何声称完成时 | 全阶段 守门 | 守门 |
| using-git-worktrees | 需要隔离工作区 | 阶段2 辅助 | 串联 |
| git-pushing | 日常Git操作 | 阶段2/4 辅助 | 嵌入 |
| finishing-a-development-branch | 开发完成，需决定集成方式 | 阶段2→3 桥接 | 串联 |
| subagent-driven-development | 有实施计划，单任务需分步执行 | 阶段2 协调 | 并行 |
| executing-plans | 有实施计划，需独立会话批量执行 | 阶段2 协调 | 并行 |
| dispatching-parallel-agents | 2+个独立任务可并行 | 阶段2 协调 | 并行 |
| requesting-code-review | 完成任务后需代码质量快检 | 阶段2 辅助 | 守门 |
| receiving-code-review | 收到评审反馈 | 阶段2 响应 | 嵌入 |

### 1.3 全景协作链路

```
                        ← [brainstorming] 需求不明确时前置
产品及项目经理（产品规划）
        ↓
                        ← [brainstorming] 需求澄清
架构师（需求分析→架构设计→架构评审→任务拆解）
        ↓               ← [writing-plans] 任务拆解后编写实施计划
                        ← [using-git-worktrees] 创建隔离工作区
开发工程师（功能开发→单元测试→本地验证）
        ↓               ← [TDD] 编码过程内嵌
                        ← [subagent-driven-development] 单任务分步执行
                        ← [executing-plans] 独立会话批量执行
                        ← [dispatching-parallel-agents] 多任务并行
                        ← [verification] 每步完成验证
代码评审工程师（代码评审）
        ↓               ← [requesting-code-review] 可选快检
测试工程师（测试策略→测试执行→Bug反馈）
        ↓               ← [systematic-debugging] Bug根因不明时前置
开发工程师（Bug修复）
        ↓               ← [TDD] 修复过程内嵌
                        ← [verification] 修复完成验证
                        ← [finishing-a-development-branch] 集成决策
测试工程师（回归测试→上线结论）
        ↓               ← [verification] 上线前验证
发布运维（文档更新→CICD验证→自动化发布→文档归档）
                        ← [verification] 最终守门
                        ← [git-pushing] 发布内嵌
```

## 3. 全局Skill嵌入规范

### 3.1 嵌入模式

全局Skill通过四种模式嵌入项目Skill主链路（详细定义见 [Skills协作.md](./Skills协作.md) 第7.1节）：

| 模式 | 说明 | 示例 |
|------|------|------|
| **串联** | 全局Skill输出作为项目Skill输入 | brainstorming → 产品规划 → 需求分析 |
| **嵌入** | 全局Skill作为方法论内嵌于项目Skill | TDD循环嵌入功能开发过程 |
| **守门** | 全局Skill作为质量关卡在关键节点拦截 | 功能开发 → [verification] → 代码评审 |
| **并行** | 全局Skill协调多个项目Skill并行执行 | dispatching-parallel-agents 调度多Agent |

### 3.2 全局Skill铁律

| Skill | 铁律 | 违反后果 |
|-------|------|---------|
| brainstorming | 不经设计批准，不写任何代码 | 方向偏差，返工风险 |
| test-driven-development | 没有失败测试，不写生产代码 | 测试覆盖不足，质量失控 |
| systematic-debugging | 没有根因调查，不提任何修复 | 治标不治本，Bug反复出现 |
| verification-before-completion | 没有新鲜验证证据，不做完成声称 | 虚假完成，质量隐患 |
| writing-plans | 每步必须包含实际内容，禁止占位符 | 计划不可执行，实施偏离 |

### 3.3 调用优先级

当多个Skill可能适用时，遵循以下优先级：

1. **流程型全局Skill**（brainstorming、systematic-debugging）— 决定如何切入问题
2. **实施型全局Skill**（writing-plans、TDD）— 指导如何执行
3. **项目Skill** — 按标准化流程交付
4. **质量守门型全局Skill**（verification-before-completion）— 最终验证

### 3.4 功能重叠决策速查

| 场景 | 决策规则 |
|------|---------|
| 需求不明确 | brainstorming（全局）→ 产品规划（项目）→ 需求分析（项目），三者串联 |
| 任务规划 | 任务拆解（项目）宏观切分 → writing-plans（全局）微观展开 |
| 编码方法 | TDD（全局）嵌入功能开发（项目），不是替代关系 |
| Bug定位 | 根因未知 → systematic-debugging（全局）→ Bug修复（项目）；根因已知 → 直接Bug修复 |
| 代码评审 | subagent模式 → requesting-code-review（全局）；标准流程 → 代码评审（项目） |
| 完成验证 | verification-before-completion（全局）叠加在所有项目Skill之上 |
| Git操作 | 版本发布 → 单人自动化发布（项目）；日常开发 → git-pushing（全局） |

## 5. 异常处理

| 异常场景 | 处理方式 | 责任人 | 关联全局Skill |
|---------|---------|-------|-------------|
| 产品规划方向不明确 | 召开产品评审会议，重新定义产品愿景 | 产品及项目经理 | brainstorming（开放性探索） |
| 需求理解存在歧义 | 逐个提问澄清，产出设计文档 | 架构师 | brainstorming（需求澄清） |
| 本地调试多次失败 | 输出调试报告，同步架构师排查 | 开发工程师 | systematic-debugging（根因定位） |
| Bug修复3次仍失败 | 停止修复，质疑架构设计 | 架构师 | systematic-debugging（Phase 3质疑架构） |
| CICD流水线连续失败 | 同步架构师、测试工程师共同分析 | 发布运维 | FIX-CI → systematic-debugging（配置型→直接修复；代码型→根因定位） |
| 架构评审不通过 | 按评审报告整改后重新提交 | 架构师 | — |
| 代码评审不通过 | 按评审报告整改后重新提交 | 开发工程师 | receiving-code-review（技术评估与实施） |
| 测试通过率<95% | 优先修复失败用例，达标后进入发布 | 开发工程师 | systematic-debugging（失败用例根因分析） |

## 7. 单人开发流程裁剪指南

本项目为单人开发模式，完整团队流程在以下场景中可合理裁剪：

### 7.1 裁剪原则

1. **方法论不可省** — TDD、systematic-debugging、verification-before-completion 始终执行
2. **文档按需产出** — 根据变更影响面决定文档完整度
3. **角色合并执行** — 同一人连续执行多角色，减少形式化切换

### 7.2 按任务类型裁剪

| 任务类型 | 执行流程 | 文档产出 |
|---------|---------|---------|
| **小优化/Bug修复**（≤半天）| systematic-debugging → TDD → verification | 仅代码+测试，可选 Bug修复报告 |
| **中等功能**（半天~2天）| brainstorming（轻量）→ TDD → verification → 代码评审 | 代码+测试+交付报告 |
| **大功能/架构变更**（>2天）| 完整流程：brainstorming → 需求分析 → 架构设计 → 任务拆解 → TDD → 代码评审 → 测试执行 | 全套文档 |
| **紧急Hotfix** | systematic-debugging（加速Phase 2-3）→ TDD → verification → 回归测试 | 修复代码+测试+回归报告 |

### 7.3 流程步骤合并建议

```
完整链路（12步）：
  产品规划 → 需求分析 → 架构设计 → 架构评审 → 任务拆解 → 
  writing-plans → 功能开发 → 代码评审 → 测试策略 → 测试执行 → 
  回归测试 → 发布

轻量链路（5步，适用大多数迭代）：
  需求确认 → [TDD] 功能开发 → [verification] 代码评审 → 
  测试执行 → 发布
```

**合并规则**：
- 产品规划 + 需求分析 → **需求确认**（一条记录即可）
- 架构设计 + 架构评审 → 仅重大变更时执行
- 任务拆解 + writing-plans → 仅复杂任务保留
- 测试策略 → 仅首次或测试框架变更时
- 文档归档 → 发布时顺带完成

### 7.4 时效规则说明

第6节时效规则定义的是团队协作场景的参考值。单人模式下，按实际优先级自行安排，但 **P0级安全/Bug问题应立即处理**。
