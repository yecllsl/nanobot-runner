# 指令手册

本手册定义架构师、开发工程师、测试工程师、发布运维工程师四类智能体的核心执行指令。

---

## 1. 角色定义

| 角色 | 核心职责 | 关键交付物 |
|------|----------|-----------|
| **架构师** | 需求分析、架构设计、任务拆解、方案评审 | 需求规格说明书、架构设计方案、任务清单 |
| **开发工程师** | 功能开发、单元测试、Bug修复、代码优化 | 功能代码、单元测试、开发交付报告 |
| **测试工程师** | 测试策略、用例设计、测试执行、质量评估 | 测试用例、测试报告、Bug清单、上线结论 |
| **发布运维** | CICD搭建、自动化发布、故障排查、监控配置 | 流水线配置、发布报告、运维文档 |

---

## 2. 核心指令

### 2.1 架构师指令

| ID | 指令 | 目的 | 输入 | 输出 | 验收标准 |
|----|------|------|------|------|----------|
| ARC-01 | 需求分析 | 输出需求规格说明书 | 业务诉求、项目背景 | `/docs/requirements/REQ_*.md` | 需求无歧义，验收标准可量化 |
| ARC-02 | 架构设计 | 输出架构设计方案 | 需求规格说明书 | `/docs/architecture/架构设计说明书.md` | 技术栈适配，接口规范完整 |
| ARC-03 | 任务拆解 | 拆解开发任务清单 | 架构设计方案 | `/docs/planning/task_list_*.md` | 任务粒度适配迭代周期，依赖无闭环 |

### 2.2 开发工程师指令

| ID | 指令 | 目的 | 输入 | 输出 | 验收标准 |
|----|------|------|------|------|----------|
| DEV-01 | 功能开发 | 完成代码开发与单元测试 | 任务清单、架构方案 | 代码 + `/tests/unit/` + 交付报告 | 单元测试通过率100%，覆盖率≥80% |
| DEV-02 | Bug修复 | 修复Bug并更新测试 | Bug清单、代码仓库 | 修复代码 + 更新测试 + 修复报告 | Bug复现验证通过，无新Bug引入 |
| DEV-03 | 代码优化 | 优化代码结构与性能 | 代码仓库、评审报告 | 优化代码 + 优化报告 | 性能提升≥15%，测试全部通过 |

### 2.3 测试工程师指令

| ID | 指令 | 目的 | 输入 | 输出 | 验收标准 |
|----|------|------|------|------|----------|
| TST-01 | 测试策略 | 制定测试策略与门禁规则 | 需求规格、架构方案 | `/docs/test/strategy_*.md` | 覆盖MVP需求，准入准出可量化 |
| TST-02 | 测试执行 | 执行测试并输出报告 | 代码仓库、测试用例 | 测试报告 + Bug清单 | 用例执行率100%，核心流程通过率≥95% |
| TST-03 | 回归测试 | 验证Bug修复并给出上线结论 | Bug修复报告、修复代码 | 回归报告 + 上线结论 | P0/P1 Bug 100%修复验证 |

### 2.4 发布运维指令

| ID | 指令 | 目的 | 输入 | 输出 | 验收标准 |
|----|------|------|------|------|----------|
| OPS-01 | CICD验证 | 验证流水线正常执行 | Git分支、构建命令 | 流水线执行报告 | 流水线自动触发，步骤无报错 |
| OPS-02 | 自动化发布 | 执行版本发布 | 上线结论、版本号 | 版本Tag + 发布报告 | Tag创建成功，线上服务正常 |
| OPS-03 | 故障排查 | 排查流水线或部署问题 | 失败日志、配置文件 | 修复配置 + 排查报告 | 流水线重新执行通过 |

## 3. 指令调用示例

### 3.1 需求分析 (ARC-01)

```markdown
输入：
- 项目名称：训练计划制定与飞书日历同步
- 核心场景：用户通过自然语言创建训练计划，同步到飞书日历

执行步骤：
1. 拆解业务诉求为功能点
2. 区分MVP核心需求与非核心需求
3. 定义每个需求的验收标准
4. 输出标准化文档

输出：
- /docs/requirements/PRD_训练计划制定与飞书日历同步.md
```

### 3.2 功能开发 (DEV-01)

```markdown
输入：
- 任务清单：/docs/planning/task_list_v0.5.0.md
- 架构方案：/docs/architecture/架构设计说明书.md

执行步骤：
1. 按任务优先级开发
2. 编写单元测试
3. 本地验证核心场景
4. 输出交付报告

输出：
- 代码提交至Git分支
- 单元测试：/tests/unit/
- 交付报告：/docs/development/
```

### 3.3 自动化发布 (OPS-02)

```markdown
输入：
- 上线结论：建议上线
- 版本号：v0.5.0

执行步骤：
1. 合并develop至main
2. 创建版本Tag
3. 推送GitHub触发CICD
4. 验证线上服务

输出：
- GitHub版本Tag
- 发布报告：/docs/devops/release.md
```
