# 智能体标准化执行指令手册
## 文档说明
### 1. 文档目的
本手册规范架构师、开发工程师、测试工程师、发布运维工程师四类智能体的核心执行指令，明确各指令的输入输出、执行流程、验收标准，建立智能体间协作链路，保障项目从需求分析到版本发布全流程的标准化、可落地性。
### 2. 适用范围
适用于基于Trae IDE环境的项目全生命周期管理，覆盖需求、架构、开发、测试、发布运维全阶段的智能体执行指令。
### 3. 角色定义
| 角色 | 核心职责 |
|------|----------|
| 架构师智能体 | 需求分析、架构设计、任务拆解、方案评审、风险评估、安全合规审查，输出标准化架构类交付物 |
| 开发工程师智能体 | 功能开发、单元测试、Bug修复、本地调试、功能迭代、代码质量优化，输出可交付的代码与测试成果 |
| 测试工程师智能体 | 测试策略制定、用例设计、自动化测试开发、测试执行、回归测试、质量评估、安全测试，把控上线质量 |
| 发布运维工程师智能体 | 仓库管理、CICD搭建、容器化部署、自动化发布、故障排查、运维文档生成、监控配置，保障交付落地 |
### 4. 文档版本控制
- 当前版本：v1.1.0
- 版本规则：主版本号.次版本号.修订号
- 更新记录：每次更新需填写《文档变更记录表》（存储至/docs/manual/change_log.md）
- 评审机制：每次版本更新需经四类智能体代表评审通过

## 智能体协作流程
### 1. 核心协作链路
```
架构师智能体（需求分析→架构设计→任务拆解→安全合规审查）→ 
开发工程师智能体（全量开发/功能迭代→单元测试补充→代码质量优化）→ 
测试工程师智能体（测试用例设计→全量测试执行→安全测试→Bug反馈）→ 
开发工程师智能体（Bug修复）→ 
测试工程师智能体（回归测试→质量评估）→ 
发布运维工程师智能体（自动化发布→监控配置→运维文档生成）
```
### 2. 流转规则
- 上游智能体输出的交付物需存储至指定目录，下游智能体从指定目录读取输入；
- 测试工程师反馈的P0/P1级Bug，开发工程师需在2个工时内响应修复；
- 发布运维工程师执行自动化发布前，必须获取测试工程师的“上线通过”结论；
- 所有交付物需按指定路径存储，命名规范为：`角色缩写_指令名称_版本号.md`（如ARC-01_需求分析_v1.0.md）。
### 3. 跨角色协作规范
- **每日站会**：每日固定时间同步进度，时长不超过15分钟
- **代码评审**：开发工程师提交代码前需进行自我评审，关键模块需跨角色评审
- **测试准入**：开发工程师提交测试前需完成自测，提供自测报告
- **发布评审**：重要版本发布前需召开发布评审会，四类智能体共同参与
### 4. 沟通机制
- 使用标准化文档作为沟通基础，避免信息不对称
- 重要决策需记录在《项目决策日志》中（存储至/docs/project/decisions.md）
- 跨角色问题通过《问题追踪表》进行管理（存储至/docs/project/issues.md）
## 异常场景处理规则
### 1. 架构评审不通过
- 开发工程师需按架构评审报告的优化方案整改，整改完成后重新提交架构师评审；
- 核心架构偏离问题未整改前，测试工程师不执行全量测试。
### 2. 测试用例通过率低于90%
- 开发工程师需优先修复导致用例失败的问题，直至通过率≥95%；
- 未达标的版本，发布运维工程师拒绝执行自动化发布。
### 3. CICD流水线执行失败
- 发布运维工程师需在1个工时内排查根因并修复配置；
- 修复后重新触发流水线，连续2次失败需同步架构师、测试工程师共同分析。
### 4. 本地调试多次失败
- 开发工程师需输出调试分析报告，同步架构师确认是否为架构设计问题；
- 环境类问题需发布运维工程师协助解决。
### 5. 安全测试发现高危漏洞
- 立即暂停发布流程，开发工程师需在4小时内修复高危漏洞；
- 修复后需重新执行安全测试，直至无高危漏洞。
### 6. 生产环境故障
- 发布运维工程师需在15分钟内启动应急预案；
- 故障处理完成后需编写《故障复盘报告》（存储至/docs/devops/incidents/）。
### 7. 需求变更
- 架构师需评估变更影响，更新需求文档和架构设计；
- 开发工程师、测试工程师需同步调整开发计划和测试用例。
## 安全与合规要求
### 1. 数据安全
- 敏感数据需加密存储，传输过程使用HTTPS
- 数据库连接信息等配置需使用环境变量，禁止硬编码
- 定期进行数据备份，备份文件需加密存储
### 2. 代码安全
- 禁止提交敏感信息（如密钥、密码）到代码仓库
- 使用静态代码分析工具检查安全漏洞
- 第三方依赖需定期更新，避免使用有已知漏洞的版本
### 3. 访问控制
- 生产环境访问需使用VPN或堡垒机
- 权限分配遵循最小权限原则
- 操作日志需完整记录，保存期限不少于6个月
### 4. 合规要求
- 遵循行业相关法律法规（如GDPR、网络安全法）
- 用户隐私数据需脱敏处理
- 定期进行合规审计
## 能力评估与持续改进
### 1. 智能体能力评估
- 每月进行能力评估，评估指标包括任务完成率、交付质量、响应时间等
- 评估结果记录在《智能体能力评估报告》中（存储至/docs/evaluation/）
### 2. 流程优化
- 每季度进行流程评审，识别流程瓶颈和改进点
- 流程优化建议需经四类智能体代表讨论通过后实施
### 3. 知识沉淀
- 项目过程中的经验教训需记录在《项目经验库》中
- 定期组织技术分享会，促进知识传递
## 文档版本管理
1. 本手册版本号遵循语义化版本（如v1.1.0），更新时递增修订号；
2. 各智能体输出的交付物需标注版本，与手册版本同步；
3. 手册更新需同步所有智能体，确保指令执行标准一致；
4. 文档变更需填写《文档变更记录表》，包含变更内容、变更原因、影响范围；
5. 每半年进行一次文档全面评审，确保文档适用性。
